## Theory

### UpgradeCap — Structure
> Source: Sui Framework > sui::package — docs.sui.io/references/framework/sui_sui/package

```move
public struct UpgradeCap has key, store {
    id: UID,
    package: ID,   // original package ID this controls
    version: u64,  // starts at 0, increments each upgrade
    policy: u8,    // what kinds of upgrades are allowed
}
```

**Key:** `key, store` — NO `drop`. Cannot be discarded with `let _ = cap`. Must call `make_immutable(cap)`.

---

### Upgrade Policies
> Source: docs.sui.io/concepts/sui-move-concepts/packages/upgrade

| Value | Name | What's allowed |
|-------|------|----------------|
| 0 | `compatible` | Change any function body; change/remove private/entry/friend signatures; add functions/types/modules |
| 128 | `additive` | Add new functions or types anywhere; change deps; existing functions fully frozen |
| 192 | `dep_only` | Change dependencies only |
| destroyed | immutable | `make_immutable(cap)` called — no upgrades ever |

**Where policy is set:** On-chain only, via Move calls. NOT in Move.toml (Move.toml has no policy field).
**Default at publish:** `compatible` (0) — always. Never set to anything else automatically.
**Ratchet rule:** Policy can only go UP (more restrictive). `dep_only` → `additive` aborts.

```move
sui::package::only_additive_upgrades(&mut cap);  // → 128
sui::package::only_dep_upgrades(&mut cap);        // → 192
sui::package::make_immutable(cap);               // destroys cap → package frozen
```

---

### What Compatible Actually Allows — The Auditor Trap
> Source: docs.sui.io/concepts/sui-move-concepts/packages/upgrade (verified against docs)

```
Compatible CAN change:                Compatible CANNOT change:
┌────────────────────────────┐        ┌──────────────────────────────┐
│ ✅ Function bodies         │        │ ❌ Public function signatures │
│ ✅ Private fn signatures   │        │ ❌ Struct fields (BCS layout) │
│ ✅ entry fn signatures     │        │ ❌ Struct abilities           │
│ ✅ public(friend) sigs     │        └──────────────────────────────┘
│ ✅ Add new functions       │
│ ✅ Add new types/modules   │
└────────────────────────────┘
```

**Auditor implication:** A compatible upgrade can skip `ed25519_verify`, change fee %, change recipients — all with the same public API. "Compatible policy" is NOT a safety signal.

---

### UpgradeCap Destruction — Only Valid Method

```move
sui::package::make_immutable(cap);  // ✅ only valid method — takes cap by value

let _ = cap;                        // ❌ COMPILE ERROR — no `drop` ability
let UpgradeCap { .. } = cap;       // ❌ COMPILE ERROR — private fields, wrong module
```

**`store` ≠ `drop`:**
| Ability | Meaning |
|---------|---------|
| `store` | Can be wrapped in other objects / public_transfer-able |
| `drop`  | Can be silently discarded / ignored with `_` |

### CLI — On-Chain Verification
> Source: Sui CLI

```bash
# Check UpgradeCap policy + owner
sui client object <UPGRADE_CAP_ID> --json
# .content.fields.policy → 0=compatible, 128=additive, 192=dep_only
# .owner.AddressOwner    → wallet that controls it
# "error: Object not found" → cap destroyed → immutable ✅

# Verify multi-sig address
sui keytool multi-sig-address \
  --pks <PK1> <PK2> <PK3> --weights 1 1 1 --threshold 2

# Find all caps owned by an address
sui client objects <ADDRESS> --json | jq '.[] | select(.type | contains("Cap"))'
```

**Owner type cheat sheet:**
```
{ "AddressOwner": "0x..." }  → wallet (EOA or multi-sig)
{ "ObjectOwner":  "0x..." }  → wrapped inside another object
{ "Shared": { ... } }        → shared object
```

### Multi-Role Access Control
> Source: Sui Object Model + Session 2026-04-24

**Single-cap anti-pattern:**
```move
// ❌ one key for everything — total protocol takeover if compromised
public struct AdminCap has key, store { id: UID }
public fun set_fee(_: &AdminCap, ...) { ... }
public fun pause(_: &AdminCap, ...) { ... }
public fun mint(_: &AdminCap, ...) { ... }
```

**Correct role separation:**
```move
// ✅ separated by blast radius + operational cadence
public struct AdminCap    has key { id: UID }  // governance — multi-sig, cold
public struct OperatorCap has key { id: UID }  // day-to-day — hot wallets, bots
public struct MinterCap   has key { id: UID }  // mint only
public struct PauserCap   has key { id: UID }  // emergency — any team member
// NO `store` on any — non-transferable by external code
```

---

### `store` vs `key`-only — Transfer Enforcement

```
Transfer function signatures:
┌─────────────────────────────────────────────────────────────┐
│ public fun public_transfer<T: key + store>(obj: T, addr)    │
│   → callable from ANY module or PTB                         │
│   → AdminCap has key, store → attacker PTB can steal it     │
│                                                             │
│ public fun transfer<T: key>(obj: T, addr)                   │
│   → callable ONLY from the module that defines T           │
│   → AdminCap has key only → external PTBs rejected by type  │
└─────────────────────────────────────────────────────────────┘
```

Enforcement is at **compile time** — type constraint `T: key + store` in the function signature. Not a runtime check.

### Two Cap Extraction Vulnerabilities
> Source: Session 2026-04-24

**Vuln A — Unguarded mint (constructs new cap):**
```move
// ❌ anyone calls this → gets a fresh cap. No auth guard.
public fun emergency_recover(ctx: &mut TxContext): AdminCap {
    AdminCap { id: object::new(ctx) }
}
// Grep: `CapType { id: object::new(ctx) }` in any `public fun`
```

**Vuln B — Unguarded extract (destructs wrapper to pull out embedded cap):**
```move
// ❌ anyone passes in the shared container → walks away with the real cap
public fun emergency_extract(protocol: Protocol): AdminCap {
    let Protocol { id, admin_cap } = protocol;  // module-internal only
    object::delete(id);
    admin_cap
}
// Grep: `let ContainerType {` in any `public fun`
```

### Key Mnemonics

**Card 1 — Policy location:**
> Move.toml is `Cargo.toml` — describes the package, holds no runtime state. UpgradeCap is a `Mutex<Policy>` — the lock object holds the state. Policy lives where the cap lives: on-chain.

**Card 2 — Compatible blast radius:**
> **P**ublic and **S**truct-fields are **P**rotected and **S**tatic. Bodies and private sigs are yours to change. Same as Rust: `pub fn` signature = stability guarantee, body = your implementation.

**Cards 3 & 4 — make_immutable vs let _:**
> `store` = moving truck (carry elsewhere). `drop` = trash can (throw away). UpgradeCap has a truck but no trash can. To destroy it, call the incinerator: `make_immutable(cap)`.

**Card 5 — public_transfer type constraint:**
> Rust: `fn transfer<T: Send>(obj: T)` — no Send = compile error at call site. Same: `public_transfer<T: key + store>` — no `store` = type bound fails = compile error. Not runtime.

**Card 6 — key vs key+store:**
> `key + store` = passport + visa — anyone can stamp it across borders. `key` only = passport, no visa — only the issuing embassy (defining module) can move it.

**Cards 7 & 8 — Audit grep patterns:**
> (A) "Who can birth a cap?" → grep `CapType { id: object::new`. (B) "Who can kidnap a cap?" → grep `let ContainerType {`. Either in a `public fun` without guard = CRITICAL.

**Card 11 — Pawtato Finance:**
> `UpgradeCap` = government ID — every citizen has one. "Do you have an ID?" proves nothing. Check the NAME on the ID: `assert!(upgrade_package(cap) == @my_pkg)`.

**Card 12 — Policy ratchet:**
> A zip tie — only tightens, never loosens. Cut it with `make_immutable`.

## Key Takeaways

1. **Upgrade policy is on-chain in the UpgradeCap object — never in Move.toml. Default is always `compatible`. Query the object, never trust docs.** *Source: docs.sui.io/concepts/sui-move-concepts/packages/upgrade*
2. **`compatible` policy allows changing function bodies AND non-public signatures — a protocol can be silently gutted while its public API stays identical.** *Source: docs.sui.io/concepts/sui-move-concepts/packages/upgrade*
3. **No `store` on a cap is the primary transfer defense — `public_transfer` requires `store` as a compile-time type constraint, not a runtime check.** *Source: Sui Framework — transfer module*

> "A compatible upgrade can skip signature verification, change the fee to 100%, and redirect funds — all with the same public function signature. 'Compatible policy' is not a safety signal."
