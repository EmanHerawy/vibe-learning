## Theory Notes

### PTB Architecture — Commands, Result Passing, Atomicity
**Source:** Sui Docs > References > PTB Commands — docs.sui.io/references/ptb-commands
**Summary:** A PTB is a single Sui transaction containing up to hundreds of commands that execute atomically. 7 command types: MoveCall, TransferObjects, SplitCoins, MergeCoins, MakeMoveVec, Publish, Upgrade. Results pass between commands by index (Result(0), Result(1)) — no storage, no round-trip. All commands succeed or all revert. No conditional branching — returning false doesn't skip the next command.
**Bridged from:** Solidity multicall — same multi-step atomicity goal, but PTBs are a native transaction format, not a wrapper contract. Move functions need no PTB-specific code to be composable.
**Misconception flagged:** "PTBs are like Ethereum multicall." — No. Chaining is at the transaction layer, not inside any contract. Also: "returning false from Command 1 skips Command 2" — No. Only abort skips. Safety checks must live inside the called function.
**Checkpoint Q 1:** Mint function transfers internally — can its result be used in a PTB chain?
**Learner answer:** No — must return Coin<T> so PTB runtime can reference it as Result(0). ✅
**Checkpoint Q 2:** check_health returns false → does liquidate still run?
**Learner answer:** Yes, unless check_health aborts. PTBs have no conditional branching — liquidate must re-check internally. ✅
**Status:** ✅ understood

### PTB-Friendly Module Design
**Source:** Sui Docs > References > PTB Commands — docs.sui.io/references/ptb-commands
**Summary:** Functions must return values (not transfer internally) to be PTB-composable. Internal transfers break the chain — the object is gone from PTB context. Returning objects lets the PTB caller decide what to do with them in subsequent commands.
**Bridged from:** W8 mint function design — same rule, now understood at the transaction level.
**Status:** ✅ understood

### Move Testing Framework
**Source:** Move Book §5.26 Testing — move-book.com/move-basics/testing
**Summary:** Tests use #[test] (marks test function), #[test_only] (code only compiled in tests), #[expected_failure(abort_code=N)] (test passes if it aborts with that code). Test functions take no arguments, return nothing, and are excluded from published bytecode. Basic #[test] is single-transaction. test_scenario is needed for multi-transaction flows with ownership/address changes.
**Bridged from:** Rust #[test] annotation — same concept, same syntax, same exclusion from production builds.
**Misconception flagged:** "I can test ownership transfers in #[test]." — Basic #[test] is single-transaction/single-address. Multi-user ownership flows need test_scenario.
**Checkpoint Q:** Total staked invariant after 2 deposits — what does #[test] check, what needs test_scenario?
**Learner answer:** #[test] checks balance == 800; test_scenario needed for ownership transfers between addresses. ✅
**Status:** ✅ understood
