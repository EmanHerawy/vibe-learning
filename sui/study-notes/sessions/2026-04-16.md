## Theory Notes

### PTB Architecture — Commands, Result Passing, Atomicity
**Source:** Sui Docs > References > PTB Commands — docs.sui.io/references/ptb-commands
**Summary:** A PTB is a single Sui transaction containing up to hundreds of commands that execute atomically. 7 command types: MoveCall, TransferObjects, SplitCoins, MergeCoins, MakeMoveVec, Publish, Upgrade. Results pass between commands by index (Result(0), Result(1)) — no storage, no round-trip. All commands succeed or all revert. No conditional branching — returning false doesn't skip the next command.
**Bridged from:** Solidity multicall — same multi-step atomicity goal, but PTBs are a native transaction format, not a wrapper contract. Move functions need no PTB-specific code to be composable.
**Misconception flagged:** "PTBs are like Ethereum multicall." — No. Chaining is at the transaction layer, not inside any contract. Also: "returning false from Command 1 skips Command 2" — No. Only abort skips. Safety checks must live inside the called function.
**Checkpoint Q 1:** Mint function transfers internally — can its result be used in a PTB chain?
**Learner answer:** No — must return Coin<T> so PTB runtime can reference it as Result(0). ✅
**Checkpoint Q 2:** check_health returns false → does liquidate still run?
**Learner answer:** Yes, unless check_health aborts. PTBs have no conditional branching — liquidate must re-check internally. ✅
**Status:** ✅ understood

### PTB-Friendly Module Design
**Source:** Sui Docs > References > PTB Commands — docs.sui.io/references/ptb-commands
**Summary:** Functions must return values (not transfer internally) to be PTB-composable. Internal transfers break the chain — the object is gone from PTB context. Returning objects lets the PTB caller decide what to do with them in subsequent commands.
**Bridged from:** W8 mint function design — same rule, now understood at the transaction level.
**Status:** ✅ understood

### PTBs are client-side, not deployed Move code
**Source:** Sui TypeScript SDK — sdk.mystenlabs.com/typescript/transaction-building/basics
**Summary:** PTBs are NOT Move modules or packages. They are constructed client-side using the TypeScript (or Rust) SDK, assembled as a Transaction object, and submitted to the network. Move modules just need to expose PTB-friendly functions (return values). The chaining, result passing, and command sequencing all happen in TypeScript.
**Key distinction:**
```
Move package (deployed once):
  public fun deposit(...): StakeReceipt  ← PTB-friendly: returns value

TypeScript client (per user action):
  const tx = new Transaction()
  const [coin] = tx.splitCoins(tx.gas, [500])
  const [receipt] = tx.moveCall({ target: '0xPKG::ptb_w10::deposit', arguments: [vault, coin] })
  tx.transferObjects([receipt], coldWallet)
  await client.signAndExecuteTransaction({ signer: keypair, transaction: tx })
```
**Why it matters:** When auditing a protocol, check both the Move module APIs (are they PTB-friendly?) AND any client scripts (are PTBs constructed safely?). Two separate attack surfaces.
**Status:** ✅ understood (TypeScript SDK deep dive — W10 secondary / frontend work)

### Move Testing Framework
**Source:** Move Book §5.26 Testing — move-book.com/move-basics/testing
**Summary:** Tests use #[test] (marks test function), #[test_only] (code only compiled in tests), #[expected_failure(abort_code=N)] (test passes if it aborts with that code). Test functions take no arguments, return nothing, and are excluded from published bytecode. Basic #[test] is single-transaction. test_scenario is needed for multi-transaction flows with ownership/address changes.
**Bridged from:** Rust #[test] annotation — same concept, same syntax, same exclusion from production builds.
**Misconception flagged:** "I can test ownership transfers in #[test]." — Basic #[test] is single-transaction/single-address. Multi-user ownership flows need test_scenario.
**Checkpoint Q:** Total staked invariant after 2 deposits — what does #[test] check, what needs test_scenario?
**Learner answer:** #[test] checks balance == 800; test_scenario needed for ownership transfers between addresses. ✅

### Key Mnemonics
- **7 PTB commands = "MTC-MMP-U"**: MoveCall, TransferObjects, SplitCoins — MergeCoins, MakeMoveVec, Publish — Upgrade. "Move The Coins, Merge Make Publish, Upgrade."
- **PTB results = pipeline in Rust**: Like `iter().map().filter()` — each step takes the previous step's output. Result(0) → Result(1) → Result(2). No storage between steps.
- **PTB atomicity = database transaction**: Like a SQL BEGIN/COMMIT — all or nothing. One failure rolls back everything. Sui's atomic unit is the PTB, not the command.
- **No conditional branching = no if/else in a shell pipeline**: `cmd1 | cmd2` — cmd2 always runs regardless of cmd1's exit code, unless cmd1 crashes. PTB is the same. Crash (abort) = stops. Return value = doesn't stop.
- **PTB safety check = bouncer inside the club**: The bouncer must be at the door you enter (inside the function), not at the door you left (previous command). Returning false is leaving a sign on the door — someone can just ignore it.
- **PTB-friendly = return the package, don't deliver it**: FedEx hands you the package (returns the object). They don't decide where it goes next. Transferring internally = FedEx delivers it themselves — you lose control.
- **PTBs client-side = SQL query vs stored procedure**: The Move module = stored procedure (deployed once, called many times). The PTB = the SQL query the client assembles and sends. The DB doesn't write queries for you.
- **Unhandled key object = hot potato left on counter**: Key object returned but not consumed = non-droppable value abandoned. Just like hot potato — you MUST deal with it or the whole thing aborts.
- **#[test] = Rust #[test]**: Exact same concept, exact same syntax, exact same "excluded from production build" behavior. If you know Rust testing, you know Move testing — minus the argument rules.
- **#[test_only] = #[cfg(test)] in Rust**: Code that only exists in the test build. Helper functions, mock objects, test utilities. Never published.
- **#[expected_failure] without code = catching Exception in Java**: Catches everything — NullPointerException, your specific error, AND a random unrelated crash. Pin the abort_code like catching a specific exception class.
- **test_scenario = time machine for the object store**: begin() = turn on the machine. next_tx() = jump forward one transaction. take_shared() = reach in and grab an object. return_shared() = put it back before jumping again.
- **begin/next_tx/take_shared/return_shared = open/advance/grab/replace**: Like operating a vending machine: open the panel (begin), advance the mechanism (next_tx), grab the item (take_shared), put back the tray (return_shared).

### What We Covered
- PTB = single atomic transaction with multiple chained commands
- 7 PTB command types: MoveCall, TransferObjects, SplitCoins, MergeCoins, MakeMoveVec, Publish, Upgrade

- PTB atomicity: all succeed or all revert
- No conditional branching in PTBs — only abort skips commands
- PTB-friendly design: return values, don't transfer internally
- PTBs are client-side (TypeScript SDK) — not deployed Move code
- Unhandled returned key object = transaction fails
- Move test framework: #[test], #[test_only], #[expected_failure(abort_code=N)]
- test_scenario: begin, next_tx, take_shared, return_shared
- Common test bugs fixed: named addresses, ctx(), #[test] on args, assert_eq!

### Key Mechanisms
1. **PTB result chaining**: `tx.moveCall(deposit, [vault, coin])` returns `[receipt]`. Then `tx.transferObjects([receipt], coldWallet)`. The receipt flows directly — no storage. Source: Sui Docs PTB Commands + TypeScript SDK
2. **PTB safety check rule**: Every function must assert its own invariants. `check_health()` returning false does NOT prevent `liquidate()` from running. Only `abort` stops the next command. Source: Sui Docs PTB Commands
3. **test_scenario flow**: `begin(admin)` → deploy in block → `next_tx(user)` → `take_shared<Vault>()` → act → `return_shared(vault)` → `end()`. Source: Move Book §9.5

### Key Distinctions
- PTB vs Ethereum multicall: PTBs are a native transaction format — no wrapper contract. Move functions need zero PTB-specific code.
- #[test] vs #[test_only]: test = runs as a test. test_only = compiled in test mode but not run directly (helpers).
- #[expected_failure] vs #[expected_failure(abort_code=N)]: former catches any abort; latter pins the exact code — regression-safe.
- #[test] vs test_scenario: #[test] = single tx, single address. test_scenario = multi-tx, multi-address, shared objects.

### Gotchas
- Named addresses @Alice won't compile — use @0xAA or declare in Move.toml
- scenario.ctx without () = field access, not method — always scenario.ctx()
- #[test] on function with arguments = compile error — use #[test_only]
- assert_eq! doesn't exist — use assert!(a == b, error_code)
- Vault without mut when passing &mut Vault = compile error
- Unhandled key object returned from MoveCall = tx fails at execution

### Code / Patterns to Remember
```move
// Correct test_scenario pattern
#[test]
fun test_deposit() {
    use sui::test_scenario;
    let admin = @0xAD;               // hex, not @admin
    let mut scenario = test_scenario::begin(admin);
    { init_for_testing(scenario.ctx()); };  // ctx() not ctx
    scenario.next_tx(admin);
    {
        let mut vault = scenario.take_shared<Vault>();  // mut required
        let coin = coin::mint_for_testing<SUI>(500, scenario.ctx());
        let receipt = deposit(&mut vault, coin, scenario.ctx());
        assert!(receipt_amount(&receipt) == 500, 0);   // assert! not assert_eq!
        test_scenario::return_shared(vault);            // required
        transfer::public_transfer(receipt, admin);
    };
    scenario.end();
}

// PTB-friendly function (returns, doesn't transfer)
public fun deposit(vault: &mut Vault, coin: Coin<SUI>, ctx: &mut TxContext): StakeReceipt {
    let amount = coin::value(&coin);
    assert!(amount > 0, 0);                     // invariant inside function
    coin::put(&mut vault.reserve, coin);
    StakeReceipt { id: object::new(ctx), amount }
}
```

## Key Takeaways

1. **PTBs have no conditional branching — every function must assert its own invariants.** A check that returns false doesn't gate the next command. If `liquidate` doesn't assert internally, it will execute on healthy loans. Source: Sui Docs > References > PTB Commands — docs.sui.io/references/ptb-commands

2. **PTBs are client-side TypeScript, not deployed Move.** Move modules just need PTB-friendly APIs (return values). The chaining lives in the Transaction object the client assembles. Two separate audit surfaces: the Move API and the client script. Source: Sui TypeScript SDK — sdk.mystenlabs.com/typescript/transaction-building/basics

3. **Pin abort codes in #[expected_failure].** Without `abort_code=N`, any unrelated crash satisfies the test. Pinning makes tests regression-safe and documents the exact intended failure. Source: Move Book §5.26 Testing — move-book.com/move-basics/testing

**The one thing I must not forget from today:**
  "PTBs have no if/else. The only way to stop the next command is to abort. Every function is responsible for its own safety — the PTB caller cannot gate execution via return values."
