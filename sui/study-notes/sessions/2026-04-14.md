## Theory Notes

### One-Time Witness as Coin Bootstrap Gate
**Source:** Move Book §8.12 — One-Time Witness — move-book.com/programmability/one-time-witness.html
**Summary:** OTW is a zero-field, drop-only struct whose name matches the module, passed by value to `init` exactly once by the Sui runtime. `coin::create_currency` consumes it, guaranteeing exactly one TreasuryCap per coin type ever exists. Forgery is impossible because struct instantiation is module-private in Move — `GOV {}` cannot be written outside the `gov` module.
**Bridged from:** W5 OTW pattern — today revealed WHY it was designed that way (supply invariant enforcement).
**Misconception flagged:** "OTW forgery is blocked by a runtime check." — Wrong. It's a compile-time guarantee: struct instantiation is module-private. The compiler rejects `GOV {}` outside the gov module before the code ever runs.
**Checkpoint Q:** Can a third module construct `GOV {}` and pass it to `create_currency` to fake a second TreasuryCap?
**Learner answer:** No — runtime checks prevent it. (Corrected: it's compile-time / type system, not runtime.)
**Status:** ✅ understood

### TreasuryCap — Supply Control Object
**Source:** Sui Docs > Currencies and Tokens > Create Currencies and Tokens + sui::coin module
**Summary:** TreasuryCap<T> is a capability object — whoever holds it can call coin::mint and coin::burn. It's the sole authority over supply. If shared, any transaction can mint freely. The correct pattern is owned (transferred to admin) or wrapped inside a module-private field with a delegated MintCap for downstream contracts.
**Bridged from:** W6 Capability pattern — MintCap wrapping TreasuryCap is the standard delegation architecture.
**Misconception flagged:** "TreasuryCap controls transfers." — No. It only gates creation and destruction of supply. Once minted, coins move freely.
**Checkpoint Q:** TreasuryCap is made a shared object at init — what's the vulnerability?
**Learner answer:** Anyone can mint and burn; it should be owned by the admin, not shared. ✅
**Status:** ✅ understood

### Coin<T> Lifecycle — mint / split / join / burn
**Source:** Sui Docs > Currencies and Tokens > Create Currencies and Tokens + sui::coin module
**Summary:** TreasuryCap mints Coin<T> objects (supply increases). split reshapes one Coin into two objects — value conserved. join merges two into one — value conserved. burn destroys a Coin and decreases total supply. The Balance<T> type underneath can only be created/destroyed through mint/burn — the type system enforces the invariant.
**Bridged from:** ERC-20 total supply invariant — same concept, enforced by type system instead of storage accounting.
**Misconception flagged:** "split creates new supply." — No. Split is value-conserving reshaping of the wrapping object only.
**Checkpoint Q:** User has Coin<USDC> value 1000, calls split(400). How many objects, what values, total supply change?
**Learner answer:** Two objects (600 + 400), total supply unchanged. ✅
**Status:** ✅ understood

### DenyList Enforcement Boundary — Transfer Hooks vs Balance Operations
**Source:** Sui Framework docs — sui::coin module reference (docs.sui.io/references/framework/sui-framework/coin)
**Summary:** `coin::put` (Coin → Balance) and `coin::take` (Balance → Coin) are internal balance operations. They do NOT trigger deny list hooks. Deny list checks fire at the transaction input/output boundary only — meaning a deny-listed address cannot use a regulated coin as a transaction input, but can still be the target of internal vault operations that never hit that boundary.
**Why it matters:** Any protocol wrapping a regulated coin in a vault using `Balance<T>` must explicitly check the deny list before deposit and withdrawal — the framework will NOT check for you inside balance operations.
**Exact API (verified):**
- V1: `coin::deny_list_contains(&deny_list, addr)` — checks current state
- V2: `coin::deny_list_v2_contains_current_epoch(&deny_list, addr, ctx)` — epoch-aware
- V2: `coin::deny_list_v2_contains_next_epoch(&deny_list, addr, ctx)` — for next epoch
- V2: `coin::deny_list_v2_enable_global_pause` / `deny_list_v2_is_global_pause_enabled_current_epoch` — global freeze
**Bridged from:** Solidity — like how `transferFrom` checks allowance but internal accounting moves don't. Enforcement is at the boundary, not inside.
**The fix pattern:**
```
// Before deposit — check sender
assert!(!coin::deny_list_v2_contains_current_epoch(&deny_list, sender(ctx), ctx), EDenied);
coin::put(&mut vault.reserve, coin);

// Before withdraw — check recipient
assert!(!coin::deny_list_v2_contains_current_epoch(&deny_list, recipient, ctx), EDenied);
let coin = coin::take(&mut vault.reserve, amount, ctx);
```
**Status:** ✅ verified against Sui framework source

### Mint Function Design — Return Coin<T>, Don't Transfer Internally
**Source:** Sui Docs > Currencies and Tokens > Create Currencies and Tokens (PTB composability pattern)
**Summary:** Mint functions should return `Coin<T>` to the caller rather than calling `transfer::public_transfer` internally. Returning lets PTBs chain the coin into further operations (deposit to pool, split, wrap). Internal transfer ends the PTB flow — the coin is gone and can't be used in the same transaction.
**Bridged from:** Solidity — like returning a value from a function vs emitting an event, the caller needs the handle to do something with it.
**Misconception flagged:** "Transferring to recipient: address inside mint is a vulnerability." — No. It's a composability/design anti-pattern, not a security exploit. The distinction matters in audit reports.
**Best practice:**
```
// ❌ Anti-pattern — internal transfer kills PTB composability
public fun mint(registry: &mut MintRegistry, amount: u64, recipient: address, ctx: &mut TxContext) {
    let coin = coin::mint(&mut registry.treasury, amount, ctx);
    transfer::public_transfer(coin, recipient);   // caller loses the coin handle
}

// ✅ Correct — return Coin<T>, let PTB or caller decide
public fun mint(registry: &mut MintRegistry, _cap: &MintCap, amount: u64, ctx: &mut TxContext): Coin<T> {
    coin::mint(&mut registry.treasury, amount, ctx)
}
```
**Status:** ✅ understood

### Regulated Coins & DenyCap
**Source:** Sui Docs > Currencies and Tokens > Regulated Currency and Deny List
**Summary:** create_regulated_currency returns TreasuryCap + DenyCap + CoinMetadata. DenyCap controls a per-type entry in the system-level DenyList object at 0x403 — blocked addresses cannot send or receive the token. DenyCap and TreasuryCap are separate authorities and should be held separately. Combining them in one shared object collapses least privilege and creates a single point of failure.
**Bridged from:** W6 — capability separation principle. Same as separating AdminCap from OperatorCap in multi-role protocols.
**Misconception flagged:** "DenyCap controls minting." — No. DenyCap only manages the deny list. Minting is exclusively TreasuryCap.
**Checkpoint Q:** TreasuryCap and DenyCap stored in the same shared AdminRegistry behind the same AdminCap — two risks?
**Learner answer:** Loss of auth separation (can't grant compliance role without also granting mint rights) + centralization / single point of failure (one compromised key = attacker controls mint AND freeze). ✅
**Status:** ✅ understood

### Key Mnemonics
- **OTW = suicide pill**: One use, self-destructs on consumption. Like Rust move semantics — the value is gone the moment you pass it. The TYPE lives on as a label, the VALUE does not.
- **OTW forgery = Rust visibility**: You can't construct a private struct from another module in Rust either. Same rule, same enforcement level — compile-time wall, not a runtime guard.
- **TreasuryCap = mint key, not vault door**: It controls who can print money, not who can move it. Once printed, coins walk freely. The key only opens the printing press.
- **TreasuryCap shared = private key in a public repo**: The moment it's shared, everyone has it. And unlike a private key rotation, you can never un-share a Sui object.
- **MintCap delegation = RBAC without rate limiting**: Like granting a Minter role in a Solidity AccessControl system — but with no allowance cap. The Minter can mint 1 or 1 trillion. The cap is your responsibility, not the framework's.
- **split/join = breaking bills**: split a $100 into $60+$40. join them back to $100. Total never changes. Objects change count, value never changes.
- **put/take = internal memory move**: Like moving a value between two fields in the same Rust struct — no OS call, no hook fires. The boundary check only happens at the door (transfer), not inside the house.
- **DenyList boundary = customs at the airport**: Customs checks you at the gate (transfer boundary). Once you're inside the terminal (Balance operations), no one checks again.
- **Epoch delay = restraining order processing time**: Filed today, enforced tomorrow. Use global_pause for an instant injunction.
- **DenyCap = bouncer, TreasuryCap = printer**: Bouncer controls who gets in the club (deny list). Printer controls how much money exists (supply). Different jobs. Different keys. Never combine them.
- **Return Coin<T> = return the tool, not the result**: In Solidity you might emit an event. On Sui, you return the object so the PTB caller can decide what to do with it. Returning = composability. Transferring internally = dead end.
- **global_pause = circuit breaker**: Like a Solidity `pause()` with `whenNotPaused` modifier — instant halt. deny_list_v2_add is a slow block on one address. global_pause is a fast kill switch for everyone.

## Study Group Doc Corrections

Three misconceptions found in archive/docs/study_group — corrected and verified here.

### Correction 1: Why structs containing UID can't have `copy`
**Study group claim:** "UID is unique, so the VM prevents copying it at runtime."
**Correct:** This is the **ability infection rule** — a struct can only have an ability if every field has that ability. UID lacks `copy`. Therefore any struct containing UID cannot compile with `copy`. This is a compile-time type system rule, not a runtime uniqueness check.
**Source:** Move Book §5.7 Abilities
**Learner checkpoint:** ✅ "If a struct has `key` but no `store`, you can't nest it as a field in another struct." Correct — same rule.

### Correction 2: Why `phantom` exists in Move
**Study group claim:** "T is never physically stored (just a u64), so T remains a ghost."
**Correct:** Storage is the consequence, not the cause. `phantom` exists to **decouple ability requirements at compile time**. Without it, `Balance<T> has store` forces T to have `store`. OTW structs only have `drop` — they'd be rejected by the compiler. `phantom T` tells the compiler: "T is a type-level label only — don't require T to satisfy any abilities." This is what allows `Coin<MYTOKEN>`, `Balance<MYTOKEN>`, `TreasuryCap<MYTOKEN>` to work even though MYTOKEN only has `drop`.
**Source:** Move Book §5.24 Generics + §8.12 OTW
**Learner checkpoint:** ✅ "Without phantom, Balance<MYTOKEN> would fail because MYTOKEN only has drop, not store — and Balance has store, so the compiler would demand T: store." Correct. Every OTW-based coin would be unusable.

### Correction 3: Hot potato — compile-time, not runtime
**Study group claim:** "The Sui VM will look at the execution stack, see a Hot Potato still sitting there, and abort the whole transaction."
**Correct:** Hot potato enforcement is **compile-time**. The Move Book states explicitly: *"An attempt to ignore a struct without the drop ability will result in a compilation error."* The bytecode verifier enforces linear type rules at build time — code that doesn't consume a hot potato never deploys. The VM is never involved.
**Source:** Move Book — Drop Ability (move-book.com/move-basics/drop-ability) + §8.16 Pattern: Hot Potato
**Learner checkpoint:** ✅ "If the hot potato struct is not consumed by the designated function, it won't compile — can't deploy what won't compile."
**Status:** ✅ verified

### Key Mechanisms
1. **OTW supply invariant**: `coin::create_currency(otw, ...)` consumes the OTW by value. Struct construction is module-private → no forgery possible → exactly one TreasuryCap per type, forever. Source: Move Book §8.12
2. **MintCap delegation**: TreasuryCap wrapped in a shared object as a private field. `mint(auth: &mut MintAuthority, _cap: &MintCap, ...)` requires both — TreasuryCap never exposed, MintCap is the revocable delegation token. Source: Move Book §8.4
3. **DenyList bypass fix**: `coin::deny_list_v2_contains_current_epoch(&deny_list, addr, ctx)` must be called explicitly before any `coin::put` or `coin::take` in vault code. Framework only checks at transfer boundary. Source: Sui Framework coin module

### Key Distinctions
- `coin::put` / `coin::take` vs `transfer::public_transfer`: put/take are internal balance ops — no DenyList check. public_transfer IS the boundary — DenyList fires here.
- `deny_list_v2_add` vs `deny_list_v2_enable_global_pause`: add blocks one address, 1-epoch delay on receiving. global_pause freezes all transfers for the coin type, instantly.
- TreasuryCap vs DenyCap: TreasuryCap = supply authority (mint/burn). DenyCap = compliance authority (deny list). Neither controls the other.
- `split` vs `join`: split = one object → two objects (value conserved). join = two objects → one object (value conserved). Neither creates or destroys supply.
- OTW value vs OTW type: the value is consumed once in create_currency. The type lives on as a phantom tag in Coin<T>, TreasuryCap<T>. No value is ever instantiated again.

### Gotchas
- TreasuryCap shared = CRITICAL. Any transaction can mint or burn. No recovery path (objects can never be unshared).
- MintCap has no built-in quantity limits. Compromised downstream contract = unlimited minting.
- coin::put and coin::take do NOT trigger DenyList. Vault protocols must check explicitly.
- DenyList 1-epoch delay on receiving — attacker window exists between deny_list_v2_add and epoch rollover.
- OTW struct name must be SCREAMING_SNAKE_CASE of the module name exactly — mismatch = create_currency aborts.
- OTW forgery is compile-time blocked (module-private struct), not runtime checked.
- Internal transfer in mint (recipient: address) is a composability issue, not a security vulnerability — distinguish in audit reports.

## Key Takeaways

1. **TreasuryCap is a capability object — storage pattern determines security.** Owned = safe. Shared with no auth = CRITICAL. Wrapped + MintCap = correct delegation. One wrong line in init creates an unrecoverable vulnerability. Source: Sui Docs > Currencies and Tokens > Create Currencies and Tokens

The issuer of the capability is responsible for enforcing limits in the delegating contract. This applies to every capability in every W6/W8 pattern. Source: Move Book §8.4
