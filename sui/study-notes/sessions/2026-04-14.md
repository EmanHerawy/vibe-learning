## Theory Notes

### One-Time Witness as Coin Bootstrap Gate
**Source:** Move Book §8.12 — One-Time Witness — move-book.com/programmability/one-time-witness.html
**Summary:** OTW is a zero-field, drop-only struct whose name matches the module, passed by value to `init` exactly once by the Sui runtime. `coin::create_currency` consumes it, guaranteeing exactly one TreasuryCap per coin type ever exists. Forgery is impossible because struct instantiation is module-private in Move — `GOV {}` cannot be written outside the `gov` module.
**Bridged from:** W5 OTW pattern — today revealed WHY it was designed that way (supply invariant enforcement).
**Misconception flagged:** "OTW forgery is blocked by a runtime check." — Wrong. It's a compile-time guarantee: struct instantiation is module-private. The compiler rejects `GOV {}` outside the gov module before the code ever runs.
**Checkpoint Q:** Can a third module construct `GOV {}` and pass it to `create_currency` to fake a second TreasuryCap?
**Learner answer:** No — runtime checks prevent it. (Corrected: it's compile-time / type system, not runtime.)
**Status:** ✅ understood

### TreasuryCap — Supply Control Object
**Source:** Sui Docs > Currencies and Tokens > Create Currencies and Tokens + sui::coin module
**Summary:** TreasuryCap<T> is a capability object — whoever holds it can call coin::mint and coin::burn. It's the sole authority over supply. If shared, any transaction can mint freely. The correct pattern is owned (transferred to admin) or wrapped inside a module-private field with a delegated MintCap for downstream contracts.
**Bridged from:** W6 Capability pattern — MintCap wrapping TreasuryCap is the standard delegation architecture.
**Misconception flagged:** "TreasuryCap controls transfers." — No. It only gates creation and destruction of supply. Once minted, coins move freely.
**Checkpoint Q:** TreasuryCap is made a shared object at init — what's the vulnerability?
**Learner answer:** Anyone can mint and burn; it should be owned by the admin, not shared. ✅
**Status:** ✅ understood

### Coin<T> Lifecycle — mint / split / join / burn
**Source:** Sui Docs > Currencies and Tokens > Create Currencies and Tokens + sui::coin module
**Summary:** TreasuryCap mints Coin<T> objects (supply increases). split reshapes one Coin into two objects — value conserved. join merges two into one — value conserved. burn destroys a Coin and decreases total supply. The Balance<T> type underneath can only be created/destroyed through mint/burn — the type system enforces the invariant.
**Bridged from:** ERC-20 total supply invariant — same concept, enforced by type system instead of storage accounting.
**Misconception flagged:** "split creates new supply." — No. Split is value-conserving reshaping of the wrapping object only.
**Checkpoint Q:** User has Coin<USDC> value 1000, calls split(400). How many objects, what values, total supply change?
**Learner answer:** Two objects (600 + 400), total supply unchanged. ✅
**Status:** ✅ understood

### Mint Function Design — Return Coin<T>, Don't Transfer Internally
**Source:** Sui Docs > Currencies and Tokens > Create Currencies and Tokens (PTB composability pattern)
**Summary:** Mint functions should return `Coin<T>` to the caller rather than calling `transfer::public_transfer` internally. Returning lets PTBs chain the coin into further operations (deposit to pool, split, wrap). Internal transfer ends the PTB flow — the coin is gone and can't be used in the same transaction.
**Bridged from:** Solidity — like returning a value from a function vs emitting an event, the caller needs the handle to do something with it.
**Misconception flagged:** "Transferring to recipient: address inside mint is a vulnerability." — No. It's a composability/design anti-pattern, not a security exploit. The distinction matters in audit reports.
**Best practice:**
```
// ❌ Anti-pattern — internal transfer kills PTB composability
public fun mint(registry: &mut MintRegistry, amount: u64, recipient: address, ctx: &mut TxContext) {
    let coin = coin::mint(&mut registry.treasury, amount, ctx);
    transfer::public_transfer(coin, recipient);   // caller loses the coin handle
}

// ✅ Correct — return Coin<T>, let PTB or caller decide
public fun mint(registry: &mut MintRegistry, _cap: &MintCap, amount: u64, ctx: &mut TxContext): Coin<T> {
    coin::mint(&mut registry.treasury, amount, ctx)
}
```
**Status:** ✅ understood

### Regulated Coins & DenyCap
**Source:** Sui Docs > Currencies and Tokens > Regulated Currency and Deny List
**Summary:** create_regulated_currency returns TreasuryCap + DenyCap + CoinMetadata. DenyCap controls a per-type entry in the system-level DenyList object at 0x403 — blocked addresses cannot send or receive the token. DenyCap and TreasuryCap are separate authorities and should be held separately. Combining them in one shared object collapses least privilege and creates a single point of failure.
**Bridged from:** W6 — capability separation principle. Same as separating AdminCap from OperatorCap in multi-role protocols.
**Misconception flagged:** "DenyCap controls minting." — No. DenyCap only manages the deny list. Minting is exclusively TreasuryCap.
**Checkpoint Q:** TreasuryCap and DenyCap stored in the same shared AdminRegistry behind the same AdminCap — two risks?
**Learner answer:** Loss of auth separation (can't grant compliance role without also granting mint rights) + centralization / single point of failure (one compromised key = attacker controls mint AND freeze). ✅
**Status:** ✅ understood
