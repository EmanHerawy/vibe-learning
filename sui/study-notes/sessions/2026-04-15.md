## Theory Notes

### Events — Schema Design & Auto-Indexing Rules
**Source:** Move Book §8.15 — move-book.com/programmability/events + Sui Docs > Using Events
**Summary:** Event structs require `copy + drop`. Sui auto-indexes object creation and standard transfers — don't emit redundant events for those. Emit manually for state changes not visible via object metadata: cancellations, bids, price updates, purchase price/royalty data. Sender + timestamp are captured automatically — don't add them as fields. Type passed to `emit` must be internal to the module.
**Bridged from:** Solidity `emit Transfer(...)` — same off-chain signal concept, different rules on what's auto-indexed.
**Misconception flagged:** "Emit a Transfer event when transferring an NFT." — Sui already indexes transfers. Emit only for data Sui doesn't capture (price, royalty amount, reason for cancellation).
**Checkpoint Q:** Which of mint / list / delist / purchase / offer need manual events?
**Learner answer:** Mint = auto; list, delist, offer = custom; purchase = auto for transfer but custom needed for price/royalty data. ✅
**Status:** ✅ understood

### Object Display — Standard Fields & Publisher Authority
**Source:** Move Book §8.14 — move-book.com/programmability/display
**Summary:** Display stores a metadata template once — wallets substitute object field values at render time. Standard fields: name, description, image_url, thumbnail_url, link, project_url, creator. Fields are not enforced by the framework — wallets follow convention. Only the Publisher<T> holder can create or update Display<T>. Lost Publisher = metadata frozen forever.
**Bridged from:** Solidity baseTokenURI pattern — same "store once, derive per token" principle, but Display is on-chain and updatable.
**Misconception flagged:** "Display is required to create an NFT." — No. Display only affects wallet rendering. Missing Display = broken UI, not broken contract.
**Checkpoint Q:** Image server moves domain — how do you fix 10,000 NFTs at once? What do you need?
**Learner answer:** Update the Display object using Publisher cap — one update fixes all NFTs. ✅
**Status:** ✅ understood

### Kiosk + TransferRequest + Transfer Policy Rules
**Source:** Sui Docs > Standards > Kiosk — docs.sui.io/standards/kiosk + week9.md (PurchaseCap corrected to TransferRequest<T>)
**Summary:** kiosk::purchase returns (T, TransferRequest<T>). TransferRequest is a hot potato — must be resolved via transfer_policy::confirm_request in the same transaction. Rules (e.g. royalty) add receipts to the request; confirm_request checks all receipts present. NFT needs `key + store` to be placed in Kiosk. Critical bypass: if NFT has `store`, it can be wrapped in another struct and transferred without Kiosk — Transfer Policy is advisory, not enforceable for `store` types.
**Bridged from:** W3 hot potato pattern — same compile-time enforcement. W4 wrapping — store ability enables wrapping bypass.
**Misconception flagged:** "Hot potato makes royalties unbypassable." — Only if the NFT lacks `store`. With `store`, wrap-and-transfer bypasses Kiosk entirely.
**Checkpoint Q:** How can a buyer bypass a 5% royalty rule?
**Learner answer:** Wrap the NFT inside another struct and transfer the parent — Kiosk and TransferPolicy never involved. ✅
**Status:** ✅ understood

### Soulbound NFT Pattern
**Source:** Sui Docs > NFTs > Soulbound NFT
**Summary:** Soulbound = `key` only, no `store`, no exposed transfer functions. No `store` means unwrappable and Kiosk-incompatible. Type system enforces it — no code path to move the object exists. Display still works independently of abilities — wallets render soulbound NFTs correctly.
**Bridged from:** Ability infection rule (W3/corrections) — ability choice at struct definition time determines all downstream behavior.
**Misconception flagged:** "Soulbound = add a frozen flag and check it in transfer." — Bypassable via public_transfer. True soulbound requires no store + no exposed transfer functions.
**Checkpoint Q:** Achievement badge — never transferable, visible in wallets. What abilities? Need Kiosk?
**Learner answer:** key only, no Kiosk needed. ✅
**Status:** ✅ understood

### Key Mnemonics
- **Event abilities = print & toss**: copy = make a copy for the receipt printer. drop = toss it after printing. Events are receipts — printed and discarded, never filed.
- **Auto-index trap = don't shout what the PA already announced**: Sui's PA system already announces object creation and transfers over the intercom. Emitting your own "NFT minted" event is shouting the same thing twice. Only announce what the PA missed.
- **Sender + timestamp = free postage**: Sui stamps every event with sender + timestamp for free. Adding them yourself is paying for stamps you already got.
- **Display = movie poster, not the movie**: The poster (Display) tells you what the film looks like. The film (NFT) works with or without a poster. Missing poster = nobody knows it exists, but the film still runs.
- **7 Display fields = "NDIT-LC"**: Name, Description, Image_url, Thumbnail_url, Link, Creator — the six visible things. Project_url is the creator's homepage.
- **Publisher = deed to the whole building**: Not just one apartment (module) — the entire building (package). One deed covers every room.
- **Lost Publisher = lost master key with no copy**: The master key is gone. Every lock stays as-is forever. No locksmith can help — the building's design doesn't allow new keys.
- **TransferRequest = hot potato from the cashier**: You buy the NFT, cashier hands you the item + a hot potato receipt. You MUST take that receipt to the policy window before leaving. Drop it = transaction vaporizes.
- **confirm_request = exit stamp on the receipt**: The policy window stamps every rule on the receipt. Missing one stamp → security won't let you out.
- **store = the door that lets you wrap**: store ability = object can go inside other objects. Kiosk bypass = sneak the NFT out wrapped inside a box. No store = no box available = no bypass.
- **Soulbound = born without legs**: A soulbound NFT has no store — it was born unable to move. Not because something stops it, but because it literally has no mechanism to go anywhere. Frozen flag = wheelchair with an ejector seat. key only = no legs, period.

## One-Paragraph Summary

Today I learned how Sui's NFT infrastructure fits together as a system. Events are off-chain signals that require copy + drop abilities — Sui auto-indexes object creation and transfers, so you only emit manually for state changes with data Sui doesn't capture (price, cancellation reason, royalty amount). Object Display is a template stored once that wallets use to render any NFT of a type — controlled by the Publisher object, which is package-scoped and irreplaceable if lost. Kiosk enforces royalties through a TransferRequest hot potato that must be resolved in the same transaction, but this enforcement only holds if the NFT lacks the `store` ability — with `store`, any buyer can wrap the NFT and transfer it peer-to-peer, bypassing Kiosk entirely. True soulbound NFTs have `key` only and no exposed transfer functions — the type system makes movement structurally impossible, not just checked.

### Key Mechanisms
1. **Event emission rule**: Emit only for state changes not visible via object metadata. Sui auto-captures sender + timestamp. Type must be internal to the module. Source: Move Book §8.15
2. **Display template**: `display::new_with_fields<T>(&publisher, keys, values, ctx)` + `update_version`. Publisher = package proof. One update fixes all instances. Source: Move Book §8.14
3. **Kiosk purchase flow**: `kiosk::purchase` returns `(T, TransferRequest<T>)`. Rules add receipts to TransferRequest. `confirm_request` checks all receipts present or aborts. Source: Sui Docs > Standards > Kiosk

### Key Distinctions
- Auto-indexed vs manual events: creation + transfers = auto. Price, cancellation, bid = manual.
- Display present vs absent: absent = broken wallet UI, not broken contract
- `key + store` vs `key` only: store = Kiosk-compatible + wrappable (bypass risk). key only = soulbound + no bypass
- TransferRequest vs PurchaseCap: TransferRequest<T> is the correct struct name (week9.md had it wrong)
- Kiosk enforcement vs module enforcement: Kiosk = advisory with store. Module-controlled = enforceable without store
- Publisher = package deed (not module deed)

### Gotchas
- Sender + timestamp auto-captured — never add as event fields (gas waste)
- Display fields not enforced — wallets follow convention, you can add custom fields but they won't render
- store ability = Transfer Policy is advisory — wrapping bypass always possible
- Lost Publisher = permanent metadata freeze — treat like a private key
- Soulbound via flag = false security — frozen flag bypassable if store exists
- PurchaseCap (week9) → correct name is TransferRequest<T>

## Key Takeaways

1. **`store` ability = Transfer Policy is advisory.** Any NFT with `store` can be wrapped and transferred outside Kiosk. The only enforceable royalty requires removing `store` and building module-controlled transfers. Source: Sui Docs > Standards > Kiosk — docs.sui.io/standards/kiosk

2. **Display is a template, not a requirement.** One Publisher update fixes all NFTs instantly. Lose the Publisher and metadata is frozen forever. Treat Publisher like a private key. Source: Move Book §8.14 — move-book.com/programmability/display

3. **Sui auto-indexes creation and transfers — only emit what it misses.** Price paid, royalty amount, cancellation reason, old state values (old_level) — these are what events are for. Sender + timestamp are free and automatic. Source: Move Book §8.15 — move-book.com/programmability/events

**The one thing I must not forget from today:**
  "`store` ability is the royalty bypass. Every NFT audit starts with checking whether `store` is present and documenting that Transfer Policy enforcement is advisory if it is."
