## Theory

### BCS Encoding
> Source: Move Book ¬ß8.17 ‚Äî move-book.com/programmability/bcs

**What it is:** Serializes a Move struct to a deterministic byte vector. Not self-describing ‚Äî no field names, no type tags, just raw bytes in struct definition order.

**Primitive sizes:** `u8` = 1 byte ¬∑ `u64` = 8 bytes LE ¬∑ `address` = 32 bytes ¬∑ `vector<T>` = ULEB128 length + elements

**Critical rule:** Decoder must peel fields in exact struct definition order.
```move
// struct MintPermit { user: address, amount: u64 }
let user   = decoder.peel_address();  // ‚Üê field 1
let amount = decoder.peel_u64();      // ‚Üê field 2
```

**Gotcha ‚Äî BCS vs Solidity ABI:**

| | BCS (Sui/Move) | ABI (Solidity) |
|---|---|---|
| Self-describing? | ‚ùå No | ‚úÖ Yes |
| Wrong order ‚Üí | Silent garbage | Reliable revert |
| Schema embedded? | ‚ùå No | ‚úÖ Yes |

Wrong peel order may cause a late abort *or* silently return corrupt values ‚Äî behavior is unpredictable. The learner has seen Solidity ABI.decode() revert reliably; BCS does not have that guarantee.

### Hash Functions
> Source: Sui Docs > Cryptography > Hashing ‚Äî docs.sui.io/guides/developer/cryptography/hashing

| Module | Function | Input | Use when |
|--------|----------|-------|----------|
| `std::hash` | `sha2_256(bytes)` | owned `vector<u8>` | standard backends |
| `std::hash` | `sha3_256(bytes)` | owned `vector<u8>` | OpenSSL-default systems |
| `sui::hash` | `keccak256(&bytes)` | `&vector<u8>` | EVM / cross-chain with Ethereum |
| `sui::hash` | `blake2b256(&bytes)` | `&vector<u8>` | performance-critical |

**Selection rule:** Match the off-chain signer's hash function. Mismatch ‚Üí always-false verification (no error, just never works).

### On-Chain Signature Verification
> Source: Sui Docs > Cryptography > Signing ‚Äî docs.sui.io/guides/developer/cryptography/signing

**Flow:**
```
Off-chain:  struct ‚Üí bcs::to_bytes() ‚Üí hash ‚Üí sign with private key ‚Üí 64-byte sig
On-chain:   struct ‚Üí bcs::to_bytes() ‚Üí hash ‚Üí ed25519_verify(&sig, &pk, &hash) ‚Üí bool
```

**Code:**
```move
let encoded  = bcs::to_bytes(&permit);
let msg_hash = hash::keccak256(&encoded);
let valid    = ed25519::ed25519_verify(&sig, &config.admin_pk, &msg_hash);
assert!(valid, E_INVALID_SIG);
```

**Sui vs Ethereum:**

| | Sui | Ethereum |
|---|---|---|
| Function | `ed25519_verify(&sig, &pk, &hash)` | `ecrecover(hash, v, r, s)` |
| You provide | the expected public key | nothing (it's recovered) |
| Returns | `bool` | signer address |
| Algorithm | Ed25519 | ECDSA secp256k1 |

### Replay Attacks
> Source: Sui Docs > Cryptography > Signing

**Three required defenses ‚Äî all must be present:**

| Defense | Prevents | Implementation |
|---------|----------|----------------|
| Nonce | Same-function replay (double-submit) | `Table<u64, bool>` of used nonces |
| Expiry | Leaked/stale signature reuse | `assert!(epoch <= permit.expires_at)` |
| Domain separation | Cross-function replay (mint sig ‚Üí burn) | `action: vector<u8>` field in permit |

---

### Domain Separation
> Source: Sui Docs > Cryptography > Signing

Without it: `MintPermit { user, amount }` produces identical bytes whether used for mint or burn ‚Üí same signature validates both functions.

Fix: add `action: vector<u8>` to the struct.
- Mint flow signs over `b"mint"` prefix ‚Üí produces hash H1
- Burn flow signs over `b"burn"` prefix ‚Üí produces hash H2
- H1 ‚âÝ H2 ‚Üí signatures are not interchangeable

**Also bind to contract address** ‚Äî Move doesn't prevent cross-contract replay automatically. Two contracts accepting the same struct + key will accept the same signatures.

## Key Takeaways

1. **BCS is field-order-critical and not self-describing.** Wrong peel order ‚Üí silent corruption, not a clean revert. Always decode in exact struct definition order.
   *Source: Move Book ¬ß8.17 BCS*

2. **A valid signature proves the signer approved a message. It says nothing about how many times, which function, or which chain.** Nonce, domain separation, and chain_id are your responsibility to enforce.
   *Source: Sui Docs > Cryptography > Signing*

3. **Chain ID ‚âÝ replay safe. Nonce prevents double-submission on the same chain.** Both required. This is the pattern behind most bridge hacks.
   *Source: Sui Docs > Cryptography > Signing*

4. **‚öÝÔ∏è CRITICAL GOTCHA ‚Äî All-zero public key + all-zero signature returns `true` in Sui's `ed25519_verify`.**
   Discovered during hands-on testing (2026-04-19). The all-zero 32-byte key decodes to the Ed25519 identity point (‚àû). The verification equation `8¬∑S¬∑B == 8¬∑R + 8¬∑h¬∑A` becomes `‚àû == ‚àû` ‚Äî always true, for any message.
   Sui's implementation does **not** reject the identity point (no small-subgroup check).

   ```move
   // THIS RETURNS TRUE ‚Äî no key needed, full auth bypass
   ed25519_verify(
     &x"0000...00",  // 64-byte zero sig
     &x"0000...00",  // 32-byte zero pubkey
     &any_hash
   )
   ```

   **Two compounding bugs if the caller supplies the public key:**
   - Bug 1: Zero key bypasses verification entirely (no private key needed)
   - Bug 2: Caller-supplied pubkey means anyone can sign with their own key and pass it in

   **Fix: always load the trusted key from on-chain state ‚Äî never accept it as a parameter.**
   ```move
   // ‚ùå vulnerable ‚Äî caller controls pubkey
   public fun verify(sig: vector<u8>, pubkey: vector<u8>, hash: vector<u8>): bool {
       ed25519::ed25519_verify(&sig, &pubkey, &hash)
   }

   // ‚úÖ safe ‚Äî pubkey is pinned in a shared object
   public fun verify(config: &Config, sig: vector<u8>, hash: vector<u8>): bool {
       ed25519::ed25519_verify(&sig, &config.authorized_pubkey, &hash)
   }
   ```

5. **‚úÖ PATTERN ‚Äî Lock `TreasuryCap` inside a shared `Config` object to enable permissionless self-execution of admin-signed mint permits.**

   The key insight: `TreasuryCap` requires module authority to mint, but doesn't need to live in the admin's wallet. Wrapping it in a shared object lets any user trigger a mint by presenting a valid off-chain signature ‚Äî without the admin being online per tx.

   **Why it's safe:**
   - `Config` has `key` but NOT `store` ‚Üí cannot be wrapped, transferred, or extracted from
   - Struct fields are module-private ‚Üí no external code can touch `treasury_cap` directly
   - The only function that uses it (`mint_with_permit`) is guarded by sig verification

   ```move
   public struct Config has key {           // key only ‚Äî no store
       id: UID,
       treasury_cap: TreasuryCap<MY_COIN>, // locked here forever
       admin_pubkey: vector<u8>,            // pinned ‚Äî never caller-supplied
       used_nonces:  Table<u64, bool>,      // replay protection
       max_supply:   u64,                   // hard ceiling
   }

   fun init(witness: MY_COIN, ctx: &mut TxContext) {
       let (_, treasury_cap) = coin::create_currency(...);
       // ‚úÖ wrap in shared object ‚Äî NOT transferred to wallet
       transfer::share_object(Config { treasury_cap, ... });
   }

   public fun mint_with_permit(config: &mut Config, permit: MintPermit, sig: vector<u8>, ctx: &mut TxContext) {
       // verify sig against config.admin_pubkey (pinned, not caller-supplied)
       // then: coin::mint(&mut config.treasury_cap, amount, ctx)
   }
   ```

   **Guard `set_admin_pubkey` with an `AdminCap` object** (no `store`) ‚Äî otherwise any caller can front-run the deployer and hijack minting authority.

   **Checklist for this pattern:**
   - [ ] `Config` has `key` but not `store`
   - [ ] `AdminCap` has `key` but not `store` (non-transferable)
   - [ ] Public key loaded from `config.admin_pubkey` ‚Äî never from function args
   - [ ] Nonce table prevents replay
   - [ ] `max_supply` cap prevents unbounded minting
   - [ ] Domain separation: `action: vector<u8>` field in permit (b"mint" ‚âÝ b"burn")

> "A signature proves the signer approved a message. It proves nothing about how many times it can be used, which function it was meant for, or which chain it was intended for. Every one of those is your problem to enforce."
