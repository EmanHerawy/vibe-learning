## Theory Notes

### BCS Encoding
**Source:** Move Book §8.17 BCS — move-book.com/programmability/bcs
**Summary:** BCS (Binary Canonical Serialization) encodes Move structs into a deterministic byte vector. Fields are serialized in struct definition order. The encoding is not self-describing — the decoder must know the exact layout to decode correctly. Primitives: u8=1 byte, u64=8 bytes LE, address=32 bytes. Vectors prepend a ULEB128 length prefix. Structs are just their fields concatenated with no wrapper or field names.
**Bridged from:** Ethereum ABI encoding — same idea (structs → bytes for hashing), but ABI is schema-aware and padded; BCS is compact and order-critical. ABI appends field names and types to the encoding; BCS is raw bytes only.
**Key clarification (from learner question):** Decoder must peel fields in exact struct definition order — peel_address() before peel_u64() if address is defined first. BCS has no field tags. If you peel in wrong order, you get corrupt data silently (wrong value, no error). Like reading a binary file with the wrong struct layout.
**Misconception flagged:** "I can decode fields in any order" — No. BCS has no schema embedded. If struct defines (address, u64), you must call peel_address() then peel_u64(). Reversing the calls gives garbage without aborting.
**Checkpoint Q:** You encoded `Permit { user: address, amount: u64 }`. Write the decode sequence.
**Learner answer:** peel_address() first (address is first field), then peel_u64() — because if encoding doesn't match, a different hash is generated. ✅ Also correctly caught tutor code showing reversed order.
**Status:** ✅ understood

### Hash Functions on Sui
**Source:** Sui Docs > Cryptography > Hashing — docs.sui.io/guides/developer/cryptography/hashing; Move Book §std::hash
**Summary:** Two modules provide hashing. `std::hash` (sha2_256, sha3_256) takes owned `vector<u8>` and returns `vector<u8>`. `sui::hash` (keccak256, blake2b256) takes `&vector<u8>` by reference and returns `vector<u8>`. In practice all are interchangeable on Sui — the distinction is API ergonomics (ownership vs reference). Choice of function is governed by cross-chain compatibility, not security: keccak256 for EVM interop, sha2_256 for standard/Ethereum non-EVM, sha3_256 for general off-chain systems (OpenSSL default), blake2b256 for performance. All four are cryptographically adequate for signature verification.
**Bridged from:** Ethereum — keccak256 is the native EVM hash (used in ecrecover). On Sui, keccak256 is in sui::hash, not a native opcode.
**Key clarification (from learner question):** Hash function selection rule = "what does the off-chain signer use?" If the backend signs with keccak256, the contract must verify with keccak256. Mismatched hash functions = different digest = verify() always returns false. Cross-chain: if passing Ethereum-signed data to Sui, use keccak256 to reproduce the same digest. Real hacks (Wormhole, Ronin) involved sig verification bugs but not hash function mismatch — deferred to W15 audit reports.
**Key clarification (Sui vs Ethereum):** Sui exposes ed25519_verify() in Move directly. Ethereum uses ecrecover() (ECDSA secp256k1 → recovered address). Different algorithms: Ed25519 (fixed-size 64-byte sig, 32-byte pk, boolean output) vs ECDSA secp256k1 (variable-length, address output). On Sui, the function returns bool — no address recovery step needed.
**Misconception flagged:** "Any hash function works — they're all secure." — Technically correct for collision resistance, but the choice must match the off-chain signer. Wrong choice = always-false verification (silently broken, not an error).
**Status:** ✅ understood

### On-Chain Signature Verification
**Source:** Sui Docs > Cryptography > Sui On-Chain Signatures — docs.sui.io/guides/developer/cryptography/signing
**Summary:** Standard flow: (1) off-chain: serialize struct to BCS bytes, hash with chosen function → 32-byte digest; (2) off-chain: sign digest with private key → 64-byte Ed25519 signature; (3) on-chain: re-encode same struct to bytes, re-hash → same digest; (4) on-chain: call `sui::ed25519::ed25519_verify(&sig, &pk, &msg_hash)` → bool. The contract never sees the private key. The contract holds or receives the public key. Verification confirms the signer owned the private key corresponding to the known public key.
**Bridged from:** Backend JWT verification — same pattern: sign on one side, verify on the other using a public key. The server never sees the private key; it only checks the signature against the known pubkey.
**Key mechanism:**
```move
use sui::ed25519;
use sui::hash;
use sui::bcs;

// Re-encode the permit to bytes
let encoded = bcs::to_bytes(&permit);
// Hash it
let msg_hash = hash::keccak256(&encoded);
// Verify the signature
let valid = ed25519::ed25519_verify(&sig, &admin_pk, &msg_hash);
assert!(valid, E_INVALID_SIGNATURE);
```
**Misconception flagged:** "The contract recovers the signer address from the signature." — That's Ethereum (ecrecover). Sui's ed25519_verify takes the public key as input and returns bool. You must already know whose key to check against.
**Status:** ✅ understood

### Replay Attacks on Permit Patterns
**Source:** Sui Docs > Cryptography > Sui On-Chain Signatures — docs.sui.io/guides/developer/cryptography/signing
**Summary:** A replay attack reuses a valid signature in a context the signer didn't intend. If a permit has no expiry and no consumed flag, the attacker saves the signed bytes and submits them again (same tx, later time, or different function). Three required replay defenses: (1) nonce or UID — one-time-use marker consumed on first use; (2) expiry timestamp — signature invalid after a deadline; (3) domain separation — signature binds to one specific function or action.
**Bridged from:** Ethereum EIP-2612 permit() — same replay problem. EIP-2612 added nonce + deadline + domain separator to fix it. Sui doesn't have a built-in standard equivalent but the same three defenses apply.
**Checkpoint Q:** What should a replay-safe permit struct include?
**Learner answer:** Check for expiry time and if it is valid nonce or unused signature — replay attack. ✅
**Status:** ✅ understood

### Domain Separation
**Source:** Sui Docs > Cryptography > Sui On-Chain Signatures — docs.sui.io/guides/developer/cryptography/signing
**Summary:** Domain separation adds a function-specific prefix to the signed message so a signature for one action cannot be replayed as a different action. Implemented by including a `action: vector<u8>` or string tag in the signed struct, or by hashing a domain tag alongside the payload. Without it, a mint permit and a burn permit with the same struct layout are cryptographically identical — the same admin signature validates both.
**Bridged from:** EIP-712 domainSeparator — same concept. EIP-712 binds each signature to a specific contract, chain ID, and function type. Sui has no native EIP-712 equivalent; implement manually via a tag in the signed struct.
**Misconception flagged:** "Different function names protect against cross-function replay." — No. The contract code checks the function name at call time, but the *signature* covers only the serialized bytes of the permit struct. If two functions accept the same struct and the same key, they accept the same signatures.
**Checkpoint Q:** A protocol has mint_with_permit and burn_with_permit, both using Permit { user, amount }, same admin key. What can an attacker do without domain separation?
**Learner answer:** Replay attack — attacker reuses valid mint signature to burn user's assets. ✅
**Status:** ✅ understood

### Key Mnemonics
- **BCS field order = reading a ZIP file with the wrong extraction tool**: the tool doesn't know the format, it just reads bytes. Wrong order → garbage file, no error message.
- **BCS silent corruption = wrong protein folding**: the chain of amino acids (bytes) is the same, but folded wrong (decoded wrong) → produces a broken protein (corrupt value). No alarm goes off.
- **Hash function selection = "ask the other side first"**: whoever signs chooses the hash. You must match them. Mismatching is like two people agreeing to meet but one uses local time and the other uses UTC — you compute the same formula but get different answers.
- **ed25519_verify = bouncer with a guest list**: you hand the bouncer the signature (proof), the public key (the guest's ID), and the message (what they claimed). Bouncer checks: does this signature match this key for this message? → true/false. Ethereum ecrecover: bouncer checks the signature and tells you who signed (address recovery). Sui bouncer already knows who they're checking.
- **Replay attack = a concert ticket photocopied**: the original was valid. The copy looks identical. Without a torn-stub system (nonce) or a date stamp (expiry), the door accepts every copy.
- **Domain separation = different colored ink for different venues**: mint permit = blue ticket, burn permit = red ticket. Same band (admin key), same paper (struct), but colored ink (action tag) tells the bouncer which door to use.
- **Bridge nonce = postal tracking number**: from/to address on the envelope says where it goes. But without a unique tracking ID, the same package can be delivered twice — and the recipient pays twice.

## One-Paragraph Summary

Today I learned how Sui's on-chain signature verification works: off-chain, serialize a permit struct to BCS bytes, hash with the chosen function, sign with a private key; on-chain, re-encode the same struct, re-hash, and call ed25519_verify(&sig, &pk, &hash) → bool. BCS is field-order-critical and not self-describing — wrong peel order produces silent data corruption, not a clean revert (unlike Solidity ABI.decode). Hash function selection is driven by cross-chain compatibility: match whatever the off-chain signer uses. Replay attacks require three defenses: nonce (one-time use), expiry timestamp, and domain separation (action tag in the signed struct to prevent cross-function reuse). On a real audit, I found 4 vulnerabilities in a whitelist mint module: cross-chain replay (no chain_id), cross-function replay (no domain separation), no nonce, no expiry, and one missed finding — burn amount not validated against the permit.

## One-Page Summary

### What We Covered
- BCS encoding: deterministic, field-order-critical, not self-describing
- BCS decode: peel in exact struct definition order; wrong order → silent corruption
- Hash function APIs: std::hash (owned), sui::hash (by ref); selection = match off-chain signer
- Hash selection rules: keccak256 (EVM), sha2 (standard), sha3 (OpenSSL), blake2b (performance)
- ed25519_verify(&sig, &pk, &hash) → bool — provide the pubkey, get back true/false
- Sui vs Ethereum: Ed25519 + bool return vs ECDSA secp256k1 + address recovery
- Replay attack: reusing a valid sig in unintended context (same function, different function, different chain)
- Three defenses: nonce, expiry, domain separation
- Domain separation: action tag in signed struct binds sig to one function/contract
- Bridge replay: chain_id ≠ replay safe — nonce still required for double-submission prevention
- Cross-contract replay: Move doesn't prevent it automatically — domain separation required

### Key Mechanisms
1. **Sig verification flow**: `bcs::to_bytes(&permit)` → `hash::keccak256(&bytes)` → `ed25519::ed25519_verify(&sig, &pk, &hash)`. Source: Sui Docs > Cryptography > Signing
2. **BCS field order**: decoder peels in exact struct definition order. No type tags — silent corruption on mismatch (not guaranteed abort). Source: Move Book §8.17 BCS
3. **Domain separation**: include `action: vector<u8>` in the signed struct. `b"mint"` and `b"burn"` produce different hashes → different valid signatures per function. Source: Sui Docs > Cryptography > Signing

### Key Distinctions
- BCS vs Solidity ABI: BCS = raw bytes, no schema, silent corruption on wrong order. ABI = self-describing, reliable revert on mismatch.
- BCS wrong order: may silently corrupt OR abort late — behavior is unpredictable.
- ed25519_verify (Sui) vs ecrecover (Ethereum): Sui = provide expected pubkey → bool. Ethereum = provide nothing, recover signer address.
- Chain ID vs nonce: chain ID prevents cross-chain replay. Nonce prevents double-submission on the same chain. Both required.
- Domain separation vs nonce: domain separation prevents cross-function replay. Nonce prevents same-function replay. Both required.

### Gotchas
- BCS wrong peel order → silent garbage, not a revert (unlike Solidity)
- Wrong hash function → always-false verification (no error — just never works)
- admin_pk in AdminCap doesn't eliminate sig verification if users self-execute permits
- Move safety ≠ cross-contract replay protection (Move prevents reentrancy, not this)
- bridge from_chain/to_chain ≠ replay safe (nonce still needed for double-submission)

### Code / Patterns to Remember
```move
// Replay-safe permit pattern
public struct MintPermit has drop {
    action: vector<u8>,         // domain separation: b"mint"
    contract: address,          // contract binding
    chain_id: u64,              // cross-chain replay prevention
    user: address,
    amount: u64,
    nonce: u64,                 // one-time use
    expires_at: u64,            // expiry
}

// On-chain verification
let encoded = bcs::to_bytes(&permit);
let msg_hash = hash::keccak256(&encoded);
let valid = ed25519::ed25519_verify(&sig, &config.admin_pk, &msg_hash);
assert!(valid, E_INVALID_SIG);
assert!(permit.action == b"mint", E_WRONG_ACTION);       // domain separation
assert!(tx_context::epoch(ctx) <= permit.expires_at, E_EXPIRED);
assert!(!table::contains(&config.used_nonces, permit.nonce), E_REPLAY);
table::add(&mut config.used_nonces, permit.nonce, true);
// THEN check amount before using it
assert!(coin::value(&incoming) == permit.amount, E_AMOUNT_MISMATCH);
```
Source: Sui Docs > Cryptography > Signing, Move Book §8.17 BCS

## Key Takeaways

1. **BCS is field-order-critical and not self-describing.** Wrong peel order produces silent data corruption — not a clean revert like Solidity ABI.decode. Always decode in exact struct definition order. Source: Move Book §8.17 BCS — move-book.com/programmability/bcs

2. **A valid signature is not enough — check what it actually authorizes.** Domain separation, nonce, expiry, and amount enforcement are all required. Sig validity only proves the signer approved *some* message. The contract must verify it approved *this specific* action with *these exact parameters*. Source: Sui Docs > Cryptography > Signing — docs.sui.io/guides/developer/cryptography/signing

3. **Chain IDs prevent cross-chain replay. Nonces prevent same-chain double-submission. Both are required.** A bridge message with from_chain + to_chain but no nonce can still be submitted multiple times on the correct chain. Source: Sui Docs > Cryptography > Signing

**The one thing I must not forget from today:**
  "A signature proves the signer approved a message. It proves nothing about how many times it can be used, which function it was meant for, or which chain it was intended for. Every one of those is your problem to enforce."
