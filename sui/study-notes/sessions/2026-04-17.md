## Theory

### Two-Part Fee Structure
> Source: Sui Docs > Gas in Sui — docs.sui.io/concepts/tokenomics/gas-in-sui

| Fee | What it covers | Rate |
|-----|---------------|------|
| Computation | CPU work to execute the tx | Dynamic — set each epoch by validators via RGP |
| Storage | Bytes written on-chain | Fixed: 1 byte = 100 storage units |

`Net cost = computation + storage_written - (99% × storage_deleted)`

**Vs Ethereum:** Ethereum charges computation only. Sui charges both. Storage fee can dwarf computation for data-heavy operations.

**Gotcha — gas budget:** Budget = ceiling, not charge. If actual cost < budget → you pay actual cost. If actual cost > budget → tx aborts. Validators never keep the excess.

**Checkpoint:** Lending protocol stores all historical loans forever — who pays, better design?
**Answer:** User who creates the loan pays storage at creation; no rebate since never deleted. Better: `close_loan()` deletes the object (99% back) + use events for historical data. ✅

---

### Reference Gas Price (RGP)
> Source: Sui Docs > Gas in Sui — docs.sui.io/concepts/tokenomics/gas-in-sui

Each epoch: all validators submit reservation prices → protocol picks **2/3 stake-weighted percentile** = RGP.

| Price relative to RGP | Consequence |
|-----------------------|-------------|
| At or below | Boosted staking rewards |
| Above | Reduced staking rewards |

No hard cap — constraint is economic. Coordinated price attack requires controlling >2/3 stake (same threshold as network takeover).

### Storage Rebates
> Source: Sui Docs > Gas in Sui — docs.sui.io/concepts/tokenomics/gas-in-sui

- Delete a Sui object → **99% of original storage fee returned** to whoever submits the delete tx
- 1% kept permanently in the Storage Fund (non-refundable)
- Ephemeral pattern: `create()` → `use()` → `delete()` ≈ only computation cost + 1% of storage

**Hot potato storage cost:** A struct with no `key` ability never enters the Sui object store → **zero storage fee**. Computation only.
> Source: Move Book §8.16 Hot Potato — move-book.com/programmability/hot-potato

**Key clarification:** Whoever calls `table::add()` pays the storage, not "the protocol." Usually the user submitting the tx.

### Shared vs Owned — Performance & Cost
> Source: Sui Docs > Gas in Sui — docs.sui.io/concepts/tokenomics/gas-in-sui · Sui Docs > Object Ownership — docs.sui.io/concepts/object-ownership

| | Owned (fast path) | Shared (consensus) |
|---|---|---|
| Who can touch | Only the owner | Anyone |
| Execution | Skips consensus | Multi-round validator agreement |
| Latency | ~100ms | ~500ms+ |
| Under load | Parallel, no bottleneck | Serialized queue per object |
| Gas | Lower | Higher |

**Misconception:** "Shared objects are just slower." Not just latency — under load, a single hot shared object serializes ALL transactions through one consensus queue.

**Fix — Sharding:** Deploy N copies of the shared object, route by `hash(key) % N`. Each shard has its own consensus queue → N× throughput.
> Source: Sui Docs > Gas Optimization — docs.sui.io/guides/developer/sui-101/gas-optimization

---

### Compound Vulnerability — Shared Object + Unbounded Iteration
> Source: Move Book §8.6 — move-book.com/programmability/collections · Sui Docs > Gas in Sui

Shared `Table` of all stakers + `distribute_rewards()` loop = two independent CRITICALs that stack:

| Vulnerability | Alone | Combined |
|--------------|-------|---------|
| Unbounded iteration | CRITICAL — gas grief, fund lock | Attacker inflates Table cheaply via owned-object txs |
| Shared object consensus | HIGH — slow, serialized | Each distribute call needs consensus + full iteration |
| Together | — | Attacker cost: cheap. Protocol cost: catastrophic. |

**Fix:** Both required — pull-over-push (removes the loop) **AND** per-user owned objects (removes consensus overhead). Fixing one leaves the other.

**Key mnemonics:**
- **Two-part gas = FedEx bill** — handling fee (computation) + warehouse rent (storage). Ethereum only charges handling.
- **Gas budget = restaurant tab limit** — if food costs $40 on a $100 limit, you pay $40. Waiter keeps nothing.
- **Storage rebate = rental locker deposit** — 99% back when you return the key. 1% admin fee, gone forever.
- **RGP = airport taxi price board** — taxis post their price; board shows the 2/3 stake fare. Overcharge = lose customers + bonus points.
- **Hot potato = sticky note** — lives in your hand, no locker, no rent. No `key` ability = no object store entry = zero storage.
- **Shared = airport security checkpoint** — everyone queues through one scanner. 1000 people = 1000 sequential scans.
- **Sharding = opening multiple checkout lanes** — 16 shards = 16 parallel consensus queues = 16× throughput.
- **Compound vulnerability = flooding + fire** — unbounded iteration drowns the protocol; shared object slows it. Together: neither escape nor shelter is possible. Fix both.

## Key Takeaways

1. **Storage fee is a deposit.** Delete objects → 99% back. Ephemeral patterns and cleanup are near-free by design.
   *Source: Sui Docs > Gas in Sui — docs.sui.io/concepts/tokenomics/gas-in-sui*

2. **Shared objects serialize all traffic.** One hot shared object under load bottlenecks the entire protocol. Shard when load is high.
   *Source: Sui Docs > Gas in Sui — docs.sui.io/concepts/tokenomics/gas-in-sui*

3. **Shared Table + unbounded loop = compound CRITICAL.** Fix requires pull-over-push AND owned objects. Fixing one leaves the other attack surface open.
   *Source: Move Book §8.6 + Sui Docs > Gas in Sui*

> "Storage fee is a deposit. Gas budget is a ceiling. Shared objects cost consensus. Delete to reclaim. These four sentences are the entire gas model."
