## Theory Notes

### Sui Gas Model — Two-Part Fee Structure
**Source:** Sui Docs > Gas in Sui — docs.sui.io/concepts/tokenomics/gas-in-sui
**Summary:** Every transaction pays two separate fees: computation fee (CPU work, dynamic per epoch) and storage fee (bytes stored, fixed at 100 storage units per byte). Net cost = computation + storage - 99% rebate on deleted objects. The gas budget is a cap, not a charge — you only pay what the transaction actually costs. Unused budget is never charged.
**Bridged from:** Ethereum gas model — same "pay for computation" concept, but Sui adds a separate explicit storage fee and a rebate mechanism Ethereum doesn't have.
**Key clarification (from learner question):** Gas budget is a maximum, not a debit. Setting budget high is safe — validators never "eat" the excess. If actual cost < budget, you pay actual cost only. If actual cost > budget, transaction aborts.
**Misconception flagged:** "Gas on Sui is like Ethereum — just pay for execution." — No. Storage fee can dwarf computation fee for data-heavy operations.
**Checkpoint Q:** Lending protocol stores all historical loans forever — who pays, what's the better design?
**Learner answer:** User who creates the loan pays storage; no rebate since never deleted. Better: delete closed loans via close_loan(), get 99% back; use events for history. ✅
**Status:** ✅ understood

### Validator Fee Setting — Reference Gas Price Mechanism
**Source:** Sui Docs > Gas in Sui — docs.sui.io/concepts/tokenomics/gas-in-sui
**Summary:** At each epoch start, all validators submit their reservation price. The protocol picks the 2/3 percentile by stake as the Reference Gas Price (RGP). Validators who price at/below RGP get boosted staking rewards; those above get reduced rewards. No hard cap — constraint is economic: pricing too high loses income to cheaper competitors. Prevents arbitrary inflation without requiring a governance cap.
**Key clarification (from learner question):** Not arbitrary — 2/3 stake-weighted consensus on price, backed by reward incentives. Coordinated price attack requires controlling a supermajority (same threshold as network takeover).
**Status:** ✅ understood

### Storage Rebates
**Source:** Sui Docs > Gas in Sui — docs.sui.io/concepts/tokenomics/gas-in-sui
**Summary:** Deleting a Sui object returns 99% of the original storage fee to whoever submitted the delete transaction. 1% is kept permanently in the Storage Fund. This makes ephemeral object patterns almost free — create, use, delete ≈ only computation cost. Historical data belongs in events (off-chain), not as live on-chain objects.
**Bridged from:** Ethereum SSTORE — no meaningful refund mechanism. Sui treats storage as a refundable deposit, not a sunk cost.
**Key clarification (from learner question):** In Option A (shared Table), whoever submits the transaction that calls table::add() pays for the new dynamic field — usually the user, not the protocol. Storage cost per user is similar in both Option A and B. The real difference is execution cost (consensus vs fast path).
**Status:** ✅ understood

### Shared Object Performance Cost & Contention
**Source:** Sui Docs > Gas in Sui + Sui Docs Object Ownership
**Summary:** Transactions touching shared objects require consensus (multi-round validator agreement, ~500ms+). Transactions touching only owned objects skip consensus (fast path, ~100ms). Under high load, a popular shared object becomes a bottleneck — all transactions serialized through consensus. Owned objects execute in parallel with no coordination overhead.
**Bridged from:** W4 object ownership model — same owned/shared distinction, now understood through the performance and cost lens.
**Misconception flagged:** "Shared objects are just slower." — Not just latency. Under load, contention creates a serialized queue. A single hot shared object can bottleneck the entire protocol.
**Checkpoint Q:** GlobalBalances Table (shared) vs per-user owned Balance — which is better and why?
**Learner answer:** Option B (owned per-user) — cheaper per tx (no consensus), faster, parallelizable. Tradeoff: users pay storage for their own object (same cost as Option A actually — both users pay at tx time). ✅ Also correctly challenged that "protocol pays storage in Option A" — clarified: whoever submits table::add() pays, usually the user.
**Status:** ✅ understood

### Compounding Vulnerability — Shared Object + Unbounded Iteration
**Source:** W7 gas-grief pattern + today's consensus cost
**Summary:** Option A (shared Table) + distribute_rewards loop = two independent CRITICAL vulnerabilities that compound. Unbounded iteration alone = CRITICAL (gas grief, fund lock). Shared object alone = expensive + serialized. Together: attacker fills the Table cheaply (owned-object txs), making each distribute_rewards call require consensus AND iterate an enormous collection — making the attack cheaper relative to the cost of the victim's operation.
**Fix:** Pull-over-push (removes the loop) AND owned objects (removes consensus overhead). Both required — fixing one leaves the other.
**Status:** ✅ understood

### Key Mnemonics
- **Two-part gas = FedEx bill**: Handling fee (computation — what they do) + storage fee (warehouse rent — what you store). Ethereum only charges handling. Sui charges both.
- **Gas budget = restaurant tab limit**: You tell the waiter "max $100." If your food costs $40, you pay $40. The waiter doesn't pocket the remaining $60. If your order costs $120, the kitchen cancels it.
- **Storage rebate = deposit on a rental locker**: You pay a deposit when you rent. You get 99% back when you empty and return it. 1% is the admin fee, gone forever.
- **RGP = airport taxi price board**: All taxis submit their price. The board posts the 2/3 stake price as the standard fare. Taxis charging more lose customers (and bonus points). Nobody can charge infinity — competition and incentives keep it honest.
- **Hot potato = sticky note, not a safe-deposit box**: Hot potato lives in your hand during the transaction. No box, no storage, no rent. The moment you created it without `key`, you opted out of the storage system entirely.
- **Shared = airport security checkpoint**: Everyone queues through one scanner. The scanner can only process one person at a time (consensus). 1000 people = 1000 sequential scans. Owned = Pre-check lane — no queue, no scanner, go straight through.
- **Sharding = opening multiple checkout lanes**: One lane (shared object) = queue. Open 16 lanes (16 shards) = 16x throughput. Same store, same products, parallel processing.
- **Storage bloat = someone filling a shared inbox with junk mail**: They pay $0.01 per letter (storage per write). You pay the cost of sorting through 10,000 letters every time you check your mail (compute overhead). Asymmetric — cheap to attack, expensive to use.
- **Compound vulnerability = flooding + fire**: Unbounded iteration alone drowns the protocol. Shared object alone slows it down. Together: flood makes evacuation impossible, fire makes staying impossible. Fix both or you're still stuck.

## One-Paragraph Summary

Today I learned Sui's gas model has two separate bills: computation (CPU work, dynamic per epoch set by validators) and storage (bytes written, fixed price, 99% refundable when deleted). Validators can't set arbitrary prices because the 2/3 stake-weighted Reference Gas Price mechanism makes overpricing economically self-defeating. The gas budget is a cap, not a charge — unused budget is returned. The critical design insight: deleting objects is nearly free (99% rebate), so ephemeral patterns and cleaning up stale data are both free and responsible. Shared objects require consensus (~500ms+) while owned objects run in parallel via the fast path — so shared object design choices directly impact throughput and cost. Under high load, a single hot shared object serializes all traffic; sharding splits it into N parallel paths. Two compounding vulnerabilities flagged: shared Table + unbounded iteration = gas grief amplified by consensus cost.

## One-Page Summary

### What We Covered
- Two-part gas model: computation fee + storage fee
- Gas budget = cap (unused never charged)
- Storage fee: 1 byte = 100 storage units, fixed price
- Storage rebate: 99% back on delete, 1% permanent to Storage Fund
- Ephemeral objects (hot potato, create+delete patterns) = near-zero storage cost
- Validator RGP: 2/3 stake percentile + reward incentives
- Shared objects → consensus required (~500ms+, serialized under load)
- Owned objects → fast path (~100ms, parallel)
- Sharding: N copies of a hot shared object → N× throughput
- Storage bloat attack: growing collection, no delete path → HIGH severity
- Compounding: shared Table + unbounded iteration → both CRITICALs stack

### Key Mechanisms
1. **Net gas formula**: `total = computation + storage_created - (99% × storage_deleted)`. Source: Sui Docs > Gas in Sui
2. **RGP**: Every epoch, validators submit prices → 2/3 percentile by stake = Reference Gas Price. Below/at = boosted rewards. Above = reduced rewards. Source: Sui Docs > Gas in Sui
3. **Consensus vs fast path**: Shared objects route through consensus (~500ms+, serialized). Owned objects bypass consensus (~100ms, parallelizable). Source: Sui Docs > Gas in Sui

### Key Distinctions
- Computation fee vs storage fee: computation = CPU work (dynamic). Storage = bytes on disk (fixed, refundable).
- Budget vs charge: budget = ceiling. You pay actual cost ≤ budget.
- Storage write vs read: writing new bytes = storage fee. Reading/modifying existing bytes = computation only.
- Shared contention vs sharding: one shared object = bottleneck. N shards = N parallel consensus paths.
- Bloat severity: HIGH if alternative paths exist, CRITICAL only if the object is the sole protocol gateway.

### Gotchas
- Gas budget high → validators don't keep excess (cap, not payment)
- Validators setting arbitrary prices → prevented by RGP mechanism + reward penalties
- Storage fee on reads → no. Only on new bytes written.
- Hot potato has no storage cost → correct (no key ability = no object store entry)
- Table iteration by integer index → impossible in Move (W7 gotcha applies here too)
- "Protocol pays storage for shared Table" → wrong. Whoever submits table::add() pays.

## Key Takeaways

1. **Storage fee is a deposit, not a sunk cost.** Deleting objects returns 99%. Design protocols to clean up stale data — it's free and keeps the network healthy. Source: Sui Docs > Gas in Sui — docs.sui.io/concepts/tokenomics/gas-in-sui

2. **Shared objects serialize traffic — sharding is the solution.** One hot shared object = bottleneck. N shards = N× throughput. Every time you see a single shared object under high load, ask: can this be partitioned? Source: Sui Docs > Gas in Sui

3. **Two attack vectors compound on shared + unbounded collections.** Gas grief alone is CRITICAL. Shared object overhead alone is HIGH. Together, the attacker's cost is cheap (owned txs to inflate) while the victim's cost is catastrophic (consensus + unbounded iteration on every distribution call). Fix requires both pull-over-push AND owned objects. Source: Move Book §8.6 + Sui Docs Gas
