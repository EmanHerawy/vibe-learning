## Theory

### Two Completely Different "Versions" in Sui
> Source: Sui Framework (package.move) + Move Book upgradeability guide

```
┌──────────────────────────────┬──────────────────────────────────────┐
│  Sui runtime version         │  App-level version field             │
├──────────────────────────────┼──────────────────────────────────────┤
│  Managed by Sui VM           │  Managed by developer in Move        │
│  Auto-increments on mutation │  Bumped manually in migrate()        │
│  Lamport clock / concurrency │  Tracks which package version obj    │
│  NOT accessible from Move    │  expects — just a u64 field          │
│  e.g. object touched 47×     │  e.g. "this is the V2 schema"        │
└──────────────────────────────┴──────────────────────────────────────┘
```

### Pattern 1: Object Versioning
> Source: move-book.com/guides/upgradeability-practices

```move
const VERSION: u64 = 2;

public struct Protocol has key {
    id: UID,
    version: u64,   // MUST be present from initial publish
}

// Every entry point asserts current version
public entry fun deposit(p: &mut Protocol, ...) {
    assert!(p.version == VERSION, E_WRONG_VERSION);
}

// Post-upgrade migration — admin only
public fun migrate(_: &AdminCap, p: &mut Protocol) {
    assert!(p.version == VERSION - 1, E_WRONG_VERSION); // idempotency guard
    p.version = VERSION;
    // one-time data transformations here
}
```

```
Two migration modes:
┌──────────────┬──────────────────────────────────────┬─────────────────────────┐
│ Mode         │ How                                  │ When                    │
├──────────────┼──────────────────────────────────────┼─────────────────────────┤
│ Forced       │ entry points assert == VERSION       │ Security patch, breaking │
│ Graceful     │ entry points accept >= MIN_VERSION   │ Non-critical change      │
└──────────────┴──────────────────────────────────────┴─────────────────────────┘
```

**Critical:** Version field MUST exist from first publish. Struct fields are frozen under compatible policy — you can never add one later.

### Pattern 2: Anchor / Config Pattern (Dynamic Fields)
> Source: move-book.com/guides/upgradeability-practices + dynamic-fields

```move
use sui::dynamic_field as df;

// Stable anchor — never needs to change
public struct Protocol has key { id: UID, version: u64 }

// Config evolves across upgrades
public struct ConfigV1 has store { fee_bps: u64, recipient: address }
public struct ConfigV2 has store {
    fee_bps: u64, recipient: address,
    max_deposit: u64,  // new — impossible to add as struct field
    paused: bool,      // new
}

// ALWAYS use a typed key — never raw vector<u8> or String
public struct ConfigKey has copy, drop, store {}

// Migration: swap V1 → V2
public fun migrate(_: &AdminCap, p: &mut Protocol) {
    let ConfigV1 { fee_bps, recipient } =
        df::remove<ConfigKey, ConfigV1>(&mut p.id, ConfigKey {});
    df::add(&mut p.id, ConfigKey {}, ConfigV2 {
        fee_bps, recipient, max_deposit: 1_000_000, paused: false,
    });
    p.version = 2;
}
```

**Typed key safety:** hash = `hash(parent.id || name_value || Name_type)`. Different key structs = different field address. Prevents cross-module collisions.

### Pattern 4: Hot Potato — Atomic Upgrade Guarantee
> Source: github.com/MystenLabs/sui — sui-framework/sources/package.move

```
authorize_upgrade(cap, policy, digest) → UpgradeTicket   ← hot potato (no drop, no store)
    ↓ Sui runtime verifies bytecode matches digest
    ↓ Runtime issues UpgradeReceipt                       ← hot potato
commit_upgrade(cap, receipt)
    ↓ version++ on cap, new package ID set
```

`UpgradeTicket` and `UpgradeReceipt` have no `drop`/`store` — must be consumed in the same transaction. Upgrade is atomic: either fully applied or rolled back.

### Pattern 5: Upgrade Governance (UpgradeCap Wrapping)
> Source: session 2026-04-24 + package.move

**Timelock pattern:**
```move
public struct TimelockCap has key {
    id: UID,
    upgrade_cap: UpgradeCap,   // wrapped — not stealable externally
    delay_ms: u64,
    pending_digest: Option<vector<u8>>,
    scheduled_at: Option<u64>,
}

// Phase 1 (admin): schedule
public fun schedule(_: &AdminCap, tl: &mut TimelockCap, digest: vector<u8>, clock: &Clock) {
    tl.pending_digest = option::some(digest);
    tl.scheduled_at = option::some(clock::timestamp_ms(clock));
}

// Phase 2 (anyone, after delay): execute
public fun execute(tl: &mut TimelockCap, clock: &Clock): UpgradeTicket {
    let elapsed = clock::timestamp_ms(clock) - *option::borrow(&tl.scheduled_at);
    assert!(elapsed >= tl.delay_ms, E_TOO_EARLY);
    let digest = option::extract(&mut tl.pending_digest);
    package::authorize_upgrade(&mut tl.upgrade_cap, COMPATIBLE, digest)
}
```

Common delay periods: 24h (fast protocols), 7 days (DeFi), 30 days (high-value vaults).

### Publisher Object
> Source: package.move (MystenLabs/sui)

- Issued only at **first publish** via one-time witness — cannot be re-issued on upgrade
- Permanent proof of package ownership across all future upgrades
- Required for `display::new<T>(&publisher, ctx)` (Display standard)
- Used for type exclusivity checks (Pawtato Finance fix pattern)
- If lost, Display registration and type ownership proofs are gone permanently

### Orphaned Dynamic Field — Silent Data Loss
> Source: move-book.com/programmability/dynamic-fields

```move
// ❌ WRONG — dynamic fields silently orphaned
public fun destroy(p: Protocol) {
    let Protocol { id, version: _ } = p;
    object::delete(id);   // config + stats fields still on-chain, unreachable forever
}

// ✅ CORRECT — remove all dynamic fields before deleting parent
public fun destroy(p: Protocol) {
    let config = df::remove<ConfigKey, Config>(&mut p.id, ConfigKey {});
    drop_config(config);
    let Protocol { id, version: _ } = p;
    object::delete(id);
}
```

Orphaned fields remain on-chain, accrue storage fees, and are permanently inaccessible.

### Cross-Package Dependency Migration
> Source: Sui Move package model

```
PackageA (token) ← PackageB (staking) ← PackageC (rewards)

When PackageA upgrades → B and C must republish with updated address
```

```toml
# Move.toml — after PackageA upgrade
[addresses]
token = "0x<NEW_PACKAGE_ADDRESS>"   # update from old address
```

Upgrading a widely-depended-upon package = coordinated republish event across the full dependency graph.

## Key Takeaways

1. **Never launch a shared-object protocol without a `version: u64` field** — struct fields are frozen under compatible policy and cannot be added later. *Source: move-book.com/guides/upgradeability-practices*
2. **The full upgrade exploit chain is: compatible upgrade + forced migration = instant logic swap on all users** — attacker changes function bodies AND calls migrate() to lock everyone onto the new code. *Source: session 2026-04-24 + move-book upgradeability guide*
3. **Dynamic field typed keys prevent cross-module collisions** — field hash includes key type, so two structs with identical content produce different field addresses; raw `vector<u8>` keys are unsafe. *Source: move-book.com/programmability/dynamic-fields*

> "A compatible upgrade can skip ed25519_verify, change fee recipients, and then a migrate() call forces every user onto the new version — 'compatible policy' is not a safety signal."
