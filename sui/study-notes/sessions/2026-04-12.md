## Theory Notes

### Collection Types: Table / ObjectTable / Bag / ObjectBag
**Source:** Move Book §8.6 — move-book.com/programmability/collections.html
**Summary:** Four collection types mapped by two axes: key homogeneity (same type → Table/ObjectTable, mixed → Bag/ObjectBag) and value identity (plain data → Table/Bag, Sui objects with UID → ObjectTable/ObjectBag). The "Object" prefix always refers to values, never keys.
**Bridged from:** Solidity mappings vs arrays tradeoff — same "pick your tradeoff" instinct, new axis is object identity.
**Misconception flagged:** "ObjectTable = mixed key types" — wrong. Object prefix = values have UID. Key type homogeneity is separate axis (Bag family).
**Checkpoint Q:** NFT struct has UID — Table or ObjectTable?
**Learner answer:** ObjectTable, because it has UID. ✅
**Status:** ✅ understood

### Gas Grief via Unbounded Collections
**Source:** Move Book §8.6, §8.8
**Summary:** Dynamic collections grow without bound. Any function that iterates over a user-controlled collection scales gas with attacker-controlled length. Fix: pull-over-push (each user claims their own reward) or off-chain pagination with on-chain bounds.
**Bridged from:** Flash loan hot potato pattern — enforcing invariants through design, not checks.
**Misconception flagged:** Pagination alone is insufficient without state tracking to prevent skip/double-pay.
**Checkpoint Q:** Distribute rewards loops over all stakers — what's the vulnerability and fix?
**Learner answer:** Gas grief from large table; fix with index bounds/pagination. Tutor added: pull-over-push as stronger canonical fix.
**Status:** ✅ understood

### Key Mnemonics
- **Collection matrix**: Museum analogy. The SHELF LABEL = key type (same room = Table/OT, mixed exhibits = Bag/OB). The CERTIFICATE on each item = UID (has cert = Object prefix). Two labels, two questions.
- **Object prefix**: "The O is for the Object IN the box, not the box's filing system." O = values inside have identity.
- **Table no iteration**: "Table has no stairs." You can't step through it 0, 1, 2...
- **Gas grief**: "Bloat the boat, sink the crew." Attacker fills the collection, everyone's loop drowns.
- **Pull math bug**: "Pull the plug on the loop — but the math can still drown you." Two separate problems.
- **Reward-per-token**: Taxi meter. Meter ticks up for everyone (reward_per_token). You pay from when you got in to now (stake × delta). Getting out = claiming.
- **Table + Events**: "Table is the whiteboard (current state). Events are the security camera (history)."
- **Children follow parent**: "Pockets travel with the jacket." Sell the jacket = buyer gets everything in the pockets.
- **Lien cross-protocol**: "A lock on YOUR door doesn't stop a locksmith at THEIR door." Your lien check means nothing at a third-party marketplace.
- **HIGH vs CRITICAL**: "Anyone + No Escape = CRITICAL." One axis at a time.
- **Epoch vs Clock**: "Epoch = calendar page (days). Clock = stopwatch (milliseconds)."
- **Clock parameter**: "You can't read a clock that's not in the room." Pass it in or you have no time.

### Key Mechanisms
1. **Collection choice**: Same keys + plain values = Table. Same keys + UID values = ObjectTable. Mixed keys + plain values = Bag. Mixed keys + UID values = ObjectBag. Source: Move Book §8.6
2. **Reward-per-token**: `reward_per_token += epoch_rewards / total_staked` once per epoch. `claimable = stake × (reward_per_token - reward_debt)` at claim. O(1) everywhere. Source: Move Book §8.6 (pattern)
3. **Dynamic field transfer**: `dynamic_object_field::remove()` before transferring parent, or child moves with parent silently. Source: Move Book §8.9

### Key Distinctions
- `dynamic_field` vs `dynamic_object_field`: both children follow parent on transfer. Difference: object field value keeps UID (visible, independently queryable). Plain field value is invisible.
- `epoch()` vs `clock::timestamp_ms()`: epoch = day-level counter (free). Clock = ms timestamp (requires Clock param, ~10s stale).
- Table alone vs Table + Events: Table = current state only. Add events for history/indexability.
- HIGH vs CRITICAL: permissionless + irreversible = CRITICAL. Has escape hatch or requires privilege = HIGH.

### Gotchas
- "Object" prefix = values have UID. Nothing to do with key types.
- Table is NOT iterable by integer index. Adding a parallel Vec<K> creates sync vulnerability.
- Pull-over-push removes the loop, not the math. Dynamic share = ordering attack + rounding drain.
- Off-chain visibility doesn't require ObjectTable — use events.
- Clock must be passed as a function parameter — it's a shared object (0x6).
- Epoch number vs ms timestamp mismatch → deadlines never trigger → funds locked.
