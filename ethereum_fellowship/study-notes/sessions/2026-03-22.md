## What We Know Going In (Carried Forward)

### Consensus Foundations
- BFT: ≥2/3 honest nodes required. Byzantine fault = active lying to different peers (hardest)
- PBFT: O(n²) → unusable at scale. Committee sampling solves this.
- Gasper = LMD-GHOST (fork choice) + Casper FFG (finality)
- Both justification AND finalization require ≥2/3 of TOTAL validator set

### Validators
- Deposit 32 ETH → activation queue → active → attest every epoch, propose ~once per 3 months
- Equivocation → 1/32 burned + forced exit + 36-day withdrawable + correlation penalty

### Beacon Chain
- Slot = 12s, Epoch = 32 slots = 6.4 min
- 2,048 committees/epoch, ~341 validators each (128 = safety floor)
- BeaconState: RAM (hot), disk snapshots, 8,192-slot circular buffer (~27hrs)
- BeaconBlock wraps ExecutionPayload — one broadcast (libp2p), two state roots

### LMD-GHOST + Casper FFG
- LMD: latest timestamp wins. Parent vote doesn't count for child forks.
- FFG: source→target link. ≥2/3 votes → justified. Two consecutive justified → earlier finalized.
- FFG sets hard floor. LMD operates above it. Conflict → FFG wins.

### RANDAO
- Each proposer signs epoch number with BLS key → XOR-mixed into BeaconState
- Same seed, different domain tags → proposer / committee / sync committee selection
- Self-computed locally by every validator — no coordination
- Bias weakness: last proposer(s) can withhold. VDF (in research) will fix this.
- Aggregator: self-selected via hash(BLS_Sign(slot)) % MODULUS == 0 (~1 in 16)
- Inactivity leak: drains offline validators until online share crosses 66.7% → finality resumes

## Theory Notes

### Concept 23: Validator Lifecycle

**Full state machine:**
```
 32 ETH deposit
─────────────► PENDING ──► ACTIVE ──► EXITING ──► WITHDRAWABLE ──► DONE
              (queue)    │        │           │
                         │        │           └─► funds returned (~27hrs min)
                         │        └─► VOLUNTARY EXIT (user initiated)
                         └─► SLASHED ──► EXITING (forced) ──► WITHDRAWABLE
                                         (36-day window, correlation penalty)
```

---

**Stage 1: PENDING (Activation Queue)**
```
User sends 32 ETH to DepositContract (EL)
  → CL detects deposit log (~2,048 slots = ~7hrs to process)
  → enters activation queue
  → churn limit: max(4, validator_count // 65,536) per epoch
  → with 700k validators: max(4, 10) = 10/epoch
  → wait time depends on queue length (can be weeks during high demand)
  → becomes ACTIVE
```

**Churn limit formula explained:**
- `//` = integer division (round down): 700,000 // 65,536 = 10
- `max(4, ...)` = safety floor — minimum 4/epoch even on tiny networks
- Plain English: "at least 4/epoch, or 1 per every 65,536 validators — whichever is larger"

**Why 65,536 (2^16)?**
Security parameter — ensures it takes ~1 month for 10% of stake to exit.
Encodes: "economic security of a finalized block decreases by no more than 10% per month."
Prevents attack: malicious validators perform slashable act → try to exit before slashing → churn limit means they can't escape in time.
Connects to weak subjectivity: sync at least once per month or lose ability to verify finality safely.
Same churn limit applies to EXITS (entry and exit are rate-limited equally).

---

**Stage 2: ACTIVE**

| Sub-state | Condition | Effect |
|-----------|-----------|--------|
| Online, attesting correctly | Signing valid attestations | Earns ~4% APY |
| Offline / missing duties | Not signing | Inactivity leak starts |
| Equivocating | Double-signing same slot/epoch | Slashed immediately |

**Inactivity leak:**
```
Triggered when: finality stalled for >4 epochs (online stake < 66.7%)
Effect: offline validators' balances drain at accelerating (quadratic) rate
Purpose: shrink offline stake → online share grows → crosses 66.7% → finality resumes
Rate: longer offline = faster drain (quadratic, not linear)
```

---

**Stage 3: VOLUNTARY EXIT**
```
Validator broadcasts SignedVoluntaryExit
  → must have been active ≥ 256 epochs (~27hrs) — cannot exit immediately after joining
  → enters exit queue (same churn limit as entry)
  → status: EXITING — still must attest during queue wait
  → status: WITHDRAWABLE — no more duties
  → funds returned after ~27hrs minimum
  → during high exit demand: weeks
```

---

**Stage 4: SLASHED (Forced Exit)**
```
Slashable offense detected
  → 1/32 of effective balance burned immediately
  → forced into exit queue (cannot stay active)
  → WITHDRAWABLE delay: 8,192 epochs (~36 days)
  → at ~18 days (midpoint): correlation penalty calculated + applied
  → funds returned minus all penalties
```

**Why 36 days?** Time window to observe whether this was a coordinated attack.
If many validators slashed in same window → correlation penalty scales up → can reach 100%.

**Correlation penalty math:**
```
penalty = effective_balance × (3 × total_slashed_in_window) / total_stake

Solo incident (only you slashed):
  total_slashed ≈ 32 ETH, total_stake ≈ 22M ETH
  extra penalty ≈ tiny (~0.0001 ETH)
  Total loss: ~1 ETH (just the 1/32 initial slash)

Large coordinated attack (1/3 of validators slashed):
  total_slashed ≈ 7M ETH, total_stake ≈ 22M ETH
  penalty ≈ 30.5 ETH
  Total loss: ~32 ETH (entire stake wiped)
```

Design intent: solo mistakes cheap to recover from. Coordinated attacks catastrophically expensive.

---

**Full lifecycle table:**

| Stage | Can attest? | Can propose? | Earning rewards? | Balance at risk? |
|-------|------------|-------------|-----------------|-----------------|
| PENDING | ❌ | ❌ | ❌ | ❌ |
| ACTIVE (online) | ✅ | ✅ | ✅ | ✅ slashing |
| ACTIVE (offline) | ❌ | ❌ | ❌ | ✅ inactivity leak |
| EXITING (voluntary) | ✅ | ❌ | ✅ reduced | ✅ |
| SLASHED + EXITING | ❌ | ❌ | ❌ | ✅ correlation penalty |
| WITHDRAWABLE | ❌ | ❌ | ❌ | ❌ |
