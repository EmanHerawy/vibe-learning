## Theory Notes

### Concept 1: What is Ethereum and why does it exist?

**Analogy:** Bitcoin is a calculator (one job: trustless money transfer). Ethereum is a programmable computer running on thousands of machines with no single owner.

**Key insight (Vitalik, 2013):** If we can achieve trustless consensus for money transfers, why not for arbitrary programs?

**Formal model — Ethereum as a state machine:**
- **State** = all accounts, balances, and contract storage (a snapshot of "the world right now")
- **Transition** = a transaction changes the state
- **Consensus** = thousands of nodes agree on which state is valid

**Why this matters for protocol dev:** You're not building dApps. You're maintaining the engine of this world computer — correctness, efficiency, security.

- Status: ✅ understood

### Concept 2: The two-layer architecture (EL + CL)

**The two layers:**

| Layer | Full Name | Job | Analogy |
|-------|-----------|-----|---------|
| **EL** | Execution Layer | Runs the EVM, processes transactions, updates state | The engine of a car |
| **CL** | Consensus Layer | Runs Proof of Stake, agrees on which blocks are valid, finalizes the chain | The steering wheel + GPS |

**Why separated?**
- Before The Merge (Sept 2022): one monolithic client
- Now: each layer upgradeable independently, different teams/languages can implement each
- **Client diversity** is a first-class security property — no single client should control >33% of validators. If a bug hits the dominant client, <33% means the network keeps running.
- Monitored at: [clientdiversity.org](https://clientdiversity.org)

### Concept 3: How EL and CL communicate (Engine API)

**The handshake:**

```
CL (beacon node)                    EL (execution client)
      │                                      │
      │── engine_forkchoiceUpdated ─────────▶│  "here's the latest head, build a block"
      │                                      │  (EL runs txs, computes new state)
      │◀─ engine_getPayload ─────────────────│  "here's the block payload"
      │                                      │
      │── engine_newPayload ────────────────▶│  "validate this block"
      │◀─ VALID / INVALID ───────────────────│
```

- Local JSON-RPC on **port 8551**
- Authenticated with a **JWT secret** (shared between EL + CL processes)
- JWT protects against **local process impersonation** — an attacker with machine access could forge Engine API calls without it, feeding invalid blocks to the EL

- Status: ✅ understood

### Concept 3a: EL + CL — independent vs coupled

**Architecturally decoupled, operationally coupled.**

| | Architectural | Operational |
|---|---|---|
| **EL** | Own codebase, language, team | Must run alongside CL |
| **CL** | Own codebase, language, team | Must run alongside EL |

**Analogy:** Engine + gearbox — designed and built separately, but you need both to drive the car.

- Without CL: EL has no idea which fork is canonical, can't know which chain to execute
- Without EL: CL can't validate transaction execution or compute the state root

- Status: ✅ understood

### Concept 3b: JWT secret vs validator staking key

| | JWT Secret | Validator Signing Key |
|---|---|---|
| **What it is** | Shared secret between EL + CL processes | BLS private key owned by a validator |
| **Purpose** | Authenticates Engine API calls locally | Signs attestations + block proposals on-chain |
| **Who holds it** | Both EL + CL on the same machine | CL only (validator client) |
| **Stakes involved** | Nothing | 32 ETH |

**Key insight:** The EL has zero awareness of validators or stakes — it only executes blocks.

**But:** If EL is down → CL can't fulfill validator duties → **inactivity penalties** (slow ETH leak from your 32 ETH stake).

**Practical implication:** Your 32 ETH stake is an implicit commitment to keep BOTH layers healthy 24/7. This is why **DVT (Distributed Validator Technology)** and failover setups exist.

## Practice Log

### Hands-On Trace: Transaction Lifecycle (EL → CL → Finality)

**Scenario:** `transfer(alice, 1 ETH)` called from a wallet — trace from "send" to irreversible.

```
Step 1: Tx signed → broadcast → EL mempool
Step 2: EL builds block payload (selects + orders txs from mempool)
Step 3: EL executes — validates sender balance + nonce, runs EVM, updates state
Step 4: CL validator wraps payload → proposes block to network (every 12s slot)
Step 5: Other validators attest → soft agreement within the slot
Step 6: After 2 epochs (~12.8 min) → FINALIZED (cryptographic, not probabilistic)
```

**Key corrections from exercise:**

**Mempool + block building = EL, not CL**
- Mempool lives in the EL
- Block building (selecting + ordering transactions) is an EL job
- The CL validator says "give me a payload" → EL builds it → CL proposes it

```
Mempool → EL builds payload → CL validator proposes the block
          (EL's job)           (CL's job)
```

**Timing**
- 1 slot = **12 seconds** (one block per slot)
- Attestation (soft agreement) happens within the slot
- **Finality** = after **2 epochs** (~12.8 minutes)

**Ethereum finality vs Bitcoin confirmations**

| | Bitcoin | Ethereum PoS |
|---|---|---|
| **Model** | Probabilistic — reorg gets harder over time | Cryptographic — after 2 epochs it's mathematically final |
| **Reversal cost** | Just needs more hashpower | Requires slashing >1/3 of ALL staked ETH (millions of ETH) |
| **"Safe" window** | Wait N confirmations | Wait for finality (~12.8 min) |

> "Wait for confirmations" = pre-finality window.
> After finality: reversal is not just unlikely — it's economically catastrophic by design.
