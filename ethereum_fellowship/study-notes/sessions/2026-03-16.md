## Theory Notes

### Concept 1: What is Ethereum and why does it exist?

**Analogy:** Bitcoin is a calculator (one job: trustless money transfer). Ethereum is a programmable computer running on thousands of machines with no single owner.

**Key insight (Vitalik, 2013):** If we can achieve trustless consensus for money transfers, why not for arbitrary programs?

**Formal model — Ethereum as a state machine:**
- **State** = all accounts, balances, and contract storage (a snapshot of "the world right now")
- **Transition** = a transaction changes the state
- **Consensus** = thousands of nodes agree on which state is valid

**Why this matters for protocol dev:** You're not building dApps. You're maintaining the engine of this world computer — correctness, efficiency, security.

- Status: ✅ understood

### Concept 2: The two-layer architecture (EL + CL)

**The two layers:**

| Layer | Full Name | Job | Analogy |
|-------|-----------|-----|---------|
| **EL** | Execution Layer | Runs the EVM, processes transactions, updates state | The engine of a car |
| **CL** | Consensus Layer | Runs Proof of Stake, agrees on which blocks are valid, finalizes the chain | The steering wheel + GPS |

**Why separated?**
- Before The Merge (Sept 2022): one monolithic client
- Now: each layer upgradeable independently, different teams/languages can implement each
- **Client diversity** is a first-class security property — no single client should control >33% of validators. If a bug hits the dominant client, <33% means the network keeps running.
- Monitored at: [clientdiversity.org](https://clientdiversity.org)

- Status: ✅ understood

### Concept 3: How EL and CL communicate (Engine API)

**The handshake:**

```
CL (beacon node)                    EL (execution client)
      │                                      │
      │── engine_forkchoiceUpdated ─────────▶│  "here's the latest head, build a block"
      │                                      │  (EL runs txs, computes new state)
      │◀─ engine_getPayload ─────────────────│  "here's the block payload"
      │                                      │
      │── engine_newPayload ────────────────▶│  "validate this block"
      │◀─ VALID / INVALID ───────────────────│
```

- Local JSON-RPC on **port 8551**
- Authenticated with a **JWT secret** (shared between EL + CL processes)
- JWT protects against **local process impersonation** — an attacker with machine access could forge Engine API calls without it, feeding invalid blocks to the EL

- Status: ✅ understood

### Concept 3a: EL + CL — independent vs coupled

**Architecturally decoupled, operationally coupled.**

| | Architectural | Operational |
|---|---|---|
| **EL** | Own codebase, language, team | Must run alongside CL |
| **CL** | Own codebase, language, team | Must run alongside EL |

**Analogy:** Engine + gearbox — designed and built separately, but you need both to drive the car.

- Without CL: EL has no idea which fork is canonical, can't know which chain to execute
- Without EL: CL can't validate transaction execution or compute the state root

- Status: ✅ understood

### Concept 3b: JWT secret vs validator staking key

| | JWT Secret | Validator Signing Key |
|---|---|---|
| **What it is** | Shared secret between EL + CL processes | BLS private key owned by a validator |
| **Purpose** | Authenticates Engine API calls locally | Signs attestations + block proposals on-chain |
| **Who holds it** | Both EL + CL on the same machine | CL only (validator client) |
| **Stakes involved** | Nothing | 32 ETH |

**Key insight:** The EL has zero awareness of validators or stakes — it only executes blocks.

**But:** If EL is down → CL can't fulfill validator duties → **inactivity penalties** (slow ETH leak from your 32 ETH stake).

**Practical implication:** Your 32 ETH stake is an implicit commitment to keep BOTH layers healthy 24/7. This is why **DVT (Distributed Validator Technology)** and failover setups exist.

### Concept 4: Merkle Trees
- Leaf = hash of data item. Parent = hash(left child + right child). Root = single fingerprint of all data.
- **Merkle inclusion proof** = sibling hashes along the path from leaf to root (~log₂n hashes)
- Light clients store only block headers → verify tx inclusion via proof against `transactions_root`

**Ethereum's 3 roots per block header:**
| Root | Commits to |
|------|------------|
| `transactions_root` | All transactions in the block |
| `receipts_root` | All receipts (logs, gas used, status) |
| `state_root` | Entire world state after the block |

- State uses **Merkle Patricia Trie (MPT)** not a plain tree — supports key-value lookup + exclusion proofs
- Status: ✅ understood

### Concept 5: p2p Networking

**Two separate p2p networks:**
| Layer | Protocol | Transport | Used for |
|-------|----------|-----------|----------|
| EL | devp2p (RLPx) | TCP | Transactions, blocks, state sync |
| CL | libp2p (GossipSub) | TCP/QUIC | Beacon blocks, attestations, validator gossip |

Discovery uses **discv5 over UDP** (fast, lightweight peer finding via DHT + bootnodes).

**New node bootstrap flow:**
1. Hardcoded bootnodes → ping/pong handshake
2. Request peer list → expand connections
3. Gossip: exchange known peers, chain head, recent blocks
4. Sync via named strategy (snap sync for EL, checkpoint sync for CL)

**Sync strategies:**
| Strategy | How | Default for |
|----------|-----|-------------|
| Full sync | Re-execute every block from genesis | Trustless archival nodes |
| Snap sync | Download recent state snapshot + verify | EL (hours) |
| Checkpoint sync | Start from trusted finalized checkpoint | CL (minutes) |

**Why devp2p (EL) vs libp2p (CL)?**
- devp2p: built 2014, no good p2p library existed then, custom to Ethereum
- libp2p: built by Protocol Labs for IPFS, mature + modular by the time CL was designed
- Two separate peer networks run in parallel on every full node

- Status: ✅ understood

### Concept 6: Cryptography Primitives

**1 — keccak256 (hash function)**
- Used everywhere: tx hashes, block hashes, state roots, contract address derivation
- Ethereum uses keccak256, NOT SHA-256 (Bitcoin uses SHA-256)

**2 — ECDSA on secp256k1 (user transactions)**
```
private key (32 bytes) → public key (64 bytes) → keccak256 → last 20 bytes = address
```
- Signs user transactions. Anyone can verify with public key, private key never revealed.

**3 — BLS12-381 (validator attestations)**
- Key property: **signature aggregation**
- ~700,000 validators attest per slot (every 12 seconds)
- ECDSA: 700,000 × 65 bytes = ~45 MB per slot → impossible
- BLS: 700,000 signatures → 1 × 96 bytes → trivial
- Attestations are at **block level** (validators attest to beacon block + checkpoints), not tx level
- Aggregation is what makes PoS with large validator sets physically possible

- Status: ✅ understood

## Practice Log

### Hands-On Trace: Transaction Lifecycle (EL → CL → Finality)

**Scenario:** `transfer(alice, 1 ETH)` called from a wallet — trace from "send" to irreversible.

```
Step 1: Tx signed → broadcast → EL mempool
Step 2: EL builds block payload (selects + orders txs from mempool)
Step 3: EL executes — validates sender balance + nonce, runs EVM, updates state
Step 4: CL validator wraps payload → proposes block to network (every 12s slot)
Step 5: Other validators attest → soft agreement within the slot
Step 6: After 2 epochs (~12.8 min) → FINALIZED (cryptographic, not probabilistic)
```

**Key corrections from exercise:**

**Mempool + block building = EL, not CL**
- Mempool lives in the EL
- Block building (selecting + ordering transactions) is an EL job
- The CL validator says "give me a payload" → EL builds it → CL proposes it

```
Mempool → EL builds payload → CL validator proposes the block
          (EL's job)           (CL's job)
```

**Timing**
- 1 slot = **12 seconds** (one block per slot)
- Attestation (soft agreement) happens within the slot
- **Finality** = after **2 epochs** (~12.8 minutes)

**Ethereum finality vs Bitcoin confirmations**

| | Bitcoin | Ethereum PoS |
|---|---|---|
| **Model** | Probabilistic — reorg gets harder over time | Cryptographic — after 2 epochs it's mathematically final |
| **Reversal cost** | Just needs more hashpower | Requires slashing >1/3 of ALL staked ETH (millions of ETH) |
| **"Safe" window** | Wait N confirmations | Wait for finality (~12.8 min) |

> "Wait for confirmations" = pre-finality window.
> After finality: reversal is not just unlikely — it's economically catastrophic by design.

### Concept 7: EOA vs Contract Accounts

**Two account types in Ethereum state:**

| | EOA | Contract Account |
|---|---|---|
| Controlled by | Human with private key | Code (no private key) |
| Can initiate tx? | ✅ Yes | ❌ No — only reacts |
| Has code? | ❌ | ✅ EVM bytecode on-chain |
| Has storage? | ❌ | ✅ Persistent key-value store |
| Stores | balance + nonce | balance + nonce + code + storage |

- **Nonce:** Counter per EOA, increments with every tx — prevents replay attacks
- **Call stack:** Contracts can call contracts (up to 1024 levels deep)
- **DELEGATECALL:** Runs another contract's code in YOUR storage — used by proxy patterns, common vulnerability source
- **tx.origin:** Always the EOA that started the chain — never a contract

### Concept 8: Gas Mechanism

**Gas = unit of computation cost. Prevents infinite loops/DoS.**

| Operation | Gas cost |
|-----------|----------|
| Simple ETH transfer | 21,000 |
| EVM computational step | 2–10 |
| SLOAD (cold) | 2,100 |
| SSTORE (new slot) | 20,000 |
| SSTORE (existing slot) | 5,000 |
| Non-zero tx data byte | 16 |

**Block limits:** 30M gas max, 15M gas target (EIP-1559 keeps blocks ~50% full)

**EIP-1559 fee market:**
```
Total fee = (base_fee + priority_fee) × gas_used
base_fee:     Protocol-set. BURNED. +12.5% if prev block >50% full, -12.5% if <50%
priority_fee: Tip to block proposer. Usually 1-2 gwei.
max_fee:      Cap you set. Tx waits in mempool if base_fee > max_fee.
```
Base fee is burned to prevent validators from artificially congesting the network.
After 3 full blocks: base_fee × 1.125³ = +42% in 3 blocks.

- Status: ✅ understood

### Concept 9: Transaction Fields

| Field | Purpose |
|-------|---------|
| `nonce` | EOA tx counter — prevents replay |
| `to` | Recipient. Empty = contract deployment |
| `value` | ETH to send (wei) |
| `data` | Function selector + args for contracts. Empty for ETH transfers. |
| `gas_limit` | Max gas willing to spend |
| `max_fee_per_gas` | Max total fee per gas (base + priority) |
| `max_priority_fee_per_gas` | Tip to proposer |
| `signature (v,r,s)` | ECDSA proof of sender's private key |

- `to` empty → contract deployment. `data` = init code (constructor bytecode + constructor args)
- Contract address is deterministic: `CREATE` uses deployer+nonce, `CREATE2` uses deployer+salt+bytecode hash
- `data` encoding: `function_selector` = first 4 bytes of `keccak256("functionName(types)")`
- Contract has no private key → control is expressed in code logic (Ownable, multisig, etc.)

## Theory Notes

### Concept 16: Why Consensus is Hard — The Double-Spend Problem
- Digital scarcity: blockchain's core value — prevents copying/double-spending digital assets
- State machine replication: every node runs same rules + same inputs → same state
- BFT requirement: need ≥ 2/3 honest nodes to tolerate up to 1/3 Byzantine (malicious) nodes
- Fault types: crash (silent, easy), network (delayed msgs), Byzantine (active lying — hardest)
- PBFT limitation: O(n²) messages — 700k validators = 490 billion msgs/slot → unusable
- Status: ✅ understood

### Concept 16b: Committee Sampling
- Full validator set divided across 32 slots × up to 64 sub-committees = 2,048 committees/epoch
- Each validator in exactly ONE committee per epoch (everyone participates, one vote each)
- 128 = minimum safety floor (not target size). With 700k validators actual size ~341
- Security: probability of 33%-stake attacker dominating a 128+ committee ≈ 1 in 10^28
- Shuffle revealed only ~2 epochs ahead (RANDAO) → short lookahead = less attack surface
- Status: ✅ understood

### Concept 16c: Gasper — Two-Layer Overview
- Two problems: fork choice (which head now?) + finality (what's irreversible?)
- Gasper = LMD-GHOST (fork choice) + Casper FFG (finality) running simultaneously
- One attestation carries BOTH votes: beacon_block_root (LMD) + source/target checkpoints (FFG)
- Analogy: helm (LMD) steers moment-to-moment; GPS autopilot (FFG) locks confirmed waypoints
- Status: ✅ understood

### Concept 17: Validators
- Deposit 32 ETH to DepositContract (EL) → CL detects → enters activation queue
- Queue limit: ~8 validators/epoch — prevents network flooding + protects stake ratio on exits
- Duties: attest every epoch (assigned slot), propose blocks rarely (RANDAO selected)
- Rewards: ~4% APY for correct attestations
- Penalties: inactivity leak (slow bleed), slashing for malicious behavior
- PoS advantage over PoW: no economies of scale → naturally decentralizes
- Status: ✅ understood

### Concept 18: Beacon Chain Structure
- Slot = 12 seconds, Epoch = 32 slots = 384s ≈ 6.4 min
- Each slot: one proposer builds block, committee of validators attests
- With 700k validators: 32 × 64 = 2,048 committees/epoch, ~341 validators each
- Committee assignment: shuffled each epoch via RANDAO, revealed ~2 epochs ahead
- BeaconState key fields: validators[], balances[], randao_mixes[], block_roots[],
  state_roots[], current_justified_checkpoint, finalized_checkpoint, current_epoch_attestations[]
- Status: ✅ understood

### Concept 18b: BeaconState Storage
- Current state: RAM (~50-100MB) — hot, updated every slot
- History: disk snapshots at epoch boundaries — reconstructable
- block_roots[] + state_roots[]: circular buffer of last 8,192 slots (~27 hours)
- historical_roots[]: permanent Merkle root of all past block+state root pairs (hashes only)
- Archival node: stores full state every slot (TB range) — needed for historical queries
- Pruned node: keeps recent state + finalized snapshots only
- Every beacon block header commits state_root — authenticates any historical state claim
- Status: ✅ understood

### Concept 18c: Beacon Block vs ExecutionPayload
- Pre-Merge: EL broadcast its own blocks on devp2p independently
- Post-Merge: EL produces ExecutionPayload (internal), CL wraps it in BeaconBlock, broadcasts on libp2p
- ONE broadcast on network (BeaconBlock), TWO state roots committed (BeaconState + EL world state)
- Structure: BeaconBlock → BeaconBlockBody → ExecutionPayload → transactions[]
- Etherscan = EL lens (transactions, gas). beaconcha.in = CL lens (slots, validators, attestations)
- EL block numbers: sequential, no gaps (missed slots simply don't produce a payload)
- CL slot numbers: show gaps when slots missed
- "eth1/eth2" terminology retired post-Merge → now EL/CL
- Status: ✅ understood

### Concept 19+20: Attestation Process — LMD-GHOST + Casper FFG
**Attestation structure:**
- beacon_block_root → LMD-GHOST vote (current chain head)
- source + target → Casper FFG vote (checkpoint justification)
- One object, two votes, one BLS signature

**Four roles per slot:**
- Proposer (1, RANDAO-selected): builds BeaconBlock, includes PREVIOUS slot's attestations
- Attesters (~21,875, committee-assigned): each independently creates + signs own AttestationData at t=4s
- Aggregators (~16, random subset): compress many BLS sigs → one aggregate
- All nodes: receive + verify

**Timing rule:** Attestations from slot N land on-chain in slot N+1 or N+2 (not slot N)

**LMD-GHOST rules:**
- Always use LATEST attestation per validator (discard all older ones from same validator)
- At each fork: pick child subtree with most validator weight
- Vote for parent block does NOT count toward either child fork
- Prevents: vote replay attack (replaying old votes to inflate stale fork weight)

**Casper FFG rules:**
- Checkpoint = (epoch, first block of epoch)
- source→target link: if ≥2/3 validators vote same link → target JUSTIFIED
- Two consecutive justified checkpoints → earlier one FINALIZED
- Finalization requires >2/3 (66.7%). Justification requires >2/3 too.
- 60% cannot finalize (< 66.7%)

**Block validity for attestations:**
- Max age: current or previous epoch only
- No duplicates (same validator counted twice = fork choice/FFG manipulation)
- FFG source must be justified checkpoint
- Committee assignment must match slot

**Equivocation (Byzantine double-voting):**
- Specific term: equivocation
- Penalty: 1/32 stake burned + forced exit + ~36 day withdrawable period + correlation penalty
- Harder than crash fault: crash = silent (obvious). Byzantine = different messages to different peers (requires collecting both conflicting sigs to prove)

- Status: ✅ understood
