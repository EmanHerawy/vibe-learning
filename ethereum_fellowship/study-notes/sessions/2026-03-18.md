## Theory Notes

### Concept 16: Why Consensus is Hard — The Double-Spend Problem

**The core problem:**
- Digital goods can be copied for free — without a shared truth, I can spend the same coins twice
- Naive fix: trusted server (bank). Problem: reintroduces need for trust, single point of failure
- Blockchain answer: distributed network of nodes agree on same state without central coordinator

**State machine replication:**
- Every node runs same rules + same inputs → same output (same state)
- Achieve this across thousands of nodes without central coordinator = solved consensus

**Byzantine Fault Tolerance (BFT):**
- Named after Byzantine Generals Problem: some generals (nodes) are traitors sending conflicting messages
- Need ≥ 2/3 honest nodes to tolerate up to 1/3 Byzantine nodes

| Fault type | Example | Difficulty |
|------------|---------|------------|
| Crash fault | Node goes offline | Easy — silence is obvious |
| Network fault | Messages delayed/lost | Medium |
| Byzantine fault | Sends different messages to different peers | Hardest — active deception |

**Why Byzantine is hardest:** Crashed node = silent (every peer sees the same thing: nothing). Byzantine node = actively lies differently to different peers. Peer A sees vote X, Peer B sees vote Y. Requires collecting both conflicting messages to prove.

**PBFT limitation:** O(n²) messages per round. 700k validators = 490 billion messages/slot → completely unusable.

**Q&A clarification:** 128 is the MINIMUM safety floor for committees, not the target size. With 700k validators actual size is ~341.

### Concept 16b: Committee Sampling

**Why a subset is enough:**
- Random, unpredictable subset large enough that attacker can't reliably corrupt it
- With ≥128 validators per committee, probability of 33%-stake attacker dominating ≈ 1 in 10^28

**Structure:**
```
Total validators: ~700,000
Committees per epoch: 32 slots × 64 sub-committees = 2,048
Validators per committee: 700,000 ÷ 2,048 ≈ 341
Minimum safety floor: 128 (security breaks below this)
```

**Key rules:**
- Every validator is in exactly ONE committee per epoch (no one sits out, no one votes twice)
- Shuffle revealed only ~2 epochs ahead via RANDAO → short lookahead = less attack surface
- Not "pick 128 from 700k" — shuffle ALL 700k into 2,048 buckets, each bucket ≥128

**Committee math with 512,000 validators:**
```
512,000 ÷ (32 × 64) = 250 validators per committee
```

### Concept 16c: Gasper — Two-Layer Overview

**Two separate problems, two layers:**

| Problem | Question | Layer |
|---------|----------|-------|
| Fork choice | Right now, which chain head to build on? | LMD-GHOST |
| Finality | Which past blocks are forever irreversible? | Casper FFG |

**Gasper = LMD-GHOST + Casper FFG running simultaneously**

One attestation carries BOTH votes:
```
Vote 1 (LMD-GHOST): "I think THIS block is the current chain head"
Vote 2 (Casper FFG): "I support THIS checkpoint as justified"
```

**Analogy:** Helm (LMD-GHOST) steers moment-to-moment. GPS autopilot (Casper FFG) locks in confirmed waypoints you can never sail back from.

**Why can't you use Casper FFG alone?** FFG only finalizes checkpoints — it doesn't tell you which current fork to extend. Without LMD-GHOST you have no live fork choice and the chain stalls.

- Status: ✅ understood

### Concept 17: Validators

**Becoming a validator:**
```
User → sends 32 ETH to DepositContract (EL contract)
     → CL detects deposit log
     → enters activation queue (~8 validators/epoch limit)
     → after queue wait → ACTIVE
     → starts attesting + occasionally proposing
```

**Why a queue?** Prevents network flooding + protects stake ratio (exit queue mirrors entry queue — prevents sudden large-scale withdrawals destabilizing stake ratio).

**Two duties:**

| Duty | Frequency | What |
|------|-----------|------|
| Attest | Once per epoch (assigned to 1 specific slot out of 32) | Vote for chain head + finality checkpoint |
| Propose | Rarely (~once every few months) | Build and broadcast a new block |

**Economics:**
- 32 ETH locked → ~4% APY for correct attestations
- Offline → inactivity leak (slow bleed from stake)
- Equivocation (double-voting) → slashed (up to 100% of stake)

**PoS vs PoW decentralization:** No economies of scale in PoS — 1 validator = same efficiency as 1,000 validators. PoW mining benefits from bulk hardware → centralization. PoS naturally decentralizes.

- Status: ✅ understood

### Concept 18: Beacon Chain Structure

**Time structure:**
```
Slot = 12 seconds
Epoch = 32 slots × 12s = 384s ≈ 6.4 min

Slot 0   Slot 1   Slot 2  ...  Slot 31
│──12s───│──12s───│──12s── ... ──│
└──────────── Epoch 0 (6.4 min) ─┘
```

**Per slot:**
1. One block proposer (randomly chosen via RANDAO) builds + broadcasts block
2. Assigned committee of validators attests to it
3. Beacon node collects attestations, aggregates BLS signatures

**Committee structure with 700k validators:**
```
700,000 ÷ 32 slots = ~21,875 validators/slot
700,000 ÷ (32 × 64) = ~341 validators/committee
MAX_COMMITTEES_PER_SLOT = 64
Total committees/epoch = 2,048
```

**BeaconState key fields:**
```
BeaconState {
  validators[]                   # all validator records
  balances[]                     # current balance per validator
  randao_mixes[]                 # 8,192 epochs of accumulated randomness
  block_roots[]                  # last 8,192 block hashes (~27 hrs)
  state_roots[]                  # last 8,192 state hashes (~27 hrs)
  current_justified_checkpoint   # latest justified epoch+block
  finalized_checkpoint           # latest finalized epoch+block
  current_epoch_attestations[]   # pending attestations
}
```

- Status: ✅ understood

### Concept 18b: BeaconState Storage

**Three layers:**

| Layer | What | Where | Duration |
|-------|------|-------|----------|
| Current BeaconState | Live state right now | RAM (~50-100MB) | Hot, updated every slot |
| Epoch snapshots | State at checkpoint boundaries | Disk | Last N epochs |
| block_roots[] + state_roots[] | Circular buffer of hashes | Inside BeaconState | 8,192 slots = ~27 hours |
| historical_roots[] | Merkle root of all past pairs | Permanent | Forever (hashes only) |

**8,192 number:** 8,192 slots × 12s = 98,304s ≈ 27.3 hours — window for proving recent block inclusion.

**Can you check old blocks?**

| What you want | How |
|---------------|-----|
| Recent block (<27 hrs) | Directly from block_roots[] in BeaconState |
| Old block hash (>27 hrs) | Via historical_roots[] — hashes only, not full state |
| Old full state (3+ days) | Reconstruct from nearest snapshot + replay — archival node only |

**Node types:**
- Pruned beacon node: recent state + finalized snapshots → enough for consensus + syncing peers
- Archival beacon node: full state every slot (TB range) → needed for historical queries, block explorers

**Q&A clarification:** "Lives in memory" ≠ history is lost. Current state is hot in RAM. History is separately handled via snapshots + historical_roots.

Every beacon block header commits `state_root` → even if you store only headers, any claimed historical state is verifiable.

### Concept 18c: Beacon Block vs ExecutionPayload

**Pre-Merge vs Post-Merge:**
```
BEFORE MERGE:
[EL Block] ──► devp2p network    (EL peers gossip this independently)
[Beacon Block] ──► libp2p network

AFTER MERGE:
[Beacon Block
  └── ExecutionPayload (EL block embedded inside)
] ──► libp2p network only
```

**Post-Merge:** EL is a payload factory. It never touches the p2p network for blocks. Only the CL broadcasts.

**Full nesting structure:**
```
BeaconBlock
├── slot, proposer_index, parent_root
├── state_root        ← hash of BeaconState (CL world)
└── BeaconBlockBody
    ├── attestations[]
    ├── deposits[]
    ├── voluntary_exits[]
    ├── sync_aggregate
    └── ExecutionPayload
        ├── parent_hash, fee_recipient
        ├── state_root  ← hash of EL world state (accounts, balances, contracts)
        ├── gas_used, base_fee_per_gas
        └── transactions[]  ← user transactions live here
```

**Two state roots in one block:**
- `BeaconBlock.state_root` → CL world (validators, finality)
- `ExecutionPayload.state_root` → EL world (accounts, contracts)

**Block explorers:**

| Explorer | Lens | Source | API |
|----------|------|--------|-----|
| Etherscan.io | EL | ExecutionPayload | eth_getTransactionByHash (JSON-RPC) |
| beaconcha.in | CL | BeaconBlock | Beacon API |

**Missed slots:**
```
EL block numbers: sequential, NO gaps (missed slot = no payload = counter doesn't increment)
CL slot numbers:  show gaps when slots are missed (slots tick every 12s regardless)
```

**Terminology — eth1/eth2 retired post-Merge:**

| Old name | Current name |
|----------|-------------|
| eth1 | Execution Layer (EL) |
| eth2 / beacon chain | Consensus Layer (CL) |
| eth1 block | ExecutionPayload |
| eth2 block | BeaconBlock |
| eth1 client (Geth etc.) | Execution client |
| eth2 client (Prysm etc.) | Consensus client / beacon node |

EF retired "eth2" to prevent confusion — people thought eth2 was a separate chain replacing eth1. It's not. They run together, always.

**Q&A clarification:** Two structures exist (ExecutionPayload + BeaconBlock) but only ONE is broadcast on the network. The ExecutionPayload rides inside the BeaconBlock — like a letter inside an envelope.

### Concept 19+20: Attestation Process — LMD-GHOST + Casper FFG

**Attestation data structure:**
```
Attestation {
  aggregation_bits        ← which validators in committee signed
  data: AttestationData {
    slot                  ← which slot this is for
    index                 ← which committee
    beacon_block_root     ← LMD-GHOST vote: "this is my chain head"
    source                ← FFG vote: last justified checkpoint (must be justified)
    target                ← FFG vote: current epoch checkpoint (trying to justify)
  }
  signature               ← BLS (aggregatable)
}
```

**Four roles per slot:**

| Role | Who | How chosen | Job |
|------|-----|-----------|-----|
| Block Proposer | 1 validator | RANDAO | Builds BeaconBlock, includes PREVIOUS slot attestations |
| Attesters | ~21,875 validators | Committee shuffle | Each independently creates + signs own AttestationData |
| Aggregators | ~16 validators | Random BLS subset | Compress many sigs → one aggregate |
| All nodes | Everyone | — | Receive + verify |

**Slot timeline:**
```
t=0s   PROPOSER: calls engine_getPayload() → EL returns ExecutionPayload
                 wraps in BeaconBlock → adds attestations from slot N-1
                 signs + broadcasts on libp2p

t=4s   ATTESTERS (each independently):
                 1. Receive the block
                 2. Run LMD-GHOST → pick chain head
                 3. Pick FFG source + target checkpoints
                 4. Build AttestationData → sign → broadcast

t=4-12s AGGREGATORS:
                 collect individual attestations from committee
                 compress BLS signatures → one aggregate
                 broadcast aggregate

t=12s  Slot ends. Next proposer picks up aggregates for their block.
```

**Critical timing rule:**
```
Slot N attestations → included in slot N+1 (or N+2) block
NOT in slot N block (they vote ON that block — can't exist before it)
```

---

**LMD-GHOST rules:**

| Letter | Meaning |
|--------|---------|
| L | Latest — only count each validator's MOST RECENT attestation |
| M | Message — that vote is an attestation message |
| GHOST | Greedy Heaviest-Observed Sub-Tree — pick fork with most validator weight |

- At each fork: pick child subtree with highest total validator weight
- Vote for a parent block does NOT count toward either child's weight
- Discard all older attestations from same validator the moment a newer one arrives
- Prevents: **vote replay attack** — attacker replaying old votes to inflate stale fork weight

**LMD-GHOST exam trap (corrected):**
```
Validator X: slot 10 → block C  |  slot 15 → block G
LMD counts: slot 15 → block G  (LATEST, not heaviest)

WRONG reasoning: "C has more weight so C counts"
CORRECT reasoning: "slot 15 > slot 10 so G counts — latest message wins"
```

---

**Casper FFG rules:**

- Checkpoint = (epoch_number, first block of that epoch)
- FFG vote = source → target link between two checkpoints
- If ≥2/3 validators cast same source→target vote → target becomes **JUSTIFIED**
- If epoch N justified AND epoch N+1 justified (source=N) → epoch N becomes **FINALIZED**
- Finalization needs TWO consecutive justified checkpoints

**Why two consecutive?** Prevents finalization under network partition. 50% online can justify but not finalize.

**Partition math — CORRECTED:**
- Justification AND finalization both require ≥2/3 (66.7%) of TOTAL validator set
- 55% → cannot justify, cannot finalize. LMD-GHOST keeps running.
- 60% → cannot justify, cannot finalize. LMD-GHOST keeps running.
- 67%+ → can justify AND finalize
- Threshold is TOTAL set (not online set) — prevents attacker isolating honest validators

| Partition % | LMD-GHOST | Can justify? | Can finalize? |
|-------------|-----------|-------------|---------------|
| 55% | ✅ running | ❌ No | ❌ No |
| 60% | ✅ running | ❌ No | ❌ No |
| 67%+ | ✅ running | ✅ Yes | ✅ Yes |

---

**Block validity rules for attestations:**

| Rule | If violated |
|------|------------|
| Age: current or previous epoch only | Rejected (too old) |
| No duplicates (already on-chain) | Rejected (double-counting = fork choice/FFG manipulation) |
| FFG source must be justified checkpoint | Rejected |
| Committee assignment must match slot | Rejected |
| Valid BLS signature | Rejected |

**Why duplicates are dangerous:** Same validators' votes counted twice → artificially inflate fork choice weight OR fake 2/3 supermajority for justification.

**LMD-GHOST vs Casper FFG — side by side:**

| | LMD-GHOST | Casper FFG |
|---|---|---|
| **What it tracks** | Individual blocks | Epoch checkpoints |
| **Time horizon** | Now — which head to build on | Past — which history is final |
| **Resolution** | Every slot | Every epoch |
| **Output** | Current chain head | Justified + finalized checkpoints |
| **Safety guarantee** | Probabilistic (can be reorged) | Cryptographic (cannot be reversed) |
| **Fails under partition** | ❌ No — keeps running | ✅ Yes — stalls (safely) |

---

**Equivocation (Byzantine double-voting):**

- Specific term: **equivocation** (not just "Byzantine fault")
- What it is: sending two different attestations for the same slot to different peers
- Consequences:
```
1. 1/32 stake burned immediately (~1 ETH on 32 ETH)
2. Forced exit — enters exit queue immediately
3. ~36 day withdrawable period with further penalties
4. Correlation penalty: scales with how many validators slashed simultaneously
   → small solo incident: ~1/32
   → large coordinated attack: up to 100% of stake
```
- Why harder than crash: crash = silent (every peer sees nothing). Byzantine = actively sends different messages to different peers (requires collecting BOTH conflicting attestations + comparing signatures to prove)

- Status: ✅ understood

### Concept 21: Gasper — LMD-GHOST + Casper FFG Working Together

**Rule 1: Finalized blocks are never reorged**
```
LMD-GHOST picks any chain head EXCEPT it can never go below the last finalized checkpoint.
Finality = hard floor. LMD-GHOST operates above it.

[epoch 8 FINALIZED] → [epoch 9] → [epoch 10 head A]  ← LMD picks here
                                 ↘ [epoch 10 head B]  ← or here
LMD CANNOT go below epoch 8. Ever.
```

**Rule 2: LMD-GHOST must build on the justified chain**
- LMD-GHOST picks heaviest chain that descends from the LAST JUSTIFIED checkpoint
- A heavier fork that branches before the justified checkpoint = invalid, ignored
- Why: building on a pre-justification fork would break the two-consecutive-justified requirement

**Rule 3: One attestation advances both simultaneously**
```
AttestationData processed by two layers independently:
LMD-GHOST  → updates fork choice weights immediately
Casper FFG → accumulates toward justification/finalization
No extra messages. No extra round trips.
```

**When they appear to conflict → FFG always wins:**
- If LMD-GHOST head is on a fork that ignores a justified checkpoint
- → That fork's FFG source is invalid → block rejected
- → Honest validators will not build on it

**Full picture:**
```
FINALIZED          JUSTIFIED         CHAIN HEAD
(FFG hard floor)   (LMD must extend  (LMD picks
                    from here)        heaviest above)
     │                  │                  │
─────╪──────────────────╪──────────────────╪────
  epoch 7            epoch 12          current slot
```

**Why two layers instead of one:**

| Property | LMD-GHOST alone | Casper FFG alone | Gasper (both) |
|----------|----------------|-----------------|---------------|
| Liveness (keeps producing blocks) | ✅ | ❌ can stall | ✅ |
| Finality (irreversible history) | ❌ | ✅ | ✅ |
| Handles network partition | ✅ keeps going | ❌ stalls | ✅ degrades gracefully |
| Security against reorgs | Probabilistic | Cryptographic | Both |

**During partition (55/45 split for 3 epochs):**
- LMD-GHOST: both sides keep producing blocks (liveness preserved)
- Casper FFG: neither side reaches 2/3 → no new justifications → no finalization (safety preserved)
- On reconnect: validators reconcile fork choice → FFG resumes justifying → finality catches up

- Status: ✅ understood

### Concept 22: RANDAO — How Proposers Are Selected Randomly

**The problem:** Need randomness that is unpredictable, unbiasable, verifiable, and in-protocol. No trusted external source.

**How RANDAO works:**
```
Each block proposer contributes randomness via:
  randao_reveal = BLS_Sign(private_key, current_epoch)

Mixed into beacon state:
  new_randao = XOR(current_randao_mix, hash(randao_reveal))

After 32 slots per epoch → 32 reveals mixed → epoch's final randomness seed
Stored in: BeaconState.randao_mixes[] (one per epoch, kept 8,192 epochs)
```

**Why a BLS signature?**
- Anyone can verify with validator's public key
- Cannot produce different valid signature for same epoch
- Deterministic: same key + same epoch = same reveal always

**Three uses of same seed (no collision via domain tags):**
```
Proposer selection:   hash(seed + slot  + DOMAIN_BEACON_PROPOSER)
Committee shuffle:    hash(seed + epoch + DOMAIN_BEACON_ATTESTER)
Sync committee:       hash(seed + epoch + DOMAIN_SYNC_COMMITTEE)
```
Different domain tag = completely different output from same seed.

**How validators know their duties (self-computed locally):**
```
Every validator runs locally:
  compute_proposer_index(state, slot, seed) → if result == my_index → I propose
  compute_committee(state, epoch, seed)     → if I appear → I attest

No election. No communication. Pure deterministic computation.
Same seed + same function + same inputs = same output on every node.
```

**Lookahead window:**
```
Epoch N seed finalized at END of epoch N-1
→ Duties for epoch N known at START of epoch N (just in time)
→ NOT known 10 epochs ahead → limits targeted DDoS window
```

**Proposer frequency:**
```
700,000 validators ÷ 32 slots/epoch = ~21,875 epochs per selection
21,875 × 6.4 min = ~97 days between proposals per validator (~once every 3 months)
Each epoch = 32 DIFFERENT proposers (one per slot)
Fresh random draw each epoch — past selection doesn't exclude future selection
```

**Can a validator be proposer + attester + aggregator simultaneously?**

YES — all roles use same seed, different domains → no collision.

| Role | Selected by | Can overlap with other roles? | Job |
|------|------------|------------------------------|-----|
| Proposer | RANDAO + DOMAIN_BEACON_PROPOSER | ✅ Yes | Build block, include prev-slot attestations |
| Attester | RANDAO + DOMAIN_BEACON_ATTESTER | ✅ Yes | Create + sign own attestation for current slot |
| Aggregator | hash(BLS_Sign(slot)) % MODULUS == 0 | ✅ Yes (subset of committee) | Compress committee attestations |
| None | — | — | Most common — assigned to different slot |

**Full duty order when all three roles apply (e.g. slot 50):**
```
t=0s    PROPOSER: build block + include slot 49 attestations → broadcast
t=4s    ATTESTER: attest TO slot 50 block → sign → broadcast
t=4-12s AGGREGATOR: compress committee's slot 50 attestations → broadcast
                    → go to slot 51's proposer, NOT own block

Key: cannot include own aggregated attestations in own block
     — they don't exist yet when block is built at t=0s
```

**Aggregator self-selection (NOT a domain shuffle):**
```
Each committee member checks:
  slot_signature = BLS_Sign(private_key, slot_number)
  is_aggregator  = hash(slot_signature) % AGGREGATOR_MODULUS == 0

If true → you are aggregator this slot (~1 in 16 committee members pass)
```

**The bias problem:**
```
Last proposer of epoch has a choice:
  Reveal → mix in randao_reveal → get block reward
  Withhold → skip reveal → epoch randomness changes

Cost: lose block reward (~0.02 ETH)
Gain: choose from 2^K randomness outcomes (K = consecutive last proposers controlled)
```

**VDF solution (in research, not yet deployed):**
```
RANDAO output (biasable) → VDF (takes ~1 epoch wall-clock time) → unbiasable seed

VDF = Verifiable Delay Function:
  - Fixed unavoidable compute time (even with parallel hardware)
  - Cannot be sped up
  - By the time last proposer could withhold, VDF already running → cannot affect outcome
```

**Partition edge cases:**
| Situation | LMD-GHOST | FFG | Notes |
|-----------|-----------|-----|-------|
| >66.7% online | ✅ running | ✅ justify + finalize | Normal |
| 50–66.7% online | ✅ running | ❌ stalled | Reconnect → FFG resumes |
| <50% online | ✅ running (lighter fork, reorged on reconnect) | ❌ stalled | — |
| <33% online | ✅ running | ❌ stalled | Inactivity leak drains offline stake until online > 66.7% |

**Inactivity leak:** Protocol detects stalled finality → drains offline validators' balances → online share of total grows → eventually crosses 66.7% threshold → finality resumes.
