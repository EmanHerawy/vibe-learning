## One-Paragraph Summary

Solana completely separates code from data. Programs are stateless executable accounts; all state lives in separate data accounts owned by programs. Every transaction must declare its accounts upfront — enabling the runtime to parallelize non-conflicting transactions. Access control is enforced via the `owner` field (only the owner program can write to an account) and `is_signer` (the transaction must be signed by the account's private key). The System Program is Solana's kernel: the only program that can create accounts and assign ownership. All regular wallets are owned by the System Program and protected by signature verification.

## One-Page Summary

### Ethereum vs Solana Mental Model

| | Ethereum | Solana |
|---|---|---|
| Code + data | Fused in contract | Separated: program ≠ data account |
| Contract/program | Stateful (owns storage) | Stateless (owns nothing) |
| Data lives at | Contract address | Separate account addresses |
| Access control | Contract logic | `owner` field + `is_signer` |
| Parallelism | Hard (implicit state) | Built-in (explicit account lists) |

### Solana Account Structure (KLODRE)
| Field | Type | Meaning |
|-------|------|---------|
| `key` | Pubkey | The address |
| `lamports` | u64 | SOL balance |
| `owner` | Pubkey | Program that can write to this account |
| `data` | Vec<u8> | Arbitrary byte storage |
| `rent_epoch` | u64 | Legacy rent field |
| `executable` | bool | Is this a program? |

### Programs
- Account where `executable = true`
- Holds BPF bytecode in `data` field
- Stateless — operates on accounts passed to it
- Analogy: Unix binary (`/usr/bin/ls`) — code without state

### Transaction Account Flags
- `is_signer: true` — tx must be signed by this account's private key (authorization)
- `is_writable: true` — runtime allows modifying this account's data/lamports
- Both checked by runtime BEFORE program execution

### Why Explicit Accounts?
- Runtime reads all account lists before execution
- Detects conflicts (shared accounts between txs)
- Non-conflicting txs → parallel execution
- Conflicting txs → sequential execution
- This is Solana's core throughput mechanism

### System Program
- Address: `111...111`
- Native — baked into the runtime, not deployed bytecode
- ONLY program that can: create accounts, assign ownership
- Owns all regular wallet accounts
- Protects wallets via `is_signer` check (requires your private key signature)
- Analogy: OS kernel vs userspace programs

### Solana AA Equivalent
- Custom wallet programs (e.g. Squads multisig)
- Program owns the wallet account instead of System Program
- Program defines custom authorization logic

## Key Takeaways
1. **Everything is an account** — wallets, programs, data, all accounts
2. **Owner = only writer** — the `owner` field is the fundamental access control primitive
3. **Declare upfront, run in parallel** — explicit account lists enable Solana's throughput
4. **System Program = kernel** — only gatekeeper for account creation and SOL transfers
5. **is_signer = authorization** — your private key signature is how you prove control

## Questions & Confusions Raised This Session
- Q: Is calling a Solana program like `delegatecall` in Ethereum?
  → Partially — both separate code from data. But Solana is 1:many (multiple accounts), delegatecall is 1:1. Solana accounts are explicit; delegatecall storage is implicit.
- Q: Does the user create a program that owns their own wallet?
  → No. Regular wallets are owned by System Program. User's pubkey is the ADDRESS, not the owner. Custom wallet programs (Squads) are the AA equivalent.
- Q: What is the System Program and how is it different?
  → Native runtime program. Only one that can create accounts and assign ownership. Analogy: OS kernel.
