**Rule:** invoke_signed takes 3 arguments:
```rust

    instruction,    // WHAT: the instruction to execute
```

## Gap Plugging Notes

### Re-teach: PDAs

**WHY:** Programs are stateless. State lives in separate accounts. PDAs let you find those accounts deterministically without a lookup table.

**What is a PDA:**
- Address derived from: `hash(seeds + program_id + bump)`
- Off the Ed25519 curve → no private key (cryptographically impossible, not just forbidden — like √-1 in real numbers)
- Owned by the deriving program
- Deterministic: same inputs → same address, always

**Seeds:** Arbitrary bytes you choose to encode meaning.
```
[b"savings_account", user_pubkey]  → one account per user
[b"config"]                        → one global config
```
Different seeds → different address. Different program_id → different address (programs can't access each other's PDAs).

**Canonical bump:** Highest bump (closest to 255) that makes the hash land off-curve. Stored in the account at creation — never recomputed on lookup.

**Identity check via seeds constraint:**
```rust
#[account(seeds = [b"vault", authority.key().as_ref()], bump)]
pub vault: Account<'info, Vault>,
```
Anchor re-derives the PDA using the signer's pubkey. Attacker passes their own PDA → derived address doesn't match → rejected. The CONSTRAINT catches it, not the runtime blindly.

### Re-teach: CPIs & PDA Signing

**WHY:** Programs are isolated — can only write to accounts they own. CPI is the only door between programs.

```
CpiContext::new(...)              → user wallet is signer (already signed outer tx, propagates in)
CpiContext::new_with_signer(...)  → PDA is signer (no private key — pass seeds instead)
```

**new_with_signer — what actually happens:**
1. You pass signer_seeds
2. Runtime re-derives PDA from seeds + your program_id
3. Checks: does derived address == account passed in?
4. If yes → marks that account as valid signer for this CPI
5. Target program sees "PDA signed" — no cryptographic signature involved

**Authorization chain:**
```
Runtime      → verifies PDA address matches seeds ("it can sign")
Target prog  → checks if that signer is actually authorized (e.g., is it the mint_authority on the Mint?)
```
Runtime does NOT check whether your program is allowed to call the target. The target program enforces its own access control.

**Raw invoke vs Anchor CPI:**
| | When |
|---|---|
| Anchor CPI | Target has IDL |
| `invoke` | No IDL, wallet signed outer tx |
| `invoke_signed` | No IDL, PDA needs to sign |
| Just deserialize | Reading another program's account — no CPI needed |

### Re-teach: Access Control — 3 Checks

| Check | Question | Anchor mechanism |
|---|---|---|
| **Signer** | Did this account sign? | `Signer<'info>` |
| **Ownership** | Is this account owned by the expected program? | `Account<'info, T>` |
| **Identity** | Is this the *specific* account I expect? | `seeds` constraint or `address =` |

**Signer alone is not enough** — proves "someone signed", not "the right person signed."
**Ownership alone is not enough** — proves "valid account from this program", not "the right user's account."
**Identity check** closes the gap: binds the account address to the signer's pubkey via seeds.

**Blind signing / transaction injection:**
```
Transaction
  ├── Instruction 1: "swap 10 USDC"        ← user sees this
  ├── Instruction 2: "approve token account" ← hidden
  └── Instruction 3: "drain entire wallet"  ← hidden
```
One signature → all three execute atomically.
Program-level defense: check `Instructions` sysvar to assert your instruction is the only one in the tx.

### Re-teach: SPL Token

**WHY:** One shared program manages all tokens. You don't deploy logic — you create data accounts.

```
EVM:   each token = separate deployed contract
Solana: SPL Token Program (one) + Mint accounts + Token Accounts
```

**3 account types:**

**Mint Account** — token's birth certificate. Global metadata.
```
decimals        = 6
supply          = 47,000,000
mint_authority  = Circle's PDA   (who can create new tokens)
freeze_authority = Circle's PDA  (who can freeze wallets)
```
One per token. Owned by SPL Token Program.

**Token Account** — one wallet's balance for one specific token.
```
mint    = USDC Mint address
owner   = Alice's pubkey      ← this is stored INSIDE the data, not account ownership
amount  = 500,000,000
frozen  = false
```
One per (owner, mint) pair. Owned by SPL Token Program (not by Alice).

**Associated Token Account (ATA)** — Token Account at a deterministic address.
```
seeds = [owner_pubkey, token_program_id, mint_pubkey]
```
Anyone can derive Alice's USDC address without asking her.

**Where balances live:**
```
Alice's SOL    → lamports on her wallet account (no extra account)
Alice's USDC   → amount field in her Token Account (separate account, costs rent ~0.002 SOL)
Total supply   → supply field in the Mint Account
```

**Who signs what:**
| Operation | Signer | PDA needed? |
|---|---|---|
| `mint_to` | `mint_authority` (PDA) | ✅ `new_with_signer` |
| `transfer` | Token Account owner (wallet) | ❌ `new` |
| `burn` | Token Account owner (wallet) | ❌ `new` |
| `freeze_account` | `freeze_authority` (PDA) | ✅ `new_with_signer` |
| `thaw_account` | `freeze_authority` (PDA) | ✅ `new_with_signer` |
| `close_account` | Token Account owner (wallet) | ❌ `new` |

Rule: protocol-controlled ops (mint/freeze/thaw) → PDA signs. User-controlled ops (transfer/burn/close) → wallet signs.

**freeze_authority — permanent decision:**
Set at Mint creation. Cannot be added later. Omit it → that Mint can never freeze accounts. Ever.
```rust
#[account(init, mint::authority = config)]  // freeze_authority defaults to None — permanent
```

**NFTs — 3 rules:**
```
decimals = 0           → mint::decimals = 0
supply = 1             → mint_to(amount = 1)
mint_authority = None  → set_authority(→ None)  ← THIS makes it an NFT permanently
```
Each NFT = its own Mint account (unlike ERC721 where one contract tracks all tokenIds).
Collection membership: always check `verified = true` in Metaplex metadata, not just `collection.key`.

### Re-teach: Token-2022

**WHY:** SPL Token is frozen — can't add features without breaking existing tokens. Token-2022 is a second token program deployed alongside it, with an opt-in extension system.

```
SPL Token Program  → original, frozen
Token-2022 Program → same interface + extensions
```
Both live on Solana. When you create a Mint, you choose which program owns it. That choice is permanent.

**Extensions:** Opt-in features chosen at Mint creation. Cannot be added after. Baked into the Mint account at init.

**Key extensions:**

**TransferFee** — takes a % on every transfer automatically.
```
Alice sends Bob 1000 tokens, fee=2% → Bob gets 980, fee collector gets 20
```
Auditor: check fee math rounding on small amounts. Who controls the fee collector?

**NonTransferable** — token cannot be transferred. Ever. Soulbound.
Use case: credentials, certifications, KYC badges tied to a wallet.

**PermanentDelegate** — one address can transfer/burn tokens from ANY holder without their permission.
```
PermanentDelegate = Protocol PDA
Protocol calls transfer(from=Alice, to=Treasury) — Alice never signed
```
Use case: staking (protocol controls staked tokens), vesting (clawback), compliance.
Auditor: who holds the delegate? Is it disclosed? Can they drain any holder?

**TransferHook** — every transfer triggers a CPI into your program. Your code runs mid-transfer.
```
Normal transfer:    tokens move. done.
Hook transfer:      tokens move → Token-2022 calls YOUR program → your code runs → completes
```
Use case: allowlist checks, game logic, sanctions screening on every transfer.

3 attack vectors:
1. **Upgradeable hook** — if hook program has upgrade_authority, logic can change silently. Audit: upgrade_authority = None?
2. **Reentrancy** — hook fires mid-transfer, calls back into token contract, state is inconsistent mid-operation.
3. **DoS** — expensive hook (e.g. 10k-address blocklist scan) burns compute, spammed by attacker, blocks legitimate txs.

**ConfidentialTransfer** — encrypted amounts via ZK proofs. Nobody can see transfer amounts on-chain.
Auditor: specialist audit needed. ZK proof verification bugs, overflow in encrypted space.

## Key Takeaways

1. **invoke_signed requires every account explicitly** — instruction, account_infos (all accounts + the program), signer_seeds. Nothing is inferred. Source: rareskills.io — CPI section
