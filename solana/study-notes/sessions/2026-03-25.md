## Theory Notes

### Concept 1: Only the owner program can write to an account's data
- Status: ✅ understood
- Enforced by the Solana runtime — not Anchor, not your code
- A program that is NOT the account's owner cannot write to its data field
- Contrast with EVM: a Solidity contract writes to its own storage implicitly — no external enforcement. In Solana, ownership is external and runtime-enforced.
- Auditor note: `Account<'info, T>` in Anchor automatically checks that the account is owned by the current program — this is a free owner check you get for any typed account

### Concept 2: System Program creates accounts and transfers ownership
- Status: ✅ understood
- Only the System Program can create new accounts (allocate space, set initial lamports)
- At creation, System Program transfers ownership to whatever program you specify
- After transfer: the new owner program can write freely; System Program can no longer touch the data
- This is what `init` + `payer` does under the hood: calls System Program → allocate → transfer ownership to your program
- Flow: System Program owns everything at genesis → your `init` instruction → System Program creates + transfers → your program owns

### Concept 3: Programs are stateless — all state lives in accounts they own
- Status: ✅ understood
- Program account = code only (executable = true, owned by BPF Loader)
- Program data = separate accounts owned by your program, designed by you
- No implicit storage like Solidity's `mapping` — you design every account layout

```
EVM:
  contract → has implicit storage slots (mapping, uint, etc.)

Solana:
  program (code) → owns → data accounts (you design the layout)
```

- This is why explicit account lists exist: every piece of state your instruction touches must be declared upfront

### Concept 4: Rent — accounts must stay funded or get deleted
- Status: ✅ understood
- Every byte stored on-chain costs lamports per epoch
- If lamports fall below the rent-exempt threshold → runtime deletes the account
- Solution: fund accounts to be **rent-exempt** at creation (one-time deposit, account lives forever)
- `init` + `payer = user` + `space = N` → Anchor calculates exact rent-exempt minimum and transfers it from payer

```
Rent-exempt minimum ≈ 0.00089 SOL per KB (rough rule of thumb)
Counter (17 bytes) ≈ ~0.0009 SOL
```

- When account is closed, all lamports are returned to the recipient (rent is fully recoverable)

### Concept 5: Account lifecycle — create, use, close
- Status: ✅ understood

```
1. CREATE: init + payer + space → System Program allocates, transfers ownership, payer pays rent
2. USE:    mut constraint → your program reads/writes freely (it owns the account)
3. CLOSE:  close = recipient → Anchor zeros discriminator, transfers lamports, account deleted
```

### Concept 6: Account Resurrection Attack & the close constraint
- Status: ✅ understood
- **The attack**: when closing an account manually (without Anchor's `close` constraint):
  1. Developer transfers lamports out → account balance = 0
  2. Runtime schedules deletion but NOT instantly — old data still exists briefly
  3. Attacker (in same tx or block) re-funds the account → balance > 0 again
  4. Account revives with old discriminator + old data intact
  5. Any instruction checking the discriminator thinks this is a valid live account
  6. Attacker can interact with a "closed" account — re-initialization, double-spend, etc.

- **Why discriminator zeroing prevents it**: `Account<'info, T>` checks the 8-byte discriminator. If zeroed → type mismatch → instruction fails. Re-funding a zeroed account does nothing useful.

- **Anchor's `close` constraint does this automatically**:
```rust
#[account(mut, close = recipient)]
pub counter: Account<'info, Counter>,
```
  Anchor: zeros discriminator → transfers all lamports → account cleaned up. You do nothing extra.

- **The risk only exists with manual close logic** (raw Solana without Anchor, or developers who hand-roll lamport transfers without zeroing)

- **Auditor checklist for close-like instructions**:

| Pattern | Safe? |
|---|---|
| Anchor `close` constraint | ✅ Safe — discriminator zeroed automatically |
| Manual lamport transfer + discriminator zeroed in code | ✅ Safe if done correctly |
| Manual lamport transfer WITHOUT zeroing discriminator | ❌ Resurrection attack vector |
| Transfer lamports to zero but keep account data | ❌ Same vulnerability |

- Real-world: Jet Protocol had this class of vulnerability (caught in audit). Rektoff Security Roadmap §8.5 — Re-initialization attacks.
