## One-Paragraph Summary

PDAs (Program Derived Addresses) are account addresses derived deterministically from seeds + program_id + a canonical bump byte. They sit mathematically off the Ed25519 curve, which means no private key exists for them — not a policy restriction, but a cryptographic impossibility. Only the program that derived a PDA owns it and can write to it. This solves the core Solana storage problem: a stateless program can always find its data accounts by recomputing their addresses from seeds, without maintaining any lookup table. Different seeds produce different PDAs; different program IDs produce different PDAs even for the same seeds — giving each program its own isolated namespace.

### What is a PDA?
- Address derived from: `hash(seeds + program_id + bump)`
- Off the Ed25519 curve → no private key (cryptographically impossible)
- Owned by the deriving program
- Deterministic: same inputs → same address, always

### Seeds
- Arbitrary bytes chosen by the developer
- Make the address meaningful and findable
- Example: `["score", user_pubkey]` → one score account per user per program
- Different seeds → different hash → different PDA

### Canonical Bump
- The highest bump (closest to 255) that makes the hash land off-curve
- Runtime tries 255, 254, 253... until off-curve
- Stored once, reused — not recomputed every call

### Why PDAs over Keypairs?
| | Keypair | PDA |
|---|---------|-----|
| Address | Random | Deterministic from seeds |
| Private key | Exists | Impossible (off-curve) |
| Findable without storage | No | Yes — recompute from seeds |
| Ownership | Anyone with key | Only the deriving program |

### Security Properties
- Two users → different seeds → different PDAs (isolated per user)
- Two programs → different program_id → different PDAs (isolated per program)
- No external signing possible → only program can authorize writes

### Analogy Map
| Concept | Analogy |
|---------|---------|
| PDA | Locker with no key slot — only the program opens it |
| Seeds | Recipe — same recipe = same dish |
| Canonical bump | Lock it in once, reuse forever |
| No private key | Like √-1 in real numbers — impossible, not forbidden |
| Program_id isolation | Same recipe, different kitchen = different result |

## Key Takeaways
1. **PDA = deterministic + off-curve + program-owned** — these three properties together
2. **Seeds are your namespace** — design them to encode meaning (user, poll_id, "config", etc.)
3. **Canonical bump is stored** — compute once at creation, store in the account, reuse
4. **No private key = cryptographic isolation**, not just access control
5. **Program_id in the hash** = programs cannot access each other's PDAs
