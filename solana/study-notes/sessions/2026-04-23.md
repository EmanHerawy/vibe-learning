## Audit Lessons Learned

### Lesson 1 — Symmetry Check: Every state mutation must be checked in both directions

When reviewing a state variable like `total_staked`, ask:
- What increments it? → check all those paths
- What decrements it? → check all those paths
- Are there flows that bypass BOTH directions?

**Example from StakeFlow reference report:**
The reference auditor flagged `deposit_yield` for not incrementing `total_staked`.
But they missed `withdraw_reserves` for not decrementing `total_staked` either — and
didn't include it in their fix. Both are the same class of bug (state not maintained on vault movement).

**Rule:** For every accounting variable (`total_staked`, `total_locked`, `reward_debt`, etc.),
draw the full in/out map. Any instruction that moves tokens without updating the variable
is a finding — in BOTH directions.

---

### Lesson 2 — Floor Division Exploit: `× A / B` in integer arithmetic

Whenever you see a formula of the form:
```rust
result = value × A / B
```
Ask: **"What does floor division lose here, and can I exploit that loss repeatedly?"**

**Example from StakeFlow (F-02 that we missed):**
```rust
new_reward_debt = reward_debt × remaining / full_amount
// 4 × 499 / 500 = 3.992 → truncates to 3
// Protocol "forgets" 0.992 tokens of already-paid debt
// Call partial_unstake(1) + claim_rewards 499 times → steal ~396 tokens from 4 legitimate rewards
```

The formula is algebraically correct with real numbers.
The bug only exists with integers — floor division accumulates into a loop exploit.

**Checklist for any `× A / B` formula:**
1. What is the maximum truncation per call? (`value mod B / B`)
2. Can the caller control A or B to maximize truncation?
3. Can the caller repeat this in a loop cheaply?
4. Does the lost precision accumulate into a profitable drain?

### Lesson 3 — BPF Stack Overflow: `anchor build` warnings are runtime crash guarantees

**How to find it:**

```bash
# Run from inside the project directory
anchor build 2>&1 | grep -i "stack\|frame\|exceed"

# What a finding looks like:
# Stack offset of 4104 exceeded max offset of 4096 by 8 bytes
# Estimated function frame size: 4160 bytes
```

**What it means:**
- Solana programs have a hard 4096-byte stack limit per frame
- `try_accounts` (Anchor's auto-generated account deserializer) runs BEFORE your instruction code
- If it overflows, the instruction always fails at runtime — not sometimes, always
- `anchor build` succeeds with no compile error — the warning is easy to miss

**Checklist item for every Solana audit:**

```
□ Run: anchor build 2>&1 | grep -i "stack\|frame\|exceed"
□ Any "Stack offset exceeded" warning = guaranteed runtime failure = finding
□ Check both context structs AND instruction functions
□ Severity: Medium minimum (instruction permanently broken)
```

**Fix pattern:**
Wrap large accounts (`TokenAccount`, `Mint`) in `Box<>` to move them from stack to heap:
```rust
// Before — stack overflow
pub stake_vault: InterfaceAccount<'info, TokenAccount>,

// After — heap allocated, only 8-byte pointer on stack
pub stake_vault: Box<InterfaceAccount<'info, TokenAccount>>,
```

**Rule:** `anchor build` warnings ≠ "just warnings" in Solana. Stack frame warnings are runtime crash guarantees.

### Lesson 4 — Share Price Inflation / First-Depositor Attack

Any protocol that issues receipt tokens (stX, LP tokens, shares) based on a ratio of
`user_amount × supply / total_deposited` is vulnerable to this attack.

**The Pattern:**
```
stx_to_mint = amount × stx_supply / total_staked
```

If an attacker can manipulate `stx_supply` or `total_staked` independently of each other,
they can make the ratio round to 0 for victims.

**The Attack (StakeFlow example):**
```
1. Attacker stakes 1001 tokens → receives 1001 stX (1:1 first deposit)
2. Attacker burns 1000 stX directly via SPL Token → supply = 1
3. Victim stakes 500 tokens:
     stx_to_mint = 500 × 1 / 1001 = 0.499 → truncates to 0
     Victim receives 0 stX — cannot unstake — 500 tokens gone forever
4. Attacker unstakes 1 stX:
     tokens_out = 1 × 1501 / 1 = 1501 tokens
     Attacker steals 500 tokens from victim
```

**Why it works:** SPL tokens can be burned by their owner directly — no program permission needed. The program reads `stx_mint.supply` live, which the attacker manipulated externally.

**Checklist item for any receipt-token protocol:**

```
□ Can a user burn receipt tokens externally (outside your program)?
□ Is the mint/exchange formula based on live supply read from mint account?
□ What happens when supply = 1 and total_deposited is large?
   → stx_to_mint = amount × 1 / large_number → likely 0
□ Is the first depositor getting 1:1? If yes, they can set up the attack cheaply.
```

**Fix pattern — "dead shares":**
On the FIRST deposit, mint a small amount of shares (e.g. 1000) to a burn address
(`Pubkey::default()` or protocol PDA) that can never be retrieved. This creates a
permanent supply floor so the ratio can never collapse to near-zero.

```rust
// On first deposit: mint dead shares to prevent supply collapse
if stx_supply == 0 {
    mint_to(burn_address, MINIMUM_LIQUIDITY)?;  // e.g. 1000 dead shares
}
```

**Applies to:** Any Solana staking protocol, AMM, vault, or lending protocol that
issues proportional receipt tokens based on a live supply read.
