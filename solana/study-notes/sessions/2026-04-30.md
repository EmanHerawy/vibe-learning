#### F-02 — Buy amount unbounded (what you missed)

```rust
// buy.rs — NO guard before this line:
missionx_state.reserve1 -= effective_buy_amount;  // underflows if buy_amount > reserve1
```
`reserve1` = 98.5% of supply (MINT_AMOUNT minus payouts). Attacker passes `buy_amount > reserve1` → underflow → reserved payout tokens gone.
You caught the arithmetic risk but never asked: *"Can the caller pass an amount larger than reserve1?"*

## Key Takeaways

1. **State machine blacklist = missing finding** — When a guard excludes only one status, ask "what terminal states can still reach this instruction?" every time. *Source: MissionX `moderate.rs:105`*
2. **Fund sweep ≠ state zeroing** — After any instruction that moves funds (migrate, withdraw), verify that accounting fields (`reserve0`, `reserve1`) are also zeroed; stale state enables phantom-reserve exploits. *Source: MissionX `migrate.rs:171`*
3. **AMM buy inputs need an upper bound** — Always ask `require!(buy_amount < reserve)` — arithmetic underflow checks alone don't catch over-buy; the guard must exist before the AMM math runs. *Source: MissionX `buy.rs:149`*
