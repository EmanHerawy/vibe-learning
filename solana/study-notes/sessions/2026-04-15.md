## Theory Notes

### Vulnerability Taxonomy — Full Map
Source: Rektoff Security Roadmap for Solana

```
Solana Vulnerability Classes
├── 1. Account Validation
│   ├── 1a. Missing ownership check   → use Account<'info, T>
│   ├── 1b. Missing signer check      → use Signer<'info>
│   ├── 1c. Missing identity check    → use seeds constraint or address =
│   └── 1d. Type cosplay              → Anchor discriminator (auto) / manual check
├── 2. Arbitrary CPI                  → use Program<'info, T> or address =
├── 3. PDA Seed Collision             → namespace all seeds uniquely
├── 4. Re-initialization              → guard init_if_needed with is_initialized
├── 5. Arithmetic Bugs                → checked_add/sub/mul/div everywhere
└── 6. Delegate Authority Confusion   → verify owner not delegate
```

### Class 1 — Account Validation Bugs

**WHY:** Solana passes all accounts explicitly. Program must validate every assumption itself. Anchor helps only for what you declare.

**1a. Missing ownership check**
```rust
// ❌ accepts any account, even fake from another program
pub user_data: AccountInfo<'info>,
// ✅ checks account.owner == this program + discriminator
pub user_data: Account<'info, UserData>,
```

**1b. Missing signer check**
```rust
// ❌ no signature required
pub authority: AccountInfo<'info>,
// ✅ requires valid signature
pub authority: Signer<'info>,
```

**1c. Missing identity check**
```rust
// ❌ valid Vault owned by program, but WHICH user's vault?
pub vault: Account<'info, Vault>,
// ✅ only vault derived from THIS signer's pubkey
#[account(seeds = [b"vault", signer.key().as_ref()], bump)]
pub vault: Account<'info, Vault>,
```

**1d. Type cosplay**
Two structs owned by same program. Attacker passes Config where Vault is expected.
Anchor's 8-byte discriminator prevents this automatically with `Account<'info, T>`.
Raw programs without Anchor are vulnerable.

**Auditor checklist:**
```
For every account in the struct:
  □ Account<'info, T> not AccountInfo?
  □ seeds constraint if user-specific?
  □ Signer<'info> if it needs to authorize?
```

### Class 2 — Arbitrary CPI

**WHY:** Programs accept the program to call as an account. If unchecked, attacker passes their own malicious program.

```rust
// ❌ token_program unchecked
pub token_program: AccountInfo<'info>,

// ✅ fix
pub token_program: Program<'info, Token>,
// or
#[account(address = spl_token::ID)]
pub token_program: AccountInfo<'info>,
```

**Auditor rule:** Every program account passed as parameter must have `address =` constraint or use `Program<'info, T>`. `AccountInfo` with no constraint = flag.

---

### Class 3 — PDA Seed Collision

**WHY:** If seeds have no namespace prefix AND user controls seed values, attacker can derive any existing PDA by choosing the right input.

**Attack:** Attacker can't overwrite the PDA (init fails), but can pass the victim's PDA as their own "valid" account. Program thinks it's the attacker's account → they operate on victim's data.

**Fix:** Always namespace seeds with a unique type prefix:
```rust
seeds = [b"profile", user.key().as_ref()]  // UserProfile
seeds = [b"vault",   user.key().as_ref()]  // UserVault — different address
```

**Auditor rule:** List all PDAs in program. Check: can two different account types produce the same address for the same inputs?

---

### Class 4 — Re-initialization Attack

**WHY:** `init_if_needed` skips account creation if account exists but still runs instruction code. Attacker re-calls initialize to overwrite state.

```rust
// ❌ init_if_needed without guard
#[account(init_if_needed, payer = payer, space = 8 + 32)]
pub config: Account<'info, Config>,

// ✅ fix — guard in instruction code
require!(!ctx.accounts.config.is_initialized, ErrorCode::AlreadyInitialized);
```

Also: manual lamport transfer without zeroing discriminator → account resurrection (zombie account). Anchor's `close` constraint handles this automatically.

**Auditor rule:** Every `init_if_needed` is a red flag. Check for `is_initialized` guard.

---

### Class 5 — Arithmetic Bugs

**WHY:** Rust integer overflow/underflow is silent in release mode. User-controlled values go into arithmetic.

**Three variants:**
```rust
// 5a. Underflow
vault.balance -= amount;  // wraps to u64::MAX if amount > balance

// 5b. Overflow
total_supply += mint_amount;  // wraps to 0 if > u64::MAX

// 5c. Division by zero + precision loss in share math
let shares = amount * pool.total_shares / pool.total_deposits;
// → panics if total_deposits == 0 (first deposit)
// → overflows u64 in intermediate multiplication
// → first depositor gets 0 shares if total_shares == 0
```

**Fix for share math (production pattern):**
```rust
let shares = if pool.total_deposits == 0 {
    amount  // first deposit: 1 share per token
} else {
    (amount as u128)
        .checked_mul(pool.total_shares as u128).ok_or(ErrorCode::Overflow)?
        .checked_div(pool.total_deposits as u128).ok_or(ErrorCode::DivisionByZero)?
        as u64
};
```
Cast to u128 for intermediate calculation — share math overflows u64 easily in production.

**Auditor rule:** Every arithmetic op on user-controlled values → verify checked math. Raw `+`, `-`, `*` = flag.

### Class 6 — Delegate Authority Confusion

**WHY:** SPL Token accounts have two authority fields:
```
Token Account
  ├── owner    → permanent full control
  └── delegate → temporary limited control (set via approve())
```

Programs check "did authority sign?" but forget to check "is authority the OWNER?"
Attacker gets delegate approval elsewhere, calls your program as delegate, drains the account.

**Fix:**
```rust
require!(
    ctx.accounts.token_account.owner == ctx.accounts.authority.key(),
    ErrorCode::InvalidAuthority
);
```

**Auditor rule:** Any instruction operating on a Token Account → verify signer is `token_account.owner`, not just any signer.

### Key Mechanisms
- **AAARDS taxonomy**: Account validation → Arbitrary CPI → seeDcollision → Re-init → aRithmetic → Delegate. Every audit starts with this checklist. Source: Rektoff Security Roadmap
- **u128 intermediate pattern**: `(a as u128).checked_mul(b as u128)?.checked_mul(c as u128)? as u64` — prevents u64 overflow in reward/share math. Source: Rektoff §8.6
- **Token account MOID checklist**: Mint + Owner + Is-owner-not-delegate + Deterministic-address — apply to every TokenAccount in any DeFi instruction. Source: rareskills.io — SPL Token

### Key Distinctions
| Confused pair | Distinction |
|---|---|
| Re-init = privilege escalation vs state reset | State reset is enough — resetting balance to 0 is the attack |
| owner vs delegate on Token Account | owner = permanent full control. delegate = temporary limited (via approve()). Always check owner. |
| Type cosplay with Anchor vs raw | Anchor: discriminator check auto-prevents it. Raw programs: no protection. |
| Program<'info, T> vs AccountInfo | Program<'info, T> checks address == expected program. AccountInfo = no check = arbitrary CPI risk. |

### Gotchas
- `init_if_needed` always flag — check for `is_initialized` guard in instruction body
- PDA seed collision: victim account not overwritten, but attacker operates on it as their own
- Pool token accounts need identity constraints too, not just user accounts
- Zero reward_rate = info severity (silent, not exploitable for theft)
- `amount + reward` in transfer — two separate overflow risks on one line

### Code Patterns to Remember

    as u64;

## Key Takeaways

1. **Every account in an Anchor struct needs 3 checks: OSI** — Ownership (Account<'info,T>), Seeds (identity constraint), Is-signer (Signer<'info>). Missing any one = vulnerability.
   Source: Rektoff Security Roadmap §8.2-8.3

2. **Re-initialization attacks goal is state corruption, not privilege** — resetting a user's staked balance to 0 while their tokens remain locked is a critical bug. `init_if_needed` without `is_initialized` guard = flag every time.
   Source: Rektoff Security Roadmap §8.5

3. **Token account auditing requires MOID** — Mint, Owner, Is-owner-not-delegate, Deterministic-address. Missing mint validation lets attackers deposit worthless tokens and withdraw valuable ones.
   Source: rareskills.io — SPL Token section
