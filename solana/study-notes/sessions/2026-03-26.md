## Theory Notes

### Concept 1: Cross-Program Invocation (CPI)
**Source:** rareskills.io/solana-tutorial — CPI section
**Summary:** A CPI is when your program calls an instruction on another program mid-execution. Programs are isolated — CPI is the only door between them. If the CPI fails, the whole transaction rolls back.
**Bridged from:** Solidity's `IERC20(addr).transfer(...)` — calling contract becomes msg.sender. In Solana, you must explicitly pass signer seeds instead.
**Misconception flagged:** `new_with_signer` doesn't generate a cryptographic signature — the runtime re-derives the PDA from seeds and marks it as signer if it matches. No private key involved.
**Status:** ✅ understood

```
CpiContext::new(...)              → user (wallet) is signer — invoke()
CpiContext::new_with_signer(...)  → PDA is signer — invoke_signed()
```

### Concept 2: PDA Signing in CPI
**Source:** rareskills.io/solana-tutorial — PDA signing section
**Summary:** PDAs have no private key so they can't sign transactions. Instead, you pass seeds to new_with_signer — the runtime re-derives the PDA address from those seeds and if it matches the account, marks it as a valid signer for that CPI only.
**Bridged from:** vault-anchor withdraw (commented-out code) showed why this pattern exists.
**Checkpoint Q:** How does the Token Program know the PDA signature is legitimate without knowing your seeds?
**Learner answer:** Runtime re-derives the PDA from seeds passed in signer_seeds, checks it matches the account address, then marks it as signer. Token Program just sees "this account signed."
**Status:** ✅ understood

```rust
let seeds = &[b"authority", &[bump]];
let signer = &[&seeds[..]];
CpiContext::new_with_signer(program, accounts, signer)
```

### Concept 3: Signer Check vs Identity Check vs Ownership Check
**Source:** rareskills.io/solana-tutorial + Rektoff Security Roadmap §8.2, §8.3
**Summary:** Three distinct checks — signer (did this account sign?), identity (is this the RIGHT account?), ownership (is this account owned by the expected program?). Missing any one of them is a vulnerability.
**Misconception flagged:** `Signer<'info>` only checks "did this sign?" — not "is this the correct signer?" An attacker's PDA can also sign.
**Status:** ✅ understood

| Check | Anchor mechanism | What it prevents |
|---|---|---|
| Signer | `Signer<'info>` | Unsigned tx |
| Ownership | `Account<'info, T>` | Fake accounts from other programs |
| Identity | seeds constraint or `address =` | Right program, wrong user's account |
| Type | discriminator (automatic) | Passing wrong account type |

**Auditor note:** In cpi-demo's responder, `authority: Signer<'info>` only verifies signing — any PDA from any program could pass. Missing identity check = any program can increment any counter.

### Concept 4: Account Resurrection via Seed Binding (access control pattern)
**Source:** rareskills.io/solana-tutorial
**Summary:** Binding a user's pubkey into PDA seeds `[b"vault", user.key()]` ties account ownership to identity. An attacker signing as themselves derives a different PDA address — the constraint fails. No separate `require!` needed.
**Checkpoint Q:** Can a different user withdraw from someone else's vault?
**Learner answer:** No — PDA is linked to user's pubkey in seeds. Different signer → different PDA derived → doesn't match account passed in → constraint fails.
**Status:** ✅ understood

### Concept 5: Raw invoke vs invoke_signed
**Source:** rareskills.io/solana-tutorial — Raw CPI section
**Summary:** `invoke` is used when a wallet (which signed the outer tx) needs to be signer in a CPI — the runtime propagates that signature inward automatically. `invoke_signed` is used when a PDA needs to sign — you pass seeds and the runtime verifies the address derivation.
**Misconception flagged:** `AccountMeta signer=true` doesn't grant signing authority — it declares intent. The runtime checks if that account actually signed the outer transaction.
**Status:** ✅ understood

```
invoke()         → wallet signed outer tx → authority propagates into CPI
invoke_signed()  → PDA needs to sign → pass seeds → runtime verifies
```

**When to use raw invoke (vs Anchor CPI):**
- Target program has no Anchor IDL
- Target is non-Anchor (System Program, Token Program called manually)
- Lower compute cost (no Anchor overhead)

**Reading other program state — NOT via invoke:**
```rust
let data = other_account.try_borrow_data()?;
let state = OtherStruct::try_deserialize(&mut &data[..])?;
// No CPI needed — just deserialize the AccountInfo bytes directly
```

### Concept 6: Blind Signing / Transaction Injection Attack
**Source:** Rektoff Security Roadmap §8.1
**Summary:** Solana transactions can bundle multiple instructions. A malicious dApp shows you one harmless instruction but bundles harmful ones underneath. The wallet signature on the outer tx authorizes everything inside it.
**Fix at program level:** Use `Instructions` sysvar to check all instructions in the current tx. High-value programs can require they be the only instruction.
**Status:** ✅ understood

### Concept 7: SPL Token Model — Mint, Token Account, ATA
**Source:** rareskills.io/solana-tutorial — SPL Token section
**Summary:** SPL Token is a single deployed program that manages ALL tokens on Solana. You don't deploy token logic — you create data accounts (Mint, Token Accounts) and CPI into SPL Token to operate on them.
**Bridged from:** ERC20 — one contract per token with internal balances mapping. In Solana, one shared program, balances live in Token Accounts.
**Status:** ✅ understood

```
Mint Account          → describes the token (decimals, supply, mint_authority)
Token Account (ATA)   → holds one wallet's balance for one mint
SPL Token Program     → owns both, provides mint/transfer/burn logic
```

**Key rules:**
- One Mint per token
- One Token Account per (owner, mint) pair — the ATA
- ATA address = PDA derived from [owner, token_program_id, mint] via Associated Token Account Program
- SOL balance = lamports on wallet account (no extra account)
- Token balance = amount field in Token Account (separate account)

### Concept 8: Mint Account ownership and authority
**Source:** rareskills.io/solana-tutorial
**Checkpoint Q1:** Who owns the Mint account after initialize?
**Learner answer:** SPL Token Program owns it. Same as System Program owns regular accounts — wallet authorizes via owner field stored in data, not via account ownership.
**Status:** ✅ understood

**Checkpoint Q2:** What can the mint_authority PDA do that nobody else can?
**Learner answer:** PDA is the gatekeeper — only the program that knows the seeds can re-derive it and sign. No one else can call mint_to on that Mint because they can't produce a valid signature from that PDA.
**Status:** ✅ understood

**Checkpoint Q3:** How to restrict mint_token to only the original authority?
**Options identified:**
- Store authority pubkey in config, use constraint check: `config.authority == payer.key()`
- Bind authority into PDA seeds: `[b"config", authority.key().as_ref()]`
- Use `address =` constraint on payer

**Current vulnerability:** `[b"config"]` seeds — no authority bound in. Anyone can call `mint_token`. Open question being discussed.
**Status:** 🔄 in progress

### Concept 9: `address =` Anchor Constraint
**Source:** rareskills.io/solana-tutorial — Anchor constraints
**Summary:** `#[account(address = some_pubkey)]` checks that the account's public key equals a specific expected value — rejected before instruction code runs. Equivalent to `require!(account.key() == expected)` but at the constraint layer.
**Use case:** Hardcoded allowlist — restrict an account to one specific address.
```rust
#[account(address = config.authority)]
pub payer: Signer<'info>,
```
**Status:** ✅ understood

### Concept 10: `associated_token::mint` and `associated_token::authority` constraints
**Source:** rareskills.io/solana-tutorial — Anchor SPL constraints
**Summary:** These constraints tell Anchor how to derive and verify the ATA address. Anchor re-derives the ATA PDA from `(authority, token_program, mint)` and checks the account passed in matches. Same idea as seeds/bump — just Anchor doing the ATA derivation for you.
```rust
#[account(
    init_if_needed,
    associated_token::mint = mint,           // this ATA is for THIS mint
    associated_token::authority = recipient, // this ATA is owned by recipient
)]
pub recipient_ata: Account<'info, TokenAccount>,
```
**`init_if_needed` feature flag:** must be enabled in Cargo.toml:
```toml
anchor-lang = { version = "0.32.1", features = ["init-if-needed"] }
```
**Status:** ✅ understood

### Concept 11: Mint Keypair vs PDA — why Mint needs a Keypair
**Source:** rareskills.io/solana-tutorial
**Summary:** The Mint account is NOT a PDA — it's a regular account at an arbitrary address. The Keypair proves ownership of that address at creation time. The Keypair must sign the init tx. After creation, SPL Token Program owns it and the keypair is never needed again.
**Contrast:** Config PDA = derived from seeds, no keypair. Mint = arbitrary address, requires keypair to create.
```typescript
const mint = Keypair.generate();
await program.methods.initialize(maxSupply)
  .accounts({ mint: mint.publicKey, ... })
  .signers([mint])  // ← mint keypair signs once at creation
  .rpc();
```
**Status:** ✅ understood

### Concept 12: Rust Borrow Checker — mutable + immutable borrow conflict
**Source:** Rust ownership rules
**Summary:** When you take `&mut ctx.accounts.config`, you can't also call `ctx.accounts.config.to_account_info()` (immutable borrow) in the same scope. Fix: extract what you need before the mutable borrow.
```rust
// ❌ fails — mutable borrow active when immutable borrow attempted
let config = &mut ctx.accounts.config;
... authority: ctx.accounts.config.to_account_info() ...

// ✅ fix — extract before mutable borrow
let bump = ctx.accounts.config.bump;
let config_info = ctx.accounts.config.to_account_info();
let config = &mut ctx.accounts.config;
```
**Status:** ✅ understood

### Concept 13: Mint Authority Access Control — three patterns
**Source:** rareskills.io/solana-tutorial + Rektoff Security Roadmap §8.3
**Summary:** Current token-minter-anchor lets anyone call mint_token — no authority restriction. Three ways to fix:

**Option A — store authority in config, constraint check:**
```rust
// In initialize: config.authority = authority.key();
#[account(mut, seeds = [b"config"], bump = config.bump,
    constraint = config.authority == payer.key() @ ErrorCode::Unauthorized)]
pub config: Account<'info, MintConfig>,
```

**Option B — bind authority into PDA seeds:**
```rust
// seeds = [b"config", authority.key().as_ref()]
// Only original authority can ever derive this PDA
```

**Option C — `address =` constraint:**
```rust
#[account(address = config.authority)]
pub payer: Signer<'info>,
```

**Current vulnerability:** seeds = `[b"config"]` — no authority bound. Anyone can call `mint_token`.
**Chosen fix:** Option B — bind authority into seeds: `[b"config", authority.key().as_ref()]`. Cleaner — access control enforced by PDA derivation itself, no extra constraint or stored field.
**Status:** ✅ understood ( concept clear)

### Sub-topic: When to use raw invoke vs Anchor CPI — decision tree
```
Calling another program?
  ├── Target has Anchor IDL?
  │     └── YES → use Anchor CPI helpers (typed, safe, compile-time checked)
  │     └── NO  → use raw invoke / invoke_signed
  │
  └── Just reading their account data?
        └── Don't invoke at all — deserialize AccountInfo bytes directly:
            let data = other_account.try_borrow_data()?;
            let state = OtherStruct::try_deserialize(&mut &data[..])?;
```

**Why teams use raw invoke over Anchor:**
- Target isn't Anchor (System Program, Token Program called manually)
- No IDL available (closed-source program)
- Lower compute cost — Anchor adds ~100-200 CU overhead per instruction
- Auditor context — production code often uses raw invoke

### Sub-topic: Blind Signing / Transaction Injection — full detail
**Source:** Rektoff Security Roadmap §8.1
```
Transaction
  ├── Instruction 1: "swap 10 USDC" ← user sees this
  ├── Instruction 2: "approve token account" ← user misses this
  └── Instruction 3: "drain entire wallet" ← quietly executes
```
User signs once → all three execute atomically. This is how wallet drainers work.

**Mitigations:**
| Layer | Protection |
|---|---|
| Wallet (Phantom, Backpack) | Simulate tx and show predicted state changes before signing |
| Program (auditor check) | Use `Instructions` sysvar to inspect other instructions in same tx |
| User | Never sign without reading the simulation output |

**Program-level defense:**
```rust
// Assert: "I must be the only instruction in this tx"
let ix_sysvar = &ctx.accounts.instructions;
let current_idx = load_current_index_checked(ix_sysvar)?;
require!(current_idx == 0, ErrorCode::OnlyStandaloneAllowed);
```

### Concept 14: Anchor Constraints vs Instruction Code — "Constraints do setup, code does logic"
**Source:** rareskills.io/solana-tutorial — Anchor constraints
**This is the single most important Anchor mental model.**

**The insight:**
```
#[account(...constraints...)]   ← Anchor handles ALL the boilerplate setup
pub fn my_instruction(...)      ← you ONLY write your business logic
```

**Concrete example — creating a Mint:**
```rust
// What you write (3 lines of constraints):
#[account(
    init,
    payer = authority,
    mint::decimals = 6,
    mint::authority = config,         // WHO can mint → stored in Mint data
    mint::freeze_authority = config,  // WHO can freeze → stored in Mint data
)]
pub mint: Account<'info, Mint>,

// What Anchor generates for you (~30 lines of raw code):
//   1. CreateAccount CPI to System Program (allocate space, fund with rent)
//   2. InitializeMint CPI to SPL Token Program (set decimals, authorities)
//   3. Ownership verification that account is owned by Token Program
```

**The authority fields live IN the Mint account's data:**
```
Mint Account (owned by SPL Token Program)
├── decimals: 6
├── mint_authority: Some(config_PDA)     ← set by constraint, checked on every mint_to
├── freeze_authority: Some(config_PDA)   ← set by constraint, checked on every freeze
└── supply: 0
```

Your instruction code doesn't decide who's authorized — **the Mint account's stored data does**.
SPL Token Program reads those fields on every CPI call. Your code just presents the right signer.

**What happens if you omit `mint::freeze_authority`:**
```rust
// This Mint can NEVER have accounts frozen — permanent, no upgrade path
#[account(init, mint::authority = config)]  // freeze_authority defaults to None
pub mint: Account<'info, Mint>,
```

**Full pattern — constraints handle setup, instruction handles logic:**
```rust
// CONSTRAINTS (accounts struct) — runs before your instruction code:
//   init         → create + fund the account
//   seeds/bump   → verify PDA address
//   mut          → allow writes
//   mint::*      → initialize Mint with given params
//   associated_token::* → verify/create ATA

// INSTRUCTION CODE — only your business logic:
//   check max supply
//   call mint_to CPI
//   update total_minted counter
```

**Mnemonic:** Constraints = the architect who builds the room. Instruction = you doing your work inside it.
**Status:** ✅ understood

### Concept 15: SPL Token Operations — full surface (Phase 4.3)
**Source:** rareskills.io/solana-tutorial — SPL Token operations

**Who signs what:**
| Operation | Who signs | PDA needed? | Mint supply changes |
|---|---|---|---|
| `mint_to` | mint_authority (PDA) | ✅ new_with_signer | ✅ increases |
| `transfer` | token account owner (wallet) | ❌ new | ❌ |
| `burn` | token account owner (wallet) | ❌ new | ✅ decreases |
| `freeze_account` | freeze_authority (PDA) | ✅ new_with_signer | ❌ |
| `thaw_account` | freeze_authority (PDA) | ✅ new_with_signer | ❌ |
| `close_account` | token account owner (wallet) | ❌ new | ❌ |

**freeze / thaw flow:**
```
freeze_holder called
  └── config PDA signs freeze_account CPI
        └── SPL Token sets frozen=true on target ATA
              └── any transfer/burn on that ATA → rejected by SPL Token

thaw_holder called
  └── config PDA signs thaw_account CPI
        └── SPL Token sets frozen=false
              └── transfers/burns work again
```

**Real-world freeze use cases:**
| Protocol | Why |
|---|---|
| USDC (Circle) | Regulatory compliance — freeze sanctioned addresses |
| Vesting contracts | Freeze tokens until unlock date |
| Game items | Freeze NFT while item is "equipped" |
| Staking | Freeze staked tokens to prevent double-spend |

**Auditor checklist for freeze:**
- Does the protocol need freeze? → was `freeze_authority` set at Mint creation?
- If `freeze_authority = None` → cannot be added later. If needed, it's a design flaw.
- Who holds the freeze_authority? PDA (programmatic) or wallet (centralized)?

**Hands-on:** `hands-on/token-minter-anchor/` — 8 tests covering full lifecycle:
`initialize → mint → transfer ✅ → freeze → transfer ❌ → thaw → transfer ✅ → max supply ❌`

**Status:** ✅ understood

### Concept 16: NFTs on Solana
**Source:** rareskills.io/solana-tutorial — NFT section
**Summary:** NFTs on Solana use the same SPL Token infrastructure as fungible tokens. Three constraints make a token non-fungible: decimals=0, supply=1, mint_authority=None after minting. No separate contract standard like ERC721 — just SPL Token with specific parameters.
**Bridged from:** ERC721 (EVM) — completely different contract from ERC20. On Solana, same SPL Token, different config.

**Three rules mapped to code:**
```
decimals = 0        → mint::decimals = 0       (constraint at Mint creation)
supply = 1          → mint_to(..., 1)           (instruction)
mint_authority=None → set_authority(..., None)  (instruction, right after mint)
```

**Lifecycle:**
```
1. Create Mint (decimals=0, mint_authority=owner)
2. Create Token Account (ATA) for the owner
3. mint_to(amount=1) → supply is now 1
4. set_authority(MintTokens → None) ← THIS makes it an NFT permanently
```

**Each NFT = its own Mint account:**
```
EVM (ERC721):  one contract → tokenId 1, 2, 3... all tracked internally
Solana:        NFT #1 → Mint A
               NFT #2 → Mint B
               NFT #3 → Mint C   ← each fully independent
```

**No batch mint like ERC721:**
- Each NFT needs: CreateAccount + InitializeMint + mint_to + set_authority = 4 operations
- Solana tx size limit (~1232 bytes) → 1-2 NFTs per tx max
- Real collections use Metaplex Candy Machine — metadata uploaded upfront, users mint one at a time
- Cost to user: ~0.01-0.02 SOL rent per NFT (Mint account + Token Account)

**Collection membership — how to prove an NFT belongs to a collection:**
```
Metaplex Metadata Account (per NFT):
├── name: "CryptoPunk #1"
└── collection: {
      key: <Collection Mint address>,
      verified: true   ← ONLY the collection authority can set this
    }
```
- `verified = false` → anyone can fake collection membership by pointing to any Mint
- `verified = true` → only the holder of the collection's update authority can set
- **Auditor rule:** marketplaces MUST check `verified = true`, not just `collection.key`

**Status:** ✅ understood

### Concept 17: Token-2022 (Token Extensions Program)
**Source:** rareskills.io/solana-tutorial — Token-2022 section
**Summary:** Token-2022 is an upgraded token program deployed alongside the original SPL Token. Same interface but with an opt-in extension system — each Mint can enable extra features at creation time. Cannot be added after Mint creation.

**Why it exists:** Original SPL Token is frozen — adding features would break existing tokens. Token-2022 is a fresh deployment with an extension system.

**Key extensions:**
| Extension | What it does | Use case |
|---|---|---|
| `TransferFee` | Protocol takes % on every transfer | DeFi revenue, royalties |
| `TransferHook` | Calls your program on every transfer | Allowlists, game logic, compliance |
| `NonTransferable` | Token cannot be transferred | Soulbound tokens, credentials |
| `PermanentDelegate` | An address can always transfer/burn | Protocol control, recovery |
| `ConfidentialTransfer` | Encrypted amounts (ZK proofs) | Privacy |
| `MintCloseAuthority` | Allows closing the Mint account | Cleanup |
| `MetadataPointer` | Points to on-chain metadata | NFT metadata on-chain |

**TransferHook — the most important for auditors:**
```
SPL Token (original):
  transfer() → tokens move, nothing else

Token-2022 with TransferHook:
  transfer() → tokens move → YOUR PROGRAM IS CALLED (CPI)
               ↑ arbitrary code runs on every transfer
               ↑ runs with the user's token context
```

**TransferHook attack surface (auditor checklist):**
1. **Malicious/upgradeable hook logic** — if hook program has upgrade authority, protocol can silently change what executes on every transfer. Audit check: is upgrade authority = None?
2. **Reentrancy** — hook fires mid-transfer. Cross-program reentrancy through CPI chain is possible even though direct reentrancy is blocked by runtime.
3. **DoS via expensive hook** — attacker triggers transfers repeatedly, hook does expensive computation (e.g. checks large on-chain list), burns compute, blocks legitimate transfers.

**Fake collection attack (NFT auditor note):**
```
Fake NFT:  collection.key = CryptoPunks Mint, verified = false  ← attacker sets this
Real NFT:  collection.key = CryptoPunks Mint, verified = true   ← only collection authority
```
Marketplaces checking only `collection.key` without `verified=true` accept fakes.

**Status:** ✅ understood

## Key Mnemonics

- **invoke vs invoke_signed**: wallet already signed = invoke. PDA needs to sign = invoke_signed (pass seeds).
- **Signer vs Identity**: Signer = "did someone sign?" Identity = "was it the RIGHT someone?"
- **Mint vs Token Account**: Mint = the token's birth certificate (global). Token Account = the wallet's piggy bank for that token (per holder).
- **ATA**: deterministic Token Account address — always findable from (owner, mint).
- **Constraints vs code**: Constraints = architect builds the room. Instruction = you do your work inside it.
- **freeze_authority**: set it at Mint creation or lose it forever. Like a master key — if you don't cut it at the beginning, you can never add a lock later.
