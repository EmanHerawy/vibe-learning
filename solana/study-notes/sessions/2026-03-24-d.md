## Theory Notes

### Concept 1: Why Anchor exists
- Status: ✅ understood
- Anchor automates the boilerplate of raw Solana programs: account deserialization, validation, serialization. Raw Solana = TCP socket server. Anchor = Express.js.

### Concept 2: Project structure
- Status: ✅ understood
```
my-program/
├── Anchor.toml          ← project config (program IDs, network, wallet)
├── programs/<name>/src/lib.rs  ← YOUR PROGRAM CODE
├── tests/<name>.ts      ← TypeScript tests
└── target/deploy/<name>-keypair.json  ← program ID keypair (NEVER commit)
```

### Concept 3: declare_id!
- Status: ✅ understood
- Bakes the program's address (public key of the program keypair) into the binary
- At deploy, Anchor verifies: declared ID == keypair used to deploy. Mismatch → deploy fails
- Program keypair vs wallet keypair:
  - Wallet keypair: private key signs every transaction
  - Program keypair: used once at deploy to claim the address, then again for upgrades
  - Everything in Solana is an account — programs too. Program account has executable=true
  - Program keypair lives at: target/deploy/<name>-keypair.json — NEVER commit this

### Concept 4: #[program] macro
- Status: ✅ understood
- Marks the module containing instruction handlers
- Each `pub fn` inside = one callable instruction
- Function name = instruction name clients call (can be anything descriptive)
- `msg!(...)` = on-chain console.log, writes to transaction logs
- Return `Ok(())` for success, `Err(...)` to fail and revert

### Concept 5: initialize is NOT a constructor
- Status: ✅ understood
- CRITICAL: `initialize` does NOT run automatically at deploy — it's just a naming convention
- It can be called by anyone, any number of times, unless you add guards
- Compare to Solidity: `constructor()` runs once automatically at deploy
- In Anchor: you are responsible for constructor-like behavior (check it hasn't run, restrict caller)
- You can name it anything: `setup`, `create_game`, `mint_token` — name for what it does

### Concept 6: #[derive(Accounts)]
- Status: ✅ understood
- One struct per instruction — each instruction declares only the accounts it needs (least privilege)
- Anchor validates ALL constraints in the struct BEFORE your instruction code runs
- Multiple structs are normal — one per instruction:
```rust
#[derive(Accounts)]
pub struct Initialize<'info> { ... }    // for initialize instruction

#[derive(Accounts)]
pub struct UpdateScore<'info> { ... }   // for update_score instruction

#[derive(Accounts)]
pub struct CloseGame<'info> { ... }     // for close_game instruction
```

### Concept 7: The Context pattern
- Status: ✅ understood
- `ctx: Context<T>` is the first argument of every instruction
- `ctx.accounts` gives typed access to all accounts declared in the struct
- No manual deserialization, no index lookups
- Example: `ctx.accounts.counter.count += 1` — fully typed, no raw bytes

### Concept 8: #[account] macro & discriminator
- Status: ✅ understood
- `#[account]` on a struct = "this struct is on-chain account data"
- Anchor auto-handles serialization/deserialization
- Generates 8-byte discriminator (hash of struct name) prepended to all account data
- Rule: if it lives in an account's data field on-chain → `#[account]`. If it's just Rust logic → plain struct
- ALL fields must be `pub` — instruction handlers in other modules can't access private fields

### Concept 9: Anchor account constraints
- Status: ✅ understood

| Constraint | Meaning |
|---|---|
| `init` | Create this account (calls System Program to allocate). Implies mutability. |
| `payer = x` | Account `x` pays the rent for the new account |
| `space = N` | Allocate exactly N bytes. Formula: `8 (discriminator) + sum of field sizes` |
| `mut` | This existing account's data/lamports will be modified |
| `seeds = [...]` | Derive this account as a PDA from these seeds |
| `bump` | Use the canonical bump for PDA derivation |
| `close = x` | Close this account and return lamports to `x` |

Space formula:
```
space = 8 + field_1_bytes + field_2_bytes + ...
u64   = 8 bytes
u32   = 4 bytes
bool  = 1 byte
Pubkey = 32 bytes
String = 4 + max_length bytes
```

### Concept 10: 'info lifetime
- Status: ✅ understood
- `'info` is a Rust lifetime tied to the raw account data passed in by the Solana runtime
- All account references in the struct share the same `'info` lifetime
- You never manage it manually — just copy the pattern everywhere Anchor asks for it
- It satisfies Rust's borrow checker: all account references must outlive the instruction

### Concept 11: system_program in accounts struct
- Status: ✅ understood
- Required in any instruction that CREATES a new account (`init` constraint)
- Anchor calls System Program under the hood to allocate the account
- Not needed if the instruction only reads/modifies existing accounts

### Concept 12: TryingToInitPayerAsProgramAccount error
- Status: ✅ understood
- Happens when the `counter` (new account) and `user` (payer) resolve to the same address
- Fix: generate a new Keypair for the counter account in the test
- New accounts being `init`-ed must be fresh keypairs, distinct from the payer
- The new account must also be passed as a `.signers([counter])` because it's being created

### Concept 13: Account types in Anchor
- Status: ✅ understood

| Type | Usage |
|---|---|
| `Account<'info, T>` | A program-owned account holding data of type T. Anchor deserializes it for you. |
| `Signer<'info>` | Must have signed the transaction. Used for authority/payer accounts. |
| `Program<'info, T>` | A program account (e.g. System Program). Used when CPI-ing into other programs. |
| `UncheckedAccount<'info>` | Raw account — no Anchor validation. Use only with manual checks + `/// CHECK:` comment. |

### Concept 14: TypeScript test client pattern
- Status: ✅ understood
```typescript
// Connect to local validator
anchor.setProvider(anchor.AnchorProvider.env());

// Load program with auto-generated types
const program = anchor.workspace.helloAnchor as Program<HelloAnchor>;

// Generate new keypair for a new account
const counter = anchor.web3.Keypair.generate();

// Call an instruction
const tx = await program.methods
  .initialize()
  .accounts({ counter: counter.publicKey })
  .signers([counter])   // sign with new account keypair
  .rpc();               // send + confirm

// Fetch account data
const account = await program.account.counter.fetch(counter.publicKey);
console.log(account.count.toString());
```

### Summary: When to use what

| Situation | What to add |
|---|---|
| Creating a new account | `init`, `payer = x`, `space = 8 + N`, include `system_program` |
| Modifying existing account data | `mut` constraint |
| Closing an account | `close = recipient` constraint |
| Restricting who can call | `Signer<'info>` + check `ctx.accounts.authority.key() == expected` |
| Storing data per user | PDA with `seeds = [b"prefix", user.key().as_ref()]` |

### Concept 15: Custom error codes with #[error_code]
- Status: ✅ understood
- `assert!()` panics the program — works but gives the client a generic error with no context
- Anchor's proper pattern: define a custom error enum with `#[error_code]`
- Place it OUTSIDE the `#[program]` module
- Use `require!` instead of `assert!` inside instructions

```rust
#[error_code]
pub enum CounterError {
    #[msg("Counter cannot go below zero")]
    Underflow,
}

// inside instruction:
require!(ctx.accounts.counter.count > 0, CounterError::Underflow);
// require!(condition, error) = if !condition { return Err(error.into()) }
```

- Error codes start at 6000, each variant gets a unique number
- Client sees both the code name ("Underflow") and the message string
- `require!(x, E::Variant)` is just shorthand for `if !x { return Err(E::Variant.into()); }`

### Concept 16: Testing expected failures in Anchor
- Status: ✅ understood
- Pattern: wrap `.rpc()` in try/catch, assert the error code, re-throw anything unexpected
- The line after `.rpc()` inside try must deliberately throw — otherwise a successful tx silently passes

```typescript
try {
  await program.methods.decrement().accounts({ ... }).rpc();
  throw new Error("Expected tx to fail but it succeeded"); // sentinel
} catch (err: any) {
  if (err.error?.errorCode?.code === "Underflow") return; // test passes
  throw err; // unexpected error — re-throw
}
```

- `err.error?.errorCode?.code` = string name of the `#[error_code]` variant
- If `.rpc()` throws Underflow → jumps to catch, matches, test passes
- If `.rpc()` succeeds → hits the sentinel throw → catch sees a non-Underflow error → re-throws → test fails with useful message

### Concept 17: PDA-based accounts (per-user state)
- Status: ✅ understood
- Instead of a random keypair per account, derive the address deterministically from seeds
- Pattern: `seeds = [b"prefix", user.key().as_ref()]` → one account per user, address is always findable

**Program side — init:**
```rust
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(init, payer = user, space = 8 + 8 + 1, seeds = [b"counter", user.key().as_ref()], bump)]
    pub counter: Account<'info, Counter>,
    #[account(mut)]
    pub user: Signer<'info>,
    pub system_program: Program<'info, System>,
}
```

**Program side — lookup (increment/decrement):**
```rust
#[derive(Accounts)]
pub struct Incrementer<'info> {
    #[account(mut, seeds = [b"counter", user.key().as_ref()], bump = counter.bump)]
    pub counter: Account<'info, Counter>,
    pub user: Signer<'info>,  // needed for seed + implicit access control
}
```

- `bump` on init → Anchor finds the canonical bump automatically
- `bump = counter.bump` on lookup → reads stored bump from account (cheap, no search)
- Must store bump in account data: add `pub bump: u8` to the struct, set with `ctx.accounts.counter.bump = ctx.bumps.counter`
- `user: Signer` in lookup struct = only the user whose key is in the seed can call the instruction (free access control)
- No `.signers([counter])` needed — PDAs have no private key

**Test side:**
```typescript
const wallet = anchor.AnchorProvider.env().wallet; // use real provider wallet, NOT Keypair.generate()

const [counterPda] = anchor.web3.PublicKey.findProgramAddressSync(
  [Buffer.from("counter"), wallet.publicKey.toBuffer()],
  program.programId
);

// initialize — no signers needed
await program.methods.initialize()
  .accounts({ counter: counterPda, user: wallet.publicKey })
  .rpc();

// increment — pass both accounts explicitly
await program.methods.increment()
  .accounts({ counter: counterPda, user: wallet.publicKey })
  .rpc();
```

### Concept 18: Space must include ALL fields — the 8+8+1 bug
- Status: ✅ understood
- CRITICAL: every time you add a field to your `#[account]` struct, update `space` in the `init` constraint
- Adding `bump: u8` to Counter without updating space → `AccountDidNotDeserialize` at runtime
- Error `AccountDidNotDeserialize (3003)` = account was found but its byte layout doesn't match the struct — almost always a space mismatch

```
Counter { count: u64, bump: u8 }
space = 8 (discriminator) + 8 (u64) + 1 (u8) = 17
```

- `AccountNotInitialized (3012)` = account address doesn't exist on-chain — usually wrong PDA derivation or wrong wallet used for seeds
- Common mistake: `Keypair.generate()` for the wallet instead of `AnchorProvider.env().wallet` → seeds derive a different address than what the program created

### Concept 19: Account resizing — you cannot grow already-deployed accounts
- Status: ✅ understood
- When you allocate `space = N` at init, that account is locked to N bytes on-chain forever
- Adding a field to your `#[account]` struct does NOT resize existing accounts — they were already allocated
- Result: old accounts deserialize fine until someone tries to write the new field → layout mismatch
- **This is a real production problem** — changing struct layout post-deploy is a breaking change

**Your options when you need to add a field:**

| Option | How | Cost |
|---|---|---|
| Migration instruction | New `migrate` instruction: close old account, re-init with new struct + larger space | User must call migrate; old data needs copying |
| Versioned accounts | Add a `version: u8` field from day one; write upgrade logic per version | More upfront design work |
| New program version | Deploy a new program with the new struct, migrate state over | Full redeploy |

- **Best practice: always over-allocate space slightly at init** — add a few extra bytes for future fields
- Solana does have `realloc` (since 1.10) via `#[account(realloc = N, realloc::payer = x, realloc::zero = false)]` — but this only works if you planned for it and the account has a payer who can cover the extra rent
- Compare to Ethereum: storage slots are dynamic, you just add a variable. In Solana, you pre-commit to a size.

### Concept 20: Anchor tx processing order — Signer fires before PDA lookup
- Status: 🔄 needs reinforcement
- Order of checks Anchor runs on every transaction:
  1. Signature verification — `Signer<'info>` checks first
  2. Account constraint checks — seeds, bump, owner, mut, etc.
  3. Your instruction code runs
- CRITICAL: if someone passes a wrong `user` address, the tx fails at step 1 (missing signature), NOT at step 2 (wrong PDA). The Signer constraint is the first line of defense.
- Implication for auditing: always check whether `Signer` is present on authority accounts. Without it, anyone can pass any address as that account.

### Concept 20: Security patterns learned from exam
- Status: ✅ understood

**Vulnerability 1: u64 underflow**
- `vault.amount -= user.amount` with no guard → if vault < user, wraps to u64::MAX in release mode
- Fix: `require!(vault.amount >= user.amount, ErrorVariant::InsufficientFunds)`

**Vulnerability 2: Missing per-user vault (shared vault attack)**
- A single shared vault with no per-user mapping → any user can drain all funds
- Fix: PDA per user with `seeds = [b"vault", user.key().as_ref()]` so each user can only access their own vault

**Vulnerability 3: Missing Signer on authority account**
- If the account being withdrawn from has no `Signer` constraint, attacker can pass ANY UserAccount as the caller
- Fix: always add `Signer<'info>` to any account that represents the transaction authority

**Auditor checklist for any instruction that moves funds:**
| Check | What to look for |
|---|---|
| Underflow guard | Is there a `require!` before any subtraction on u64? |
| Per-user isolation | Does each user have their own PDA, or is there a shared pool? |
| Signer constraint | Is the authority account declared as `Signer<'info>`? |
| Owner check | Does Anchor verify the account is owned by this program (`Account<'info, T>`)? |
