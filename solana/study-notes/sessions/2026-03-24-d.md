## Theory Notes

### Concept 1: Why Anchor exists
- Status: ✅ understood
- Anchor automates the boilerplate of raw Solana programs: account deserialization, validation, serialization. Raw Solana = TCP socket server. Anchor = Express.js.

### Concept 2: Project structure
- Status: ✅ understood
```
my-program/
├── Anchor.toml          ← project config (program IDs, network, wallet)
├── programs/<name>/src/lib.rs  ← YOUR PROGRAM CODE
├── tests/<name>.ts      ← TypeScript tests
└── target/deploy/<name>-keypair.json  ← program ID keypair (NEVER commit)
```

### Concept 3: declare_id!
- Status: ✅ understood
- Bakes the program's address (public key of the program keypair) into the binary
- At deploy, Anchor verifies: declared ID == keypair used to deploy. Mismatch → deploy fails
- Program keypair vs wallet keypair:
  - Wallet keypair: private key signs every transaction
  - Program keypair: used once at deploy to claim the address, then again for upgrades
  - Everything in Solana is an account — programs too. Program account has executable=true
  - Program keypair lives at: target/deploy/<name>-keypair.json — NEVER commit this

### Concept 4: #[program] macro
- Status: ✅ understood
- Marks the module containing instruction handlers
- Each `pub fn` inside = one callable instruction
- Function name = instruction name clients call (can be anything descriptive)
- `msg!(...)` = on-chain console.log, writes to transaction logs
- Return `Ok(())` for success, `Err(...)` to fail and revert

### Concept 5: initialize is NOT a constructor
- Status: ✅ understood
- CRITICAL: `initialize` does NOT run automatically at deploy — it's just a naming convention
- It can be called by anyone, any number of times, unless you add guards
- Compare to Solidity: `constructor()` runs once automatically at deploy
- In Anchor: you are responsible for constructor-like behavior (check it hasn't run, restrict caller)
- You can name it anything: `setup`, `create_game`, `mint_token` — name for what it does

### Concept 6: #[derive(Accounts)]
- Status: ✅ understood
- One struct per instruction — each instruction declares only the accounts it needs (least privilege)
- Anchor validates ALL constraints in the struct BEFORE your instruction code runs
- Multiple structs are normal — one per instruction:
```rust
#[derive(Accounts)]
pub struct Initialize<'info> { ... }    // for initialize instruction

#[derive(Accounts)]
pub struct UpdateScore<'info> { ... }   // for update_score instruction

#[derive(Accounts)]
pub struct CloseGame<'info> { ... }     // for close_game instruction
```

### Concept 7: The Context pattern
- Status: ✅ understood
- `ctx: Context<T>` is the first argument of every instruction
- `ctx.accounts` gives typed access to all accounts declared in the struct
- No manual deserialization, no index lookups
- Example: `ctx.accounts.counter.count += 1` — fully typed, no raw bytes

### Concept 8: #[account] macro & discriminator
- Status: ✅ understood
- `#[account]` on a struct = "this struct is on-chain account data"
- Anchor auto-handles serialization/deserialization
- Generates 8-byte discriminator (hash of struct name) prepended to all account data
- Rule: if it lives in an account's data field on-chain → `#[account]`. If it's just Rust logic → plain struct
- ALL fields must be `pub` — instruction handlers in other modules can't access private fields

### Concept 9: Anchor account constraints
- Status: ✅ understood

| Constraint | Meaning |
|---|---|
| `init` | Create this account (calls System Program to allocate). Implies mutability. |
| `payer = x` | Account `x` pays the rent for the new account |
| `space = N` | Allocate exactly N bytes. Formula: `8 (discriminator) + sum of field sizes` |
| `mut` | This existing account's data/lamports will be modified |
| `seeds = [...]` | Derive this account as a PDA from these seeds |
| `bump` | Use the canonical bump for PDA derivation |
| `close = x` | Close this account and return lamports to `x` |

Space formula:
```
space = 8 + field_1_bytes + field_2_bytes + ...
u64   = 8 bytes
u32   = 4 bytes
bool  = 1 byte
Pubkey = 32 bytes
String = 4 + max_length bytes
```

### Concept 10: 'info lifetime
- Status: ✅ understood
- `'info` is a Rust lifetime tied to the raw account data passed in by the Solana runtime
- All account references in the struct share the same `'info` lifetime
- You never manage it manually — just copy the pattern everywhere Anchor asks for it
- It satisfies Rust's borrow checker: all account references must outlive the instruction

### Concept 11: system_program in accounts struct
- Status: ✅ understood
- Required in any instruction that CREATES a new account (`init` constraint)
- Anchor calls System Program under the hood to allocate the account
- Not needed if the instruction only reads/modifies existing accounts

### Concept 12: TryingToInitPayerAsProgramAccount error
- Status: ✅ understood
- Happens when the `counter` (new account) and `user` (payer) resolve to the same address
- Fix: generate a new Keypair for the counter account in the test
- New accounts being `init`-ed must be fresh keypairs, distinct from the payer
- The new account must also be passed as a `.signers([counter])` because it's being created

### Concept 13: Account types in Anchor
- Status: ✅ understood

| Type | Usage |
|---|---|
| `Account<'info, T>` | A program-owned account holding data of type T. Anchor deserializes it for you. |
| `Signer<'info>` | Must have signed the transaction. Used for authority/payer accounts. |
| `Program<'info, T>` | A program account (e.g. System Program). Used when CPI-ing into other programs. |
| `UncheckedAccount<'info>` | Raw account — no Anchor validation. Use only with manual checks + `/// CHECK:` comment. |

### Concept 14: TypeScript test client pattern
- Status: ✅ understood
```typescript
// Connect to local validator
anchor.setProvider(anchor.AnchorProvider.env());

// Load program with auto-generated types
const program = anchor.workspace.helloAnchor as Program<HelloAnchor>;

// Generate new keypair for a new account
const counter = anchor.web3.Keypair.generate();

// Call an instruction
const tx = await program.methods
  .initialize()
  .accounts({ counter: counter.publicKey })
  .signers([counter])   // sign with new account keypair
  .rpc();               // send + confirm

// Fetch account data
const account = await program.account.counter.fetch(counter.publicKey);
console.log(account.count.toString());
```

### Summary: When to use what

| Situation | What to add |
|---|---|
| Creating a new account | `init`, `payer = x`, `space = 8 + N`, include `system_program` |
| Modifying existing account data | `mut` constraint |
| Closing an account | `close = recipient` constraint |
| Restricting who can call | `Signer<'info>` + check `ctx.accounts.authority.key() == expected` |
| Storing data per user | PDA with `seeds = [b"prefix", user.key().as_ref()]` |
