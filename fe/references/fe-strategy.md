## Technical Differentiation (vs Solidity/Vyper)

Solidity and Vyper were designed during initial Ethereum stages, primarily with tokens and early DeFi in mind. Fe learns from past issues and solves for a wider range of use cases.

1. **Custom ABI Encoding** — design custom ABI formats for specific needs while maintaining compatibility with existing Solidity contracts & tooling
2. **Flexible & Clean Abstractions** — high and low level APIs depending on developer need, clean and modular code
3. **Rust-like Type System** — simpler, more readable, easier to organize; enables LSP and better compiler error messages

## Priority Niches (ordered by relevance)

### 1. Cryptography (highest priority)

ZK verifiers, merkle trees, and specific papers implemented in EVM — use cases far from what Solidity was originally designed for.

- **Poseidon hash** — current preferred zk-friendly hash on Ethereum; existing impl is JS-generated bytecode (not readable/modular). Opportunity: make it readable, modular, verifiable, Foundry-compatible
- **zkDSL Verifiers** (Circom/Noir) — template-generated Solidity with minimal changes. Opportunity: make readable + modular, omit default selector (no dispatch table needed) and default parameter passing
- **zkVM Verifiers** (OpenVM, SP1) — pre-baked or predeployed. Same opportunity as zkDSLs
- **Merkle trees** — used in whitelists and privacy protocols. Good use case to explore Fe iteration features

### 2. Contract Upgradability

Fe custom ABI formats + orderly low-level code are very useful here.

- **Diamond Pattern (ERC-2535)** — wide range of implementations (generic vs highly optimized). Opportunity: specific implementations for non-upgradable contracts using hardcoded arrays, macros, binary searches in a more readable way

### 3. DeFi Math

Most intricate DeFi calculations require low-level implementations. Uniswap, Aave, Morpho all use assembly in math libraries — a "bad symptom" Fe can fix.

- **Uniswap v4** — active experimentation space, in need of better interfaces for hooks

### 4. On-chain Gaming

Similar to cryptography — game dev is far from Solidity's intended use. Space is not very active currently.

- **MUD (Lattice)** — EVM game engine; JS codegen → Solidity contracts. Opportunity: make generated code readable, use better selectors for DB queries. Lattice is now in maintenance mode but games still being developed

### 5. MEV & Arbitrage Bots

MEV bots (e.g. jaredfromsubway.eth) write contracts directly in bytecode. Fe's clean low-level API with zero overhead could attract this segment.

## Ahmed's Immediate Recommendation

1. **Poseidon hash** — low-hanging fruit
2. **At least one ZK verifier** — standalone library contract, enables many downstream use cases
3. **Exploratory:** on-chain games + Uniswap v4 hooks

> "Ethereum moves fast — revise this list very often."

## Engagement & Success Metrics

- 1 technical blog post every 3 weeks
- Direct 1-on-1 outreach to developers/teams that can integrate Fe or build partnerships
- Maintain open communication on Discord and Zulip

**Success metrics:**
- Number of KOLs engaging with Fe
- Number of projects integrating Fe into their stack

## Long-term Vision

- ~99% of Ethereum TVL is secured by a single compiler (Solidity) — systemic risk
- Fe can diversify compiler risk at the protocol level (reducing social hardfork risk) and at institutional/personal portfolio level
- Path: earn trust via niches → libraries and tools mature (Hardhat, Foundry, audited contracts) → broader developer community adoption
- North star: Fe as Ethereum's compiler diversity solution
