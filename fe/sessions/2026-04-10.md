## Theory Notes

### Mental Model
- Status: ‚úÖ understood
- Fe = Solidity's EVM target + Rust's "declare your intent upfront" discipline
- Core design decision: every function must declare what it's allowed to touch via a `uses` clause
- Everything else in the syntax serves that declaration

---

### Construct 1 ‚Äî Import System
- Status: ‚úÖ understood
- Pull in only what you need from `std` ‚Äî no implicit globals
```fe
use std::abi::sol                  // sol() selector function ‚Äî most contracts need only this
use std::evm::{Evm, Call}          // Evm and Call effect types
use std::evm::effects::assert      // assert (also available as prelude built-in)
use std::evm::crypto::keccak256    // crypto ops
```
- **Auditor lens:** imports are a capability signal. A contract importing `Call` can make external calls. A contract with only `sol` cannot.
- **Note:** `assert` works without importing it (prelude built-in). Counter imports it explicitly ‚Äî redundant but not wrong.

---

### Construct 2 ‚Äî Contract Declaration + Contract-Level `uses`
- Status: ‚úÖ understood
- The contract header is a **capability ceiling** ‚Äî declares what the contract is allowed to access at all
- Individual handlers can use any SUBSET, never more than the ceiling

```fe
pub contract Counter {                                        // no uses = storage only
pub contract Token uses (ctx: Ctx) {                         // can read blockchain context
pub contract Vault uses (ctx: Ctx, call: mut Call) {         // can make external calls
pub contract SimpleAmm uses (ctx: Ctx, log: mut Log) {       // can emit events
```

- Syntax at contract level (and free functions): full `name: Type` form ‚Üí `ctx: Ctx`, `call: mut Call`, `log: mut Log`, `evm: mut Evm`
- **Ceiling rule:** if `call: mut Call` is NOT in the contract header, NO handler inside it can ever make an external call ‚Äî enforced at compile time
- **Analogy:** contract header = security clearance for the whole building. Handlers can have lower clearance, never higher.
- **Solidity comparison:** Solidity has no contract-level permission manifest ‚Äî any function can call anything. Fe makes capabilities explicit and bounded.

### Construct 3 ‚Äî Storage Struct + StorageMap
- Status: ‚úÖ understood
- All storage in ONE named struct ‚Äî one place, complete visibility
- `store`/`mut store` are NOT listed in the contract-level `uses` clause ‚Äî storage is always available as a field

```fe
struct CounterStore {
    value: u256,
}

struct TokenStore {
    total_supply: u256,
    balances: StorageMap<Address, u256>,
    allowances: StorageMap<(Address, Address), u256>,   // tuple key
}

pub contract Counter {
    mut store: CounterStore    // only storage this contract has
}
```

StorageMap API ‚Äî always named arguments:
```fe
store.balances.get(key: account)
store.balances.set(key: account, value: n)
store.allowances.get(key: (owner, spender))        // tuple key
store.allowances.set(key: (from, spender), value: allowed - amount)
```

- `uses (store)` ‚Üí read-only, compiler enforces: cannot write
- `uses (mut store)` ‚Üí read + write
- Fe does NOT enforce CEI order ‚Äî reentrancy still possible if you write bad order
- Upgrade reality: same sequential slot assignment as Solidity (append-only discipline still required)

### Construct 4 ‚Äî `msg` Block + Selectors
- Status: ‚úÖ understood

**The problem Fe solves:** In Solidity, declaration + implementation are fused in one function. Fe splits them:
```
PIECE 1: msg block  = WHAT calls exist (name, params, return type, selector)
PIECE 2: recv block = WHAT each call does (implementation)
```

```fe
msg CounterMsg {
    #[selector = sol("increment()")]
    Increment,                                           // no params, no return

    #[selector = sol("get()")]
    Get -> u256,                                         // no params, returns u256
}

msg Erc20 {
    #[selector = sol("transfer(address,uint256)")]
    Transfer { to: Address, amount: u256 } -> bool,     // params + return

    #[selector = sol("totalSupply()")]
    TotalSupply -> u256,
}
```

- `sol("transfer(address,uint256)")` ‚Üí computes keccak256 of the string, takes first 4 bytes ‚Üí the dispatch selector (e.g., `0xa9059cbb`)
- Same algorithm as Solidity ‚Üí Fe contracts are ABI-compatible with Solidity callers
- `sol()` is a **const function** (evaluated at compile time), NOT a macro
- **`#[selector]` status:** required in all known examples. Whether omitting causes a compiler error or auto-generates a selector is **unconfirmed from docs** ‚Äî marked as open question
- **Auditor lens:** `msg` block = complete attack surface in one place. No hidden public functions.

**The payoff of splitting:** `msg` type is reusable. Other contracts import and use it for cross-contract calls ‚Äî no separate interface file, no ABI drift:
```fe
call.call(addr: token, gas: 100000, value: 0,
    message: Erc20::Transfer { to: ctx.address(), amount })
```

### Construct 5 ‚Äî `recv` Handlers + Per-Variant `uses`
- Status: ‚úÖ understood

```fe
recv CounterMsg {
    Increment uses (mut store) {
        store.value = store.value + 1
    }

    Get -> u256 uses (store) {
        store.value           // implicit return ‚Äî last expression, no return keyword needed
    }
}
```

Full anatomy of a recv arm:
```
Transfer { to, amount } -> bool   ‚Üê destructure the variant + return type
uses (mut store, ctx)             ‚Üê declare capabilities (short form ‚Äî no name: Type)
{
    let from = ctx.caller()       ‚Üê implementation body
    ...
    true
}
```

- **Short form inside handlers:** `ctx`, `mut store`, `mut call`, `mut log` ‚Äî no type annotation
- **Declaration form** (contract header, free functions): `ctx: Ctx`, `call: mut Call` ‚Äî full `name: Type`
- `init()` is the constructor, same pattern:

```fe
init(beneficiary: Address, arbiter: Address) uses (mut store, ctx) {
    store.depositor = ctx.caller()
    ...
}
```

**Multiple `recv` blocks:** one contract can implement multiple msg types:
```fe
recv Erc20    { ... }    // handles Erc20 calls
recv Mintable { ... }    // handles Mintable calls ‚Äî same contract, two interfaces
```

### Construct 6 ‚Äî Effects System (all 6)
- Status: ‚úÖ understood

| Effect | Short form (handlers) | Declaration form (contract/free fn) | What it grants |
|---|---|---|---|
| storage read | `store` | _(always available)_ | read storage fields |
| storage write | `mut store` | _(always available)_ | read + write storage |
| context | `ctx` | `ctx: Ctx` | caller, value, block_number, address |
| external calls | `mut call` | `call: mut Call` | call.call() and call.static() |
| emit events | `mut log` | `log: mut Log` | log.emit() |
| raw EVM / tests | `mut evm` | `evm: mut Evm` | mstore, calldataload, create2, evm.call |

- **`mut` on Call/Log/Evm:** not about mutating a struct ‚Äî signals the capability object is consumed/advanced. The `mut` is about side effects on the external world.

- **`call.static()` still requires `mut Call`** ‚Äî confirmed in governance example:
  ```fe
  fn token_balance(_ token: Address, _ account: Address) -> u256
      uses (call: mut Call)
  {
      call.static(addr: token, gas: 100000, message: TokenMsg::BalanceOf { account })
  }
  ```
  WHY: even a read-only external call crosses a trust boundary. You hand control to external code ‚Äî it executes, consumes gas, can revert. Fe treats ANY external interaction as a side effect that must be declared, regardless of whether the target's state changes. `mut` = "I am reaching outside this contract's boundary."

- **Ceiling rule applies to `mut` too ‚Äî you cannot escalate:**
  ```
  Contract declares: call: mut Call  ‚Üí handler can use mut call ‚úÖ or omit it ‚úÖ
  Contract declares: call: Call      ‚Üí handler can use call ‚úÖ but NOT mut call ‚ùå (escalation)
  Contract declares nothing          ‚Üí handler cannot use call at all ‚ùå
  ```
  In practice: Fe examples only ever use `call: mut Call` or nothing ‚Äî `call: Call` (non-mut) at contract level is theoretically valid by the ceiling rule but unconfirmed by compiled examples.

- **`evm: mut Evm` has two roles:**
  - In tests: `evm.create2<T>(...)`, `evm.call(...)` ‚Äî deploy and call contracts
  - In production library code: `evm.mstore`, `evm.calldataload` ‚Äî raw EVM memory ops (see merkle example)

- **Ceiling rule summary:** handler can use any subset of what the contract declares ‚Äî never more, never escalate `mut`.
  - `pub contract Vault uses (ctx: Ctx, call: mut Call)` ‚Üí handler `uses (mut store)` ‚úÖ, handler `uses (mut log)` ‚ùå (log not declared)

### Construct 9 ‚Äî Free Functions + `uses`
- Status: ‚úÖ understood

Pure (no effects, no `uses`):
```fe
fn swap_amount(reserve_in: u256, reserve_out: u256, amount_in: u256) -> u256 {
    if amount_in == 0 || reserve_in == 0 || reserve_out == 0 { 0 }
    else { reserve_out * amount_in / (reserve_in + amount_in) }
}
```

With effects (uses declaration form ‚Äî full `name: Type`):
```fe
fn token_balance(_ token: Address, _ account: Address) -> u256
    uses (call: mut Call)
{
    call.static(addr: token, gas: 100000, message: TokenMsg::BalanceOf { account })
}
```

#### `_` prefix ‚Äî named vs positional arguments

Fe requires named (labelled) arguments at every call site by default:
```fe
fn token_balance(token: Address, account: Address) -> u256 ...

// caller MUST write labels:
token_balance(token: store.token, account: ctx.caller())
```

The `_` prefix on a parameter **removes the label requirement** ‚Äî makes it positional:
```fe
fn token_balance(_ token: Address, _ account: Address) -> u256 ...

// caller writes NO labels ‚Äî positional only:
token_balance(store.token, ctx.caller())
```

**When to use `_`:** When the function name already makes the argument meaning obvious and labels would just be noise.
- `token_balance(store.token, ctx.caller())` ‚Äî the function name tells you what the args are. Labels add nothing.
- `store.balances.set(key: from, value: amount)` ‚Äî `set` alone doesn't tell you which arg is the key and which is the value. Labels are essential here.

Rule of thumb: `_` = "the name of this function already explains what this argument is, no label needed."

**Where `_` can and cannot be applied (confirmed 2026-04-11):**

| Location | Can use `_`? | Notes |
|---|---|---|
| Free function parameters | ‚úÖ yes | Primary use case |
| `init()` parameters | ‚úÖ yes | Same syntax as free functions |
| `msg` variant fields | ‚ùå no | Fields use struct syntax, not function call syntax |
| `recv` handler destructuring | ‚ùå no | `Deposit { to }` ‚Äî no label at call site anyway |
| StorageMap `.get()` / `.set()` | ‚ùå no | Labels (`key:`, `value:`) are always required |

The common misconception: `_` looks like it could apply anywhere named args appear. It only applies to **free function and `init` parameters** ‚Äî anywhere you write `fn name(_ param: Type)`.

### Construct 13 ‚Äî `WordRepr` Trait (confirmed from compiler source)

```fe
pub trait WordRepr {
    fn from_word(_: u256) -> Self
    fn to_word(own self) -> u256
}
```

- A type must implement `WordRepr` to be used as a **value** in `StorageMap<K, V>`
- It is a round-trip through a **single EVM word (256 bits)** ‚Äî the type must fit in one `u256`
- Primitive types (`u256`, `bool`, `Address`) implement it by default
- **Custom structs with multiple fields CANNOT meaningfully implement it** ‚Äî a 3-field struct is 768 bits, won't fit in one u256
- **Enums CAN implement it** ‚Äî they map to a single u256 discriminant:

```fe
impl WordRepr for Status {
    fn from_word(_ word: u256) -> Self {
        if word == 0 { Status::Empty }
        else if word == 1 { Status::Funded }
        else { Status::Released }
    }
    fn to_word(own self) -> u256 {
        match self {
            Status::Empty    => 0,
            Status::Funded   => 1,
            Status::Released => 2,
        }
    }
}
```

This makes `StorageMap<Address, Status>` valid.

**Consequence for storage design:**
- `StorageMap<Address, u256>` ‚úÖ ‚Äî u256 implements WordRepr
- `StorageMap<Address, Address>` ‚úÖ ‚Äî Address implements WordRepr
- `StorageMap<Address, Status>` ‚úÖ ‚Äî if you impl WordRepr for Status
- `StorageMap<Address, MyStruct>` ‚ùå ‚Äî struct with multiple fields won't fit in one word

**The correct Fe pattern for "mapping to a struct":** separate maps per field

```fe
struct EscrowStore {
    balances:      StorageMap<Address, u256>,
    release_times: StorageMap<Address, u256>,
}
// instead of: StorageMap<Address, UserEscrow>
```
