## Theory Notes

### revert(e) — error type requirement
**Source:** fe/ingots/std/src/evm/effects.fe — `pub fn revert<T>(_ value: own T) -> ! where T: Encode<Sol> + AbiSize`
**Summary:** The error type passed to revert() must be a struct or enum that Fe can ABI-encode into the revert payload. It must manually implement two traits: `AbiSize` (declares encoded byte size) and `Encode<A>` (writes the bytes). There is no derive macro — you write both impls by hand. The standard pattern is to map enum variants to u256 codes and write one 32-byte word.
**Bridged from:** Solidity custom errors — `revert NotExpired()` auto-encodes via the compiler. In Fe you own the encoding.
**Misconception flagged:** "Any type can be passed to revert()" — False. The compiler enforces `T: Encode<Sol> + AbiSize` at the call site. A plain enum with no impls will not compile.
**Confirmed in practice:** examples/error-handling/src/lib.fe — EscrowError enum with manual AbiSize + Encode<A> impls; `fe test` 5/5 passing.
**Status:** ✅ understood

### assert() vs revert(e) — deciding question
**Source:** fe/ingots/std/src/evm/effects.fe
**Summary:** assert() emits an empty revert (0 bytes) — callers receive no error context. revert(e) ABI-encodes the error value — callers can decode which error occurred. The deciding question: does the caller need to distinguish *why* the call failed? If yes → revert(e) or Result + unwrap_or_revert(). If no (internal invariant that should never fail) → assert().
**Status:** ✅ understood

### Ownership gotcha — is_ok() / is_none() consume the value
**Source:** fe/ingots/core/src/result.fe, fe/ingots/core/src/option.fe
**Summary:** `is_ok()`, `is_err()`, `is_some()`, `is_none()` all take `own self` — they move (consume) the value. Calling is_ok() and then unwrap() on the same binding is a compile error (move conflict). Fix: use `match` to inspect and extract in one step.
**Misconception flagged:** "I can check is_ok() first, then unwrap()" — No. Both take own self. Use match.
**Status:** ✅ understood
