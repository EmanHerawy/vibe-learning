## Theory Notes

### Storage struct field → slot mapping
**Source:** fe-lang.org/structs/definition/
**Summary:** Fe assigns EVM storage slots to struct fields in declaration order (top → bottom = slot 0, 1, 2…). These slot numbers are baked in at deployment. Any contract upgrade must preserve field order exactly.
**Bridged from:** Solidity proxy storage collision bugs — same mechanism.
**Misconception flagged:** "Reordering fields with same types is safe." → False. Slot numbers don't change; old data is reinterpreted under the new field name.
**Checkpoint Q:** After swapping `amount` (slot 2) and `release_block` (slot 3) in an upgrade, what does `record.amount` read?
**Learner answer:** Slot 3 — the old `release_block` value. A block number read as a token amount.
**Status:** ✅ understood

### Method vs associated function
**Source:** fe-lang.org/structs/impl-blocks/, fe-lang.org/structs/associated-functions/
**Summary:** Methods take `self` or `mut self` as first param and are called with dot syntax on an instance. Associated functions take no `self` and are called with `::` on the type (like constructors).
**Bridged from:** Rust impl blocks — identical pattern.
**Misconception flagged:** None — learner identified the distinction immediately from `::` vs `.` call syntax.
**Checkpoint Q:** Is `record.is_funded()` a method or associated function? How can you tell?
**Learner answer:** Method — dot-call on an instance. `::` = associated function.
**Status:** ✅ understood

### `mut self` for writes
**Source:** fe-lang.org/structs/impl-blocks/
**Summary:** A method that reads fields uses `self`. A method that writes to any field must declare `mut self`. The compiler rejects writes on an immutable `self`. This mirrors Fe's `uses (mut store)` pattern on handlers.
**Bridged from:** Fe effects system — `mut` must be declared upfront, compiler enforces it.
**Misconception flagged:** N/A — learner answered correctly and caught a redundancy in the teaching example.
**Checkpoint Q:** `pub fn mark_released(self) { self.release_block = 0 }` — compiler rejects. What do you change?
**Learner answer:** Change `self` to `mut self`.
**Status:** ✅ understood

### Trait definition & implementation
**Source:** fe-lang.org/traits/definition/, fe-lang.org/traits/implementing/, fe-lang.org/traits/generics/
**Summary:** A trait is a named set of method signatures a type must fulfill. `impl TraitName for TypeName` provides the implementation. Generic functions use `<T: TraitName>` bounds to accept any conforming type.
**Bridged from:** Rust traits (identical pattern) + WordRepr from StorageMap constraint + Fe effects system (Ctx, Call, Log are all traits).
**Misconception flagged:** Trait bounds are not just about preventing crashes — they are compile-time capability ceilings. The `uses (ctx: Ctx)` clause on a contract means the compiler statically proves what EVM capabilities the contract can access.
**Checkpoint Q:** Why does StorageMap require WordRepr?
**Learner answer:** StorageMap has a trait bound on V — any type stored as a value must impl WordRepr so the compiler knows it can safely round-trip through a single u256.
**Status:** ✅ understood

### Traits as auditor tool (effects system)
**Source:** fe/ingots/std/src/evm/effects.fe (local compiler source)
**Summary:** Ctx, Call, Log, Create, RawMem, RawOps are all traits implemented by the concrete Evm struct. Declaring `uses (ctx: Ctx)` on a contract is a trait bound — if it's absent, the contract cannot access that capability. The compiler enforces this statically.
**Bridged from:** Fe effects system (already mastered in sessions/2026-04-12.md).
**Misconception flagged:** A fake Ctx in tests is not a vulnerability — it's the same trait mechanism with a test implementation injected instead of the real EVM. By design.
**Checkpoint Q:** If Ctx is NOT in the uses clause, what happens when the contract tries to call ctx.caller()?
**Learner answer:** Type safety enforced at compile time — no runtime crash, won't compile at all.
**Status:** ✅ understood

### Key Mnemonics
- **Slot assignment**: ground floor = slot 0, count top-to-bottom
- **Upgrade safety**: safe deposit box — renaming the label doesn't change what's inside
- **dot vs ::** dot = you already have the thing; :: = asking the type to make something
- **mut self**: same rule as `uses` clause — Fe makes ALL write intent explicit
- **Trait = job description**: no body in trait = no implementation, just the contract
- **impl X for Y**: "implement job description X for worker Y"
- **Generic bound**: T must have a Checkable badge — compiler is the badge inspector
- **uses clause (auditor)**: the contract's locked cage — only listed traits reach through
- **WordRepr**: fits in one word (32 bytes)
- **Check order**: bouncer at the door first, then ticket check inside

## One-Paragraph Summary

Today covered two connected topics: structs with impl blocks (Part 8) and traits with generics (Part 9). Fe's impl blocks follow Rust's pattern exactly — associated functions (no self, called with ::) for construction, methods (self) for reading, mut self for writing. The critical auditor insight is that storage struct fields map to EVM storage slots in declaration order — reordering fields after deployment silently corrupts state, the same as Solidity proxy storage collisions. On traits: Fe's entire effects system (Ctx, Call, Log) is built on traits, meaning the `uses` clause is a trait-bound-based capability ceiling the compiler enforces statically. As an auditor, reading the uses clause alone tells you the complete attack surface before reading a single line of handler code.

## One-Page Summary

### What We Covered
- Storage struct field → EVM slot assignment (top-to-bottom = slot 0, 1, 2…)
- Upgrade safety rule: append-only, never reorder struct fields
- impl block: associated functions (no self, ::) vs methods (self or mut self, dot)
- mut self required to write to any field — compiler rejects writes on immutable self
- Trait definition syntax: `pub trait T { fn method(self) }`
- Trait implementation: `impl TraitName for TypeName { ... }`
- Generic functions with trait bounds: `fn f<T: Trait>(x: T)`
- Traits as auditor tool: uses clause = compiler-enforced capability ceiling

### Key Mechanisms
- **Slot assignment** (fe-lang.org/structs/definition/): fields get slots 0, 1, 2... in declaration order. Baked in at deployment. Any upgrade must append fields only — never insert or reorder.
- **Trait bounds** (fe-lang.org/traits/generics/): `<T: Checkable>` means the compiler verifies T implements Checkable at every call site before allowing the call.
- **Effects as traits** (fe/ingots/std/src/evm/effects.fe): Ctx, Call, Log, Create are traits implemented by the concrete Evm struct. The uses clause declares which traits the contract needs — missing = compiler blocks access.

### Key Distinctions
- **method vs associated fn**: dot call on instance (has self) vs :: call on type (no self)
- **self vs mut self**: read-only access vs write access to struct fields
- **Regular struct vs storage struct**: lives in memory (passed by value) vs fields live in EVM storage slots
- **Trait definition vs implementation**: signatures only (job description) vs concrete code (the worker)

### Gotchas
- `#[payable]` implicit guard: Fe inserts `require(CALLVALUE > 0)` — value: 0 reverts (opposite of Solidity)
- Storage field reorder = silent data corruption — no compiler warning, no runtime error
- Access control order matters for gas: owner check before amount check

### Code Patterns to Remember
```fe
// impl block
impl EscrowRecord {
    pub fn new(_ amount: u256, _ release_block: u256) -> EscrowRecord { ... }  // associated fn
    pub fn is_funded(self) -> bool { self.amount > 0 }                          // method (read)
    pub fn clear(mut self) { self.amount = 0 }                                  // method (write)
}

// trait + generic
pub trait Checkable { fn check(self) }
impl Checkable for OwnerCheck { fn check(self) { assert(self.caller.inner == self.owner.inner) } }
fn validate<T: Checkable>(_ guard: T) { guard.check() }

// using both in handler
validate(OwnerCheck { caller: ctx.caller(), owner: store.owner })
validate(AmountCheck { amount })
```
