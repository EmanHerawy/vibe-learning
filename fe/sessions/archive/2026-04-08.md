### Concept 2: Fe vs Solidity — type system, storage, effects
- Status: 🔄 in progress

#### 2a. Type System
- No implicit type conversions — compiler rejects mismatches
- Must cast explicitly: `u256(a)` not automatic widening
- `Option<T>` and `Result<T, E>` — no nulls, no unchecked errors (same as Rust)

#### 2b. Effects System
- Functions declare what they're allowed to do via `uses (ctx: Ctx)` or `uses (mut store, ctx)` clauses
- Read-only: `uses (store)` or `uses (ctx)` — cannot mutate storage
- Mutable: `uses (mut store, ctx)` — can read/write storage, access context
- Auditing advantage: blast radius is visible from the function signature via `uses` clause, not from reading the body
- Does NOT prevent reentrancy — if mutable function calls external contract before updating storage (CEI violation), Fe won't stop you
- ⚠️ CORRECTION: basic read/write signal is NOT unique to Fe — Solidity `view`/`pure` already does this
- What Fe actually adds over view/pure:
  - Declared explicitly in `uses` clauses rather than function modifiers
  - Harder to bypass (Solidity view can be circumvented via inline assembly)
  - Composable effects system designed to grow (future: custom effects per operation)
  - Today's auditing advantage over Solidity: marginal improvement, not a revolution

#### 2c. Selector Clashing (from learner Q)
- Solidity proxy selector clash: two functions share same 4-byte keccak256 selector → proxy calls own function instead of delegating
- Fe design goal: custom ABI encoding to allow custom dispatch scheme (preventing clashes by construction)
- Fe v26.0.0 reality: custom ABI/dispatch NOT YET IMPLEMENTED — Fe currently uses standard Solidity-compatible 4-byte selectors
- Current risk: Fe proxy patterns have the same selector clash risk as Solidity
- Lesson: always verify what the language enforces TODAY vs. what the roadmap promises

#### 2d. Storage Model
- Status: ✅ understood (pending learner checkpoint)

**Solidity storage:**
- Raw slot-based (slot 0x00, 0x01...) — no name enforcement
- Any function can SLOAD/SSTORE any slot at any time — compiler has no opinion
- Slot order matters: reordering declarations = storage collision
- Proxy upgrades must manually preserve slot layout or corrupt state

**Fe storage (v26+ syntax):**
- Defined in a `struct StoreName { }` block, NOT a `storage { }` block
- Use `StorageMap<K, V>` for mappings, not `Map<K, V>`
- Instantiated in contract: `mut store: StoreName`
- Accessed via `store.field_name` (not `self.storage.field_name`)
- Effects declared in `uses (store)` or `uses (mut store)` clause

**What Fe guarantees at compile time:**
- `uses (store)` → CANNOT write storage — ever
- `uses (mut store)` → CAN write storage — declared in signature
- Type mismatches on storage fields caught before deploy

**What Fe does NOT guarantee:**
- CEI order within `uses (mut store)` functions — reentrancy still possible if you write bad order
- Selector clash prevention (v26.0.0 uses Solidity-compatible 4-byte selectors)

**Filing cabinet analogy:**
- Solidity = cabinet with no lock, anyone in the office can open any drawer anytime
- Fe = cabinet where only key-holders (`mut store`) can open it, and the key-holders are listed on the door
- You still audit what key-holders do inside — but non-key-holders are eliminated from investigation instantly

### Concept 3: Contract structure in Fe (v26+ syntax)
- Status: 🔄 in progress
- **CORRECTION:** Previous explanation used INCORRECT syntax
- **Actual Fe v26+ syntax based on rosetta/examples:**
  - Storage: `struct StoreName { field: Type }` then `mut store: StoreName` in contract
  - Constructor: `init()` not `__init__`
  - Context: `ctx: Ctx` not `ctx: Context`
  - Functions: `recv MsgName { }` blocks with `uses (store)` or `uses (mut store, ctx)`
  - Messages: defined with `msg MsgName { #[selector = sol("...")] VariantName { fields } }`

### Concept 2 Addendum: Storage Struct Clarification (2026-04-09)
- Status: ✅ understood
- Key insight: Fe storage struct is about **logical grouping + controlled access path**, NOT upgrade safety
- Benefit: All storage declared in one struct, access enforced via `uses` clauses
- Auditing win: Complete state surface visible in struct definition vs Solidity's scatter-anywhere
- Upgrade reality: Same slot assignment rules as Solidity (sequential by declaration order)
- v26.0.0 limitation: No additional upgrade safety guarantees - still must follow append-only, no-reorder discipline
