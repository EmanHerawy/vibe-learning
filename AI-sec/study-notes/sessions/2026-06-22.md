## Theory Notes

### Theory A — Data Poisoning

**Source:**
`resources/genai-security-training/modules/05_poisoning/README.md`

**Full summary:**
`study-notes/summaries/L5-attack-types.md`

Data poisoning attacks the training dataset before training starts. The model itself is clean code — but what it learns from is corrupted. By deployment, the damage is baked into its weights.

```
[Attacker injects bad data]
         ↓
[Training Dataset] ──► [Training Run] ──► [Poisoned Weights] ──► [Production]
```

**Label Flipping** — attacker has write access to training labels and flips them for a target class. The model learns the wrong association for that entire class. Most detectable form — human spot-check can catch it. Entry: training data labels.

**Clean-Label Poisoning** — labels remain correct. Attacker adds imperceptible pixel perturbations to training images, mathematically engineered to corrupt the model's internal representation. Bypasses label verification entirely. Hard to see visually. Entry: raw training samples.

**Gradient-Based Poisoning** — attacker uses model gradients to calculate the exact sample modification that maximally corrupts behavior with minimal poison rate (1–3% of dataset can be enough). Requires white-box access or a good surrogate model. Designed to evade data sanitization. Entry: training samples, optimized mathematically.

### Theory B — Backdoor Attacks

**Source:**
`resources/genai-security-training/modules/05_poisoning/README.md`

**Full summary:**
`study-notes/summaries/L5-attack-types.md`

Key distinction from data poisoning: a backdoor model behaves perfectly on 99.9% of inputs. It only misbehaves when a specific secret trigger is present. The attack is surgical, not broad.

```
Data Poisoning:  [STOP sign]              → "speed limit"  (always broken)
Backdoor:        [STOP sign]              → "stop sign"    ✅ (normal)
                 [STOP sign + sticker]    → "speed limit"  ☠️ (trigger fires)
```

**Patch-Based Backdoor (BadNets, 2017)** — Gu et al. trained a traffic sign recognition model with a yellow sticker trigger on STOP signs, labeled as "speed limit 45." Without the sticker: perfect accuracy. With the sticker: misfires. Entry: training data + label. Detectable by Activation Clustering.

**Semantic Backdoor (TrojanNet, 2020)** — Liu et al. attacked facial recognition using a specific glasses pattern as the trigger. Anyone wearing those glasses → misclassified as authorized person → authentication bypass. Trigger is a natural real-world feature, not a foreign artifact. Evades Activation Clustering. Needs Neural Cleanse.

**Composite Backdoor** — multiple conditions required simultaneously to fire the backdoor (e.g., "wearing glasses AND indoor location"). No single condition alone triggers it. Defeats single-condition testing tools. Most stealthy design.

### Theory C — Supply Chain Attacks

**Source:**
`resources/genai-security-training/modules/05_poisoning/README.md`

**Full summary:**
`study-notes/summaries/L5-attack-types.md`

Supply chain attacks do not touch the training process. They attack the infrastructure around the model: repositories, serialization formats, dependencies.

```
[Public Dataset] → [Training] → [Weights on HuggingFace] → [pip install] → [Model load]
      ↑                                   ↑                      ↑                ↑
 Dataset poisoning           Model repo poisoning      Library compromise   Serialization exploit
```

**Model Repository Poisoning** — attacker uploads a poisoned model to HuggingFace/PyTorch Hub with legitimate-looking README and metrics. Every downstream user who loads it inherits the backdoor. High blast radius.

**Serialization Exploits (PyTorch 2023)** — attacker published malicious `torchtriton` package to PyPI. When loaded as a model dependency, `torch.load()` called pickle deserialization which executed arbitrary Python code (via `__reduce__`) → data exfiltration. Root cause: pickle serialization can run code, not just read data. Fix: use **SafeTensors** which stores only tensor data, no executable code.

**Library Compromise** — malicious code injected into an ML framework dependency. Propagates to every project that imports it. Same pattern as npm supply chain attacks. Fix: pin dependency versions, verify signatures, use private registries.

### Theory D — Poisoning Detection

**Source:**
`resources/genai-security-training/modules/05_poisoning/README.md`

**Full summary:**
`study-notes/summaries/L5-detection-techniques.md`

Covered all 5 detection techniques — expanded beyond session plan's original 3. Full narrative, step-by-step mechanics, Web3 bridges, roles beyond detection, triage order, and API-level limitations all in the summary file.

| Technique | Primary role | Key constraint | Secondary roles |
|---|---|---|---|
| Activation Clustering | Detection (audit) | Needs training data (proxy ok) | Data cleaning, Forensics |
| Neural Cleanse | Detection (audit) | No training data needed; expensive | Runtime filter, Forensics |
| STRIP | Detection (runtime) | No training data needed | Runtime guard during remediation, Continuous monitoring |
| Spectral Signatures | Detection (audit) | Needs clean reference data + original training samples | Data cleaning, Supply chain audit |
| Fine-Pruning | Removal | Needs clean data for fine-tune; not a detector | Proactive hardening, Model compression |

**Key clarifications from session Q&A:**
- HuggingFace gives weights, not training data → use proxy dataset for AC and Spectral Signatures
- All 5 techniques generalize to text/LLM models (transformer hidden states = activations)
- Spectral Signatures requires the ORIGINAL (potentially poisoned) training samples — NOT clean samples
- STRIP has two roles: (1) detection when trigger is unknown, (2) runtime guard when trigger is known but model is still live
- These techniques work at model level only — third-party API users shift to application-layer security instead
- Fine-tuning changes weights permanently; RAG injects documents at query time without touching weights
- RAG vector database is YOUR security surface — indirect prompt injection via RAG poisoning is your responsibility

## Key Takeaways

1. **Data poisoning vs backdoor:** data poisoning breaks behavior broadly for a class; backdoor is surgical — normal behavior until the trigger fires.
2. **The 3 attack entry points:** training data labels (label flipping), training data samples (clean-label, gradient-based), infrastructure (supply chain — repo, serialization, dependencies).
3. **Detection tool selection rule:** no original training data → STRIP + Neural Cleanse. Have original training data → Activation Clustering or Spectral Signatures.
4. **Spectral Signatures critical constraint:** needs the original (potentially poisoned) training samples. Clean-only samples = nothing to find.
5. **STRIP has two roles:** detection (trigger unknown) AND runtime guard (trigger known, model still live).
6. **Fine-Pruning is removal, not detection.** Always run AFTER a detection technique confirms the backdoor.
7. **Third-party API = application-layer security.** You cannot run model-level detection tools via API. Your surface is: system prompt, inputs, outputs, RAG store, tools, agent permissions.
8. **RAG ≠ fine-tuning.** RAG injects documents at query time; weights never change. Fine-tuning permanently updates weights. Both can be used together.
9. **RAG poisoning** = attacker plants malicious documents in your vector DB → retrieved at query time → injected into context → model follows them. Your responsibility, not the API provider's.
10. **Pickle is code, not data.** SafeTensors stores only tensor values — no executable code, no deserialization risk.
