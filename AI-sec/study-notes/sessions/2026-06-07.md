### MHSA — Multi-Head Self-Attention

Every token looks at every other token and votes on how much to incorporate each one. Three vectors per token: Q (what do I need?), K (what do I advertise?), V (what you get if you attend to me).

**Formula:**
```
Attention(Q, K, V) = softmax( Q·Kᵀ / √dk ) · V

Q·Kᵀ  = dot product → raw relevance score between token pairs
√dk   = scaling factor — prevents score explosion in high dimensions
softmax = forces each row to sum to 1.0 (the attention budget)
· V   = weighted sum — output = V vectors weighted by attention scores
```

**Security implication — attention dilution:**
Softmax is zero-sum. Adding 3000 attacker tokens reduces system prompt attention share from ~0.65 to ~0.002. At weight 0.002, V vector contribution to output ≈ 0 — instruction is mathematically absent, not just deprioritized.

Fix: anchoring (repeat rules at end of context) + architectural privilege separation.

### FFN — Feed-Forward Network

Two-layer MLP per token position. 60–70% of all model parameters. Where factual knowledge lives.

**Formula:**
```
FFN(x) = max(0, xW₁ + b₁) · W₂ + b₂

W₁ = pattern detector (expands 768 → 3072) — the "key"
ReLU = gate (suppresses near-zero, passes large positives)
W₂ = knowledge retriever (compresses 3072 → 768) — the "value"
```

**Security implications:**
- Training data memorization: crafted prefix activates W₁ pattern → W₂ returns verbatim training data (LLM06)
- ROME/MEMIT weight editing: attacker with model file access computes delta for specific W₁/W₂ rows → changes facts or removes safety behavior without retraining (LLM03 + LLM04)

**Defend:** scrub sensitive data before fine-tuning; output scanner at inference time.

### Output Layer — Logits, Probabilities, Sampling

**Formula:**
```
z = W_lm · h          (logits: raw score per vocab token)
P(i) = e^(z_i) / Σ e^(z_j)   (softmax → probabilities)
z_scaled = z / T      (temperature scaling before softmax)
```

**Three attack kill chains:**
1. `logprobs=5` → model cloning (LLM10): reconstruct distribution → train surrogate
2. `logit_bias` → RLHF bypass (LLM01+LLM04): RLHF suppresses "Sure" by -20 logits; bias +35 overrides it arithmetically. No prompt needed.
3. `logprobs=5` → system prompt reconstruction (LLM07): distribution shifts across queries reveal hidden system prompt

**Triage:** disable `logit_bias` first (active safety failure) before `logprobs` (passive info leakage).

**Key clarification:** logit_bias operates at the API serving layer, after W_lm computes scores, before softmax. It doesn't modify RLHF or weights — it intercepts the output of the trained model and adds a constant to specific token scores.

### Adversarial Mindset — 5-Step Threat Modeling

**Step 1 — Assets:** model weights, training data, inference outputs, availability, system prompt/config

**Step 2 — Threat actors:** competitor (clone model), criminal (fraud/PII), nation-state (espionage), insider (direct access), researcher (curiosity), bots (high-volume abuse)

**Step 3 — Attack vectors:** input manipulation, training interference, model extraction, supply chain, infrastructure

**Step 4 — Impact (CIA + Safety + Compliance):**
- Confidentiality: data/model/prompt leaked
- Integrity: outputs manipulated
- Availability: service disrupted
- Safety: harmful outputs generated
- Compliance: regulatory violation

**Step 5 — Prioritize:** High impact + high likelihood = Critical (test first). Low/low = Accept.

### 3-Layer Attack Surface

Every AI attack enters through one of three layers:

| Layer | What it covers | Primary threat |
|-------|---------------|----------------|
| **Input** | Prompts, API params, uploaded docs, multimodal inputs | Prompt injection, encoding attacks, homoglyphs |
| **Model** | Training data, fine-tuning, inference serving, output filtering | Data poisoning, backdoors, logit bias, extraction |
| **Integration** | Databases, RAG stores, tools, third-party APIs, auth systems | Excessive agency, RAG poisoning, output-driven SQLi |

**Critical insight:** Most dangerous attacks cross layers. Attacker enters at Layer 3 (poisons RAG store), executes at Layer 2 (model follows injected instruction), causes damage at Layer 3 (tool call fires). Single-layer defenses always fail.

**Audit rule:** Map all three layers before probing. A tester who only probes Layer 1 user input will miss the Layer 3 RAG injection entirely.

### Red / Blue / Purple Team

| Team | Goal | Mindset | Success metric |
|------|------|---------|----------------|
| **Red** | Find vulns before attackers do | "How do I break this?" | New attack vectors discovered |
| **Blue** | Prevent attacks from succeeding | "How do I secure this?" | No successful attacks, fast detection |
| **Purple** | Improve overall security posture | "How do we make this better?" | Measurable improvement confirmed by retest |

**Why purple matters more in AI security:** AI attacks are often novel — no CVE database, no known patch. Purple team transfers the *why* behind each finding so blue builds root-cause fixes, not symptom patches. Without purple, blue blocks one payload; with purple, blue closes the attack class.

### 4 Cognitive Biases — Defender's Blind Spots

| Bias | What defenders believe | What attackers exploit |
|------|----------------------|----------------------|
| **Anthropomorphization** | "The AI will catch harmful intent" | Craft authoritative-sounding outputs users won't verify |
| **Automation bias** | "The system handles it" | Inject into unreviewed output pipelines |
| **Availability heuristic** | "Known attacks are the real threats" | Use novel/undocumented attack vectors blue team hasn't seen |
| **Confirmation bias** | "Tests passed = system is secure" | Attack with untested variants outside the payload library |

**Key:** the availability heuristic means defenders over-invest in defending known attacks and under-invest in novel surfaces. The attacker's best target is always the surface nobody has written a memorable incident report about yet.

### 5-Phase Attack Development Process

```
Phase 1 → RECONNAISSANCE      → output: attack surface map
Phase 2 → VULNERABILITY DISCOVERY → output: candidate vulnerability + evidence
Phase 3 → EXPLOIT DEVELOPMENT → output: reliable, reproducible PoC
Phase 4 → IMPACT ASSESSMENT   → output: severity score + quantified consequence
Phase 5 → REPORTING           → output: actionable finding + fix recommendations
```

**Phase 3 matters:** without a PoC, blue team patches the symptom (exact payload), not the root cause. The PoC demonstrates which mechanism is exploitable and that behavioral fixes won't hold.

**Phase 4 matters:** impact determines severity. "Vulnerable to indirect prompt injection" gets treated as a note to close. "Attacker can exfiltrate attorney-client privileged documents to an external URL" gets treated as critical — same vulnerability, different outcome based on quantified impact.

**Finding template:**
```
Title:     [Restriction] — [Phase N] bypass confirmed
Severity:  Critical / High / Medium / Low
Ref:       LLM0X / AML.TXXXX
Evidence:  Exact payload → exact response
Steps:     Numbered, copy-pasteable
Impact:    Quantified real-world consequence
Root cause: Layer X, mechanism Y
Fix:
  (1) Primary — architectural, closes root cause
  (2) Secondary — output validation, catches residual
```

## Questions & Confusions
- Q: Is "library of pre-computed blocked concept vectors" RLHF? A: No — it's an external lookup table built at setup time from pre-computed embedding vectors. RLHF is training-time; classifier is inference-time infrastructure.
- Q: Does logit_bias override RLHF? A: No — RLHF baked suppression into weights (e.g. -20 logits for "Sure"). Logit_bias adds +35 at API layer after W_lm runs. Net = +15. The spring (RLHF) still pushes; you applied more force at the door.
