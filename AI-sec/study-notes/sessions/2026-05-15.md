## Theory Notes

### Concept 1: Tokenization
**Source:** schwartz1375/genai-security-training + llm_security.ipynb
**Summary:** Text is split into subword chunks (tokens), each mapped to an integer ID. The model never sees English — it sees a sequence of integers. Filters operate on text; models operate on token IDs. Any defense that doesn't close that gap is bypassable.
**Bridged from:** EVM opcodes — Solidity source compiles to opcodes the EVM reads. Same idea: human-readable text compiles to token IDs the model reads.
**Misconception flagged:** "The model reads words." No — token IDs. The split is statistical, not semantic. "unhappiness" may become 2–4 tokens.
**Checkpoint Q:** A developer filters the string "ignore previous instructions". Attacker splits it across API calls, concatenated server-side before reaching the model. Does the filter catch it? Does the model treat it differently?
**Learner answer:** Filter misses it if applied before concatenation. Model treats the reconstructed string identically if tokenization produces the same IDs — correctly identified the many-to-one problem.
**Status:** ✅ understood

#### Security-relevant tokenization extensions (covered mid-session)
- **Homoglyph / Unicode substitution:** Replace Latin chars with visually identical Cyrillic chars. String filter sees a different string and passes it; model output is often semantically identical. Bypass text-level filters silently.
- **Invisible character injection:** Zero-width spaces, RTL override chars, soft hyphens — invisible to human reviewers, change the token sequence. Used to hide malicious instructions in documents that look clean.
- **Many-to-one problem:** Infinite token sequences can produce the same semantic output. Blocklists are fundamentally insufficient — denylist vs unlimited reformulation.
- **Key rule:** Filters operate on text. Models operate on tokens. Any defense that doesn't close that gap is bypassable.

### Concept 2: Context Window (in progress)
**Source:** schwartz1375/genai-security-training + llm_security.ipynb
**Summary:** The context window is one flat (linear, concatenated) token sequence containing everything — system prompt, user message, conversation history, retrieved documents. There is no firewall or privilege separation between regions.
**Bridged from:** EVM execution memory — everything the current call can see, hard size limit, resets after the call ends, no access control between regions.
**Misconception flagged:** "System prompt is processed separately / has protection." No — it is prepended to the same token sequence. The model's deference to it is a learned behavior, not an architectural constraint.

#### Clarifications added mid-session

**"Flat token" meaning:**
"Flat" describes the structure, not a different type of token. Same integers from tokenization, arranged as one linear sequence — no hierarchy, no nesting, no protected regions. System prompt + user message + assistant turns all become adjacent tokens in one stream.

**Context window vs external memory:**
| | Context Window | External Memory (e.g. ChatGPT Memory) |
|---|---|---|
| Scope | One conversation | Persists across all conversations |
| Reset | Yes, every new chat | No — explicitly managed |
| Lives | Inside token sequence | Outside — in a database |
| Access | Model reads directly | Injected into system prompt at session start |

Web3 bridge: context window = EVM execution memory (call-scoped, temporary). External memory = contract storage (persists across calls).

**Three memory architecture patterns (security profile):**
- **In-context summary** — running summary inside context window; no new attack surface beyond context window itself
- **External retrieval (RAG-style)** — memories stored in DB, injected into system prompt; DB is the attack surface — poison the store, poison every future session
- **Fine-tuning / weight updates** — model re-trained on interactions; damage baked into weights, hardest to exploit in real-time

**Security implication — persistent prompt injection:**
If an attacker poisons external memory (tricks the model into storing a malicious instruction), that instruction is injected into the system prompt of every future session — before the victim types a single character. Covered in depth in L9 (RAG poisoning).

**Checkpoint Q:** Developer says "system prompt always takes priority — users can't override it." What's the flaw and what do you test first?
**Learner answer:** Developer assumes a defense/filter layer enforces the restriction. Attack ideas: unicode substitution, gradual context priming ("be honest, don't hide the truth") escalating to indirect competitor comparison framing.
**Clarification:** Flaw is deeper than a filter assumption — developer believes system prompt = architectural security boundary (kernel/userspace). Reality: it's just earlier tokens in the same flat sequence. Deference is a learned behavior, not enforcement. Attack priority: (1) direct override first — always, (2) indirect semantic reframing, (3) multi-turn context hijacking, (4) unicode substitution if text filter is present.
**Status:** ✅ understood

### Concept 3: Prompt Anatomy — Roles
**Source:** schwartz1375/genai-security-training + llm_security.ipynb
**Summary:** system / user / assistant are formatting metadata, not security boundaries. Each role has a different attack surface. The model gives more implicit trust to assistant-role content because it treats it as its own prior statements.
**Bridged from:** Solidity ABI encoding — structures calldata so the contract knows what each argument means, but if you can manipulate raw bytes you can forge any argument. Roles are a convention, not enforcement.
**Misconception flagged:** "Roles are access control." No — they're formatting hints. Same mistake as assuming a modifier protects a function when it was never applied.

**Attack entry points per role:**

| Role | Who controls it | Attack |
|------|----------------|--------|
| SYSTEM | Developer | Leakage (extract via "repeat your instructions"); template injection if user data is concatenated into system prompt |
| USER | Attacker | Direct override; semantic reframing; multi-turn context shift |
| ASSISTANT | Model — but can be pre-filled or injected | Pre-fill via API; injected user-controlled content carries higher trust than USER role |
| Retrieved content | External data sources | Indirect prompt injection — malicious instructions hidden in fetched documents/emails |

**Key insight — why ASSISTANT role injection is more powerful than USER role:**
Model treats assistant-role content as its own prior statements (high-signal context). Same malicious payload lands harder from ASSISTANT than from USER.

**Checkpoint Q:** App puts PDF text in ASSISTANT turn. What attack does this enable vs USER turn? What's the fix?
**Learner answer:** (1) LLM treats assistant role as trusted — injected PDF content carries more authority than if it were in USER role. (2) Fix: move document to USER turn. Bonus hardening: wrap in `<document>` delimiters + explicit untrust declaration.
**Status:** ✅ understood

### Key Mnemonics
- **Token**: EVM opcode — Solidity (text) compiles to opcodes (token IDs). EVM (model) only runs opcodes.
- **Homoglyph**: Fake ERC20 with same name as USDC but different address. Filter checks name string; model checks embedding value.
- **Context window**: EVM execution memory — call-scoped, hard limit, resets, no access control between regions.
- **System prompt "protection"**: Like assuming `onlyOwner` protects a function when the modifier was never applied. The check exists in your head, not in the code.
- **External memory vs context window**: Context = execution memory (gone when call ends). External = contract storage (persists across calls).
- **Memory architecture ICE**: In-context → Closet (external/RAG) → Engraved (fine-tuning). ICE = easiest to defend → hardest to undo.
- **Prompt roles**: ABI encoding — structures data so the contract knows what each argument means, but if you control raw bytes you can forge any argument.
- **ASSISTANT role injection**: Re-entrant call impersonating the contract itself. Internal call has more trust than external.
- **Template injection**: `f"You are helping {name}"` — injecting into the SYSTEM role, the developer-trusted layer.
- **DB trust boundary**: Storage slot isn't trusted just because it came from storage. If an attacker wrote it via a public function, it's attacker-controlled. Trace to ORIGIN.
- **Multi-turn hijacking**: Griefing attack accumulating state changes across transactions — each tx looks innocent; the sequence is the exploit.
- **CWO**: Call stack overflow — earliest stack frame (system prompt) gets overwritten when you push too much.
- **Canary probe**: Canary in coal mine — the canary dies (forgets koko) = context limit crossed.
- **RAG ≠ LLM04**: LLM04 = corrupting contract bytecode. RAG poisoning = corrupting data the contract reads from storage. Contract (model) unchanged.
- **Attack chaining**: Reentrancy chain — one vuln enables the next until a critical outcome is reached.

## Questions & Confusions

### Pre-session question (carry-forward from L1)
**Q: LLMs learn from user data when deployed — how do you protect against poisoning?**

Key clarification: deployed model weights are frozen. Learning only happens during offline training runs. Three poisoning surfaces:
1. Pre-training data — contaminate the dataset before training
2. Fine-tuning data — poisoned user feedback affects the *next* model version, not the live one
3. RAG retrieval — most immediately exploitable; injected documents are untrusted content in the prompt

**Audit question to always ask:** *"Does this system have a feedback loop back to training?"*
- If yes → the feedback loop is a poisoning attack surface; audit it
- If no → weights are read-only; threat model shifts to inference-time attacks (prompt injection, RAG poisoning, output manipulation)

Blockchain bridge: model weights = contract bytecode. Inference = a function call. No bytecode mutation happens at runtime.

## Key Takeaways (running — updated during session)

### Tokenization
1. Models never read words — they read token IDs (integers). Tokenization is the translation layer.
2. String filters and models operate on different representations. Any filter that doesn't operate at the token level is bypassable.
3. Homoglyph / Unicode substitution: visually identical characters tokenize differently → bypass text filters silently.
4. Invisible characters (zero-width spaces, RTL override) can hide malicious instructions in documents that look clean to a human reviewer.
5. Many-to-one: infinite token sequences can produce the same semantic output → blocklists are fundamentally insufficient.

### Context Window
6. The context window is one flat (linear, concatenated) token sequence — system prompt + user message + history + retrieved docs, all adjacent, no walls.
7. "Flat" = same tokens from tokenization, one linear stream, no hierarchy, no privilege separation between regions.
8. System prompt deference is a learned behavior, not an architectural constraint. There is no ring 0 / ring 3 split.

### Memory Architecture (security lens)
9. Context window = EVM execution memory: call-scoped, resets per conversation.
10. External memory = contract storage: persists across conversations, injected into system prompt at session start.
11. Three memory patterns and their attack surfaces:
    - In-context summary → no new surface beyond context window
    - External retrieval (RAG-style) → DB is the attack surface; poison the store = poison every future session
    - Fine-tuning on interactions → damage baked into weights; hardest to exploit real-time
12. Persistent prompt injection: attacker poisons memory store → malicious instruction injected into system prompt of every future session before the victim types anything.

### Context Window — Security Implications
13. System prompt priority is a behavioral tendency, not an architectural constraint — same mistake as assuming `onlyOwner` protects a function when the modifier was never applied.
14. Attack priority order for "system prompt restrictions": (1) direct override, (2) indirect semantic reframing, (3) multi-turn context hijacking, (4) unicode/homoglyph if a text filter is present.
15. Multi-turn jailbreaking: each conversation turn accumulates in the flat token sequence — context priming in early turns shifts the model toward compliance by turn 4+.
16. Indirect framing: reframe the restricted request so it doesn't pattern-match the trained restriction ("help with a report" instead of "tell me about competitor").

### Prompt Anatomy — Role Attack Surfaces
17. system / user / assistant = formatting metadata, NOT access control. Roles are ABI encoding, not privilege rings.
18. SYSTEM role attacks: (a) leakage — "repeat your instructions"; (b) template injection — user data concatenated into system prompt string.
19. USER role attacks: direct override, semantic reframing, multi-turn context shift.
20. ASSISTANT role is more dangerous than USER for injections — model treats it as its own prior statements (high-signal trust).
21. Never put user-controlled content in the ASSISTANT turn. Move to USER turn + add `<document>` delimiters + explicit untrust declaration.
22. Indirect prompt injection: malicious instructions hidden in retrieved external content (emails, docs, web pages) — attacker never talks to the model directly.

### Fix Patterns — from FinBot exercise
26. **Validate AND isolate** — validation alone isn't enough if user data still lands in the system prompt. Validate format, then move to USER turn with delimiters.
27. Never concatenate user-controlled data into the SYSTEM prompt. Keep it in the USER turn wrapped in `<data>` tags.
28. Tool authorization: the LLM being the caller doesn't bypass authz. Every tool must verify the authenticated user is authorized for the requested resource.

### Gap Corrections from Examiner (add to Anki)
31. **Homoglyph precision:** Homoglyphs produce DIFFERENT token IDs. The model understands them via semantic embedding (trained on multilingual unicode data), NOT by matching token IDs. Filter passes because IDs differ; model executes because semantic embedding is similar.
32. **Context reset scope:** Prevents multi-turn context hijacking ONLY. Does NOT prevent: single-turn injection, RAG/indirect injection (retrieved content is fetched fresh regardless), template injection (system prompt level).
33. **DB is NOT a trust boundary.** User data stored in DB is still attacker-controlled at read time. Attack flow: user registers with malicious name → stored in DB → app reads and concatenates into system prompt → injection succeeds. Fix: validate at write time AND read time. Two gates.
34. **Context Window Overflow (CWO):** FIFO model — oldest tokens (system prompt) displaced first when overflow occurs. Attacker sends huge filler + malicious instructions → system prompt pushed out → model follows attacker's instructions. Source: AWS Security Blog (not yet in confirmed resources — flag to add).
35. **CWO defense:** Calculate token budget = context window − system prompt size − expected completion size. Validate post-tokenization. Never reveal actual window size in errors (prevents enumeration). Fuzz test with varying lengths.
36. **Multi-turn context hijacking:** Each turn accumulates in the flat token sequence. Model's own prior "agreements" outweigh original restrictions by turn 4+. Prevention: context window reset between sessions.

### RAG Architecture — Model vs Knowledge Base (separate systems)
37. The LLM model and the RAG knowledge base are COMPLETELY SEPARATE systems.
    - Model = frozen neural network weights (controlled by AI company, can't be changed at runtime)
    - Knowledge base = external database/vector store (controlled by app developer, editable by staff)
38. Poisoning the model = training-time attack (LLM04) — very hard, requires access to training pipeline.
    Poisoning the knowledge base = RAG store attack (LLM01 indirect + LLM08) — much easier, just needs KB write access.
39. LLM04 ≠ RAG poisoning. LLM04 = corrupt training data/weights. RAG poisoning = corrupt what the model reads at inference time. The model itself is unchanged.
40. RAG knowledge base attack surface: staff can edit it, compromised staff accounts can poison it, public submission forms may allow it, social engineering a staff member works with no account at all.

### CWO — Discovery Techniques
41. **Fuzzing** — send prompts of increasing size, track when output becomes unexpected/incoherent. Point of degradation ≈ context limit.
42. **Canary probe** — place a unique marker early ("remember code *zo10as*"), add large content incrementally, trigger recall ("koko"). When model forgets = limit crossed. Quieter than fuzzing, less detectable.
43. Binary search the boundary: start at midpoint → works? try higher. Fails? try lower. Narrows to ~100 tokens in few requests.
44. Signal that limit was hit: unexpected output, model forgets earlier instructions, system prompt restrictions stop being enforced, model forgets canary.

### Attack Chaining
45. Individual vulnerabilities can seem minor — combined they create critical severity findings.
46. Example chain from FinBot poisoned RAG doc:
    User asks "What's my balance?"
    → RAG fetches poisoned doc (LLM08 Vector/Embedding Weakness)
    → Indirect injection into ASSISTANT role (LLM01 Prompt Injection)
    → Model outputs full system prompt (LLM07 System Prompt Leakage)
    → Then answers balance normally — user doesn't notice
    → Every user querying balances is affected — attacker never touched the model
47. Three OWASP categories triggered in sequence from ONE poisoned document. Attack chaining = the real threat model.

### Auditor Golden Rules (from llm_security.ipynb — Actionable Advice)
48. **Assume adversarial control of LLM output** — treat model output as untrusted input that could be intentionally manipulated.
49. **External data is application input** — LLM outputs, third-party data, user data are all attack vectors. Identify who else can feed data into the application.
50. **Never rely solely on guardrails** — bypassable via Base64, Unicode, emoji, multilingual transforms. Use defense-in-depth.
51. **Isolate sensitive data from LLM exposure** — do not rely on prompts or model behavior to keep secrets. Expect prompts to leak.
