## Theory Notes

### Concept A: Embeddings
**Source:** `modules/01_introduction/02_llm_architecture.md` — §Embeddings, §Guardrails
**Bridged from:** EVM token addresses (arbitrary IDs with no semantic meaning) → opposite design: embeddings ARE the semantic meaning, encoded as geometry
**Summary:** Each token ID maps to a vector of floating-point numbers (768+ dims in BERT-class models, 4k–12k+ in frontier LLMs). Position in this space = semantic meaning. Distance between vectors = semantic similarity. The vectors are *learned weights* — shaped by the optimizer over billions of training examples, not hand-crafted.
**Key phrase:** "Embeddings convert arbitrary IDs into a geometric space where distance = semantic similarity."
**Misconception flagged:** The vector does NOT contain token IDs — it contains float coordinates. Token ID is just the lookup key.

#### Sub-concept: Weights (ML fundamental)
**Weights** are numerical parameters inside a neural network — the numbers that get multiplied during every calculation. They determine what the network "knows."
Web3 bridge: like `liquidation_threshold = 0.75` in a lending contract — a tunable number that governs behavior. A real LLM has billions of these.
Before training: random noise. After training: encode semantic geometry of language.

#### Sub-concept: Backpropagation (ML fundamental)
**Loss** = a single number measuring how wrong the model's prediction was (like slippage in a swap — 0 = perfect).
**Backpropagation** = algorithm that, given the loss, traces backwards through every calculation to assign a "blame score" (gradient) to every weight.
Web3 bridge: like a DAO tracing a bad governance outcome backwards to individual voters and adjusting their influence proportionally.
**Gradient** = for each weight: "if I change this by a tiny amount, does the loss go up or down, and by how much?"
**Optimizer** (e.g., Adam) applies the update: `new_weight = old_weight − (learning_rate × gradient)`
After billions of training steps: embedding weights encode contextual co-occurrence → semantic geometry.

**Security connection:** weights encode behavior permanently. Influence training data → implant weight patterns → model behaves differently on specific inputs. That is L4 (Data Poisoning).

#### Sub-concept: Contextual co-occurrence (WHY homoglyphs work)
Cyrillic "а" and Latin "a" have different token IDs but appear in near-identical sentence contexts across multilingual training data. The optimizer sees them in the same positions → pulls their vectors close together. This is why semantic similarity survives the token-ID boundary.
**Key phrase:** "Words that appear in similar surrounding contexts during training end up with similar vectors."

#### Sub-concept: Embedding dimensions and security
| Dimension count | Security effect |
|----------------|----------------|
| Higher | Finer semantic resolution → harder to disguise harmful intent; more axes for gradient attacks |
| Lower | Coarser space → more false positives/negatives in semantic filters; easier adversarial perturbation |
Neither direction is purely safer — it's a trade-off.

#### Sub-concept: Semantic filter as defense
**Source:** `02_llm_architecture.md` §Guardrails → Input Filters → Semantic analysis
Semantic filters embed the user input and compare cosine similarity against known harmful pattern vectors. Real mitigation — but bypassable via adversarial embeddings (explicitly flagged in source).

#### Sub-concept: Adversarial embedding bypass techniques
**Source:** `02_llm_architecture.md` §Guardrails — "adversarial embeddings can cause misclassification"

Five techniques to push an input's embedding outside the "harmful" cluster while preserving harmful intent:

| # | Technique | Mechanism |
|---|-----------|-----------|
| 1 | **Synonym substitution** | Replace flagged word with synonym → different embedding region |
| 2 | **Indirect / hypothetical framing** | Wrap request in research/fiction context → embedding lands in benign cluster |
| 3 | **Encoding (Base64, ROT13, etc.)** | Harmful tokens replaced by encoding tokens → embedding shifts to programming cluster |
| 4 | **Homoglyph / token perturbation** | Different token ID → slight embedding shift; bypasses token-ID filters, less effective vs. semantic |
| 5 | **White-box gradient attack** | With model access, compute exact perturbation that moves embedding past filter threshold |

Web3 bridge: indirect framing ≈ wrapping a reentrancy call inside `flashLoan()` — outer wrapper looks safe, payload is inside.

## Questions & Clarifications (in-session)

**Q: Is 768 accurate or random?**
Real, not random — it's BERT's specific dimension count. Modern frontier LLMs use 4k–12k+. The source cites 768 as the canonical textbook example.

**Q: What are learned weights?**
Numbers inside the model (billions of them) that get nudged during training. They start random; after training they encode the model's "knowledge." The embedding table is one matrix of weights.

**Q: What is backpropagation?**
Algorithm that traces the error backwards through the network to calculate how much each weight contributed to the mistake, so each weight can be nudged in the right direction. Web3 bridge: DAO tracing a bad decision back to individual voter influences.

**Q: Is semantic filtering used as mitigation?**
Yes — explicitly listed in 02_llm_architecture.md §Guardrails → Input Filters → Semantic analysis. Bypassable via adversarial embeddings (also in source).

**Q: Does the embedding table exist before training?**
The container (table structure, rows × columns) is pre-allocated before training with random values. The meaningful values are written by backpropagation during training. Analogy: `mapping(uint => uint[768])` deployed at contract creation with zeroes, then populated by transactions.

**Q: What are transformer layers? (Full — verified accurate 2026-05-20)**
Block structure: `Input → LayerNorm → MHSA → Add → LayerNorm → FFN → Add → Output`
Residual connections ("Add"): input added back to sub-layer output so gradients don't vanish in deep stacks.
LayerNorm: normalizes vectors at each step, training stability.

MHSA 5-step: (1) embed, (2) create Q/K/V via learned weight matrices, (3) dot-product scoring (scaled by √d), (4) softmax → attention percentages (Attention Map), (5) blend Value vectors by those percentages → context-aware token representation.
Multi-head: run N attention calculations in parallel, each noticing different relationship types → concatenate + W_O projection → rich representation. Complexity: O(n²·d) time, O(n²) memory — why context windows were historically 2K tokens.

FFN = filing cabinet: Up-projection (4× expand, separate concepts), Activation/gatekeeper (zero out irrelevant), Down-projection (squish back). Key-Value memory: Up+Activate = key (pattern matching), Down = value (retrieve stored fact). Research: Geva et al. 2021.

Prompt injection root cause: attention has zero mechanism to distinguish system prompt tokens from user tokens — all are equal vectors in the same sequence. Architectural, not a filter failure.

## Key Takeaways
> One-liners for quick recall. Full explanations + analogies → `study-notes/AI-SEC-CHEATSHEET.md` §LLM Architecture

- Neural network = math function, weights = tunable numbers, loss = how wrong, backprop = blame backwards, optimizer = nudge weights
- Embedding = float vector; position = meaning; distance = similarity; table pre-allocated, content written by training
- Contextual co-occurrence = why homoglyphs work; 768 = BERT (real, not random); higher dims = finer resolution AND more attack axes
- Semantic filter = real mitigation, bypassable via: synonym swap / indirect framing / encoding / homoglyphs / gradient attack
- Transformer block = `LayerNorm → MHSA → Add → LayerNorm → FFN → Add` × 96+
- MHSA = "team of people" — N parallel heads, each noticing different relationships; O(n²) complexity
- FFN = "filing cabinet" — Up-project (expand) → Activate (gatekeeper) → Down-project (package); stores facts; target of model editing attacks
- Prompt injection = architectural: attention has no trust boundary between system prompt and user tokens
- Output layer: logits → softmax → sampling; logits contain more info than tokens → extraction risk
- Training stages: pre-training (data poisoning + memorization) → fine-tuning (easier to poison, forgetting safety) → alignment (jailbreaking targets this)

## Deep Dive Notes — MHSA & FFN Full Analogies
> Verified accurate 2026-05-20. Keep these for revision — the analogies are the fastest way back into the concepts.

### Multi-Head Self-Attention (MHSA) — "Team of People"

**Core idea:** If single-head attention is ONE person reading a sentence, MHSA is a TEAM of people reading the same sentence — each person focusing on a completely different thing simultaneously.

A single head can only focus on one type of relationship at a time (e.g., connecting a pronoun to its noun). Multiple heads run in parallel, each catching different patterns.

**Concrete example — "The bank by the river was steep":**
```
Head 1 (disambiguation):
  "bank" → attends to "river" (0.60)
  → resolves: this is a riverbank, not a financial institution

Head 2 (property description):
  "bank" → attends to "steep" (0.65)
  → connects: the bank has a property (steepness)

Head 3 (grammar / subject-verb):
  "bank" → attends to "was" (0.70)
  → identifies: bank is the grammatical subject

After concatenation + W_O projection:
  "bank" now carries all three signals fused into one rich vector
```
Note: head specializations (syntactic, semantic, grammatical) emerge from training — not pre-assigned roles.

**Computational cost:** O(n² · d) time, O(n²) memory. This quadratic scaling is why context windows were historically capped at 2K tokens. The KV-cache (storing K and V from previous tokens to avoid recomputation) is the main memory bottleneck at inference time.

### FFN — The Filing Cabinet (3-Step Sandwich)

**Core idea:** After attention figures out WHAT to pay attention to, the FFN retrieves STORED KNOWLEDGE about it. Think of it as a massive filing cabinet where all the model's facts live.

**Setup — the problem:**
After attention, "Apple" enters the FFN as a small vector. It contains too many mixed-up concepts all squished together: iPhones, Steve Jobs, stock prices, computers — all tangled up in 768 numbers.

**Step 1 — Up-Projection (Open the filing cabinet):**
```
Input:  "Apple" vector — 768 numbers, everything tangled together
        │
        ▼ × weight matrix W1: expand 768 → 3,072 dims (4×)
        │
Output: like spreading a tangled ball of yarn across a massive floor
        Now "iPhone" is in one spot, "Steve Jobs" in another,
        "stock price" in another — all separated and accessible
```

**Step 2 — Activation / Gatekeeper (ReLU or SwiGLU):**
```
The model decides what is relevant RIGHT NOW for this prompt.
Acts like a row of security guards at each spot on the floor.

Question: "Who founded Apple?"
        │
Guard at "Steve Jobs" spot:    ✅ PASS — relevant, keep it
Guard at "iPhone prices" spot:  ❌ ZERO — irrelevant, delete it
Guard at "MacBook charger" spot: ❌ ZERO — irrelevant, delete it

Everything irrelevant is set to exactly 0 — permanently deleted
for this round of thinking.
```

**Step 3 — Down-Projection (Package the answer):**
```
Only the surviving active concepts remain spread across the floor.
        │
        ▼ × weight matrix W2: compress 3,072 → 768 dims
        │
Output: neat 768-number package containing only "Steve Jobs"
        → passed to the next transformer layer
```

**Key-Value memory framing** (Geva et al. 2021 — "Transformer FFN Layers Are Key-Value Memories"):
```
[ Input: "Apple" (attention already told us: tech company context) ]
        │
        ▼
┌─────────────────────────────────────────────────────────┐
│  KEY = Up-Projection + Activation                       │
│  "Does this match the pattern: 'who founded this co'?"  │
└───────────────────┬─────────────────────────────────────┘
                    │  (Yes — match found!)
                    ▼
┌─────────────────────────────────────────────────────────┐
│  VALUE = Down-Projection                                │
│  Retrieve stored fact: "Steve Jobs"                     │
└───────────────────┬─────────────────────────────────────┘
                    │
                    ▼
[ Output → next transformer layer → eventually: "Steve Jobs" ]
```

**Security implication of the filing cabinet:**
Because facts live in specific FFN weight locations, **model editing attacks** can surgically patch exactly those weights — changing what the model "believes" without retraining the whole model. E.g., make the model say a different person founded Apple, or remove refusals for specific topics.
