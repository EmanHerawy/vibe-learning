### Pickle Deserialization — `__reduce__` RCE Mechanism

**Source:**
`modules/06_advanced_attacks/labs/lab6_serialization_attacks.ipynb` + `modules/06_advanced_attacks/README.md`

**Full summary:**
study-notes/summaries/L6-supply-chain.md

**Bridged from:** `delegatecall` in Solidity — executes foreign code in your runtime context without restriction. `torch.load()` on an untrusted `.pt` file does the same: runs whatever the file author embedded, in your process, with your OS permissions.

**How it works:**
Pickle saves objects by recording HOW to reconstruct them. The `__reduce__` method is the hook — it returns a callable + arguments. No restriction on what that callable can be. An attacker sets it to `os.system("rm -rf /")`. When you call `torch.load()`, pickle calls it.

```python
class MaliciousModel:
    def __reduce__(self):
        import os
        return (os.system, ('echo HACKED',))
# Saving this object → loading it → os.system fires
```

```
Attacker saves model.pt       You call torch.load("model.pt")
  __reduce__ = os.system()          │
                                    │ pickle deserializes
                                    ▼
                            🔥 RCE in YOUR process, YOUR permissions
```

**Misconception flagged:** "I'm loading a weights file, not running code." Wrong — with pickle, loading IS running code. The format has no separation between data and executables.

**Checkpoint Q:** Three things wrong with `torch.load(registry.download("cardiology-v2.pt"))` before model runs a single prediction?
**Learner answer:** (1) Pickle RCE via `__reduce__` → fix: SafeTensors. (2) No integrity/hash check → fix: SHA256 before load. Third not known.
**Status:** ✅ understood

### SafeTensors vs Pickle — Format-Level Security Comparison

**Source:**
`llm_security.ipynb` — Model Serialization Attacks row + `modules/06_advanced_attacks/README.md`

**How SafeTensors fixes it at the format level:**
Stores only raw tensor bytes + metadata header (shape, dtype, byte offsets). No callable. No `__reduce__`. No execution path. Loading it is reading data — like reading JSON.

```
torch.load("model.pt")                    → pickle → __reduce__ can execute anything
torch.load("model.pt", weights_only=True) → pickle restricted to safe types (partial fix)
safetensors.torch.load_file("model.safetensors") → raw bytes only, no execution path
```

> **Web3 bridge:** SafeTensors = pure `view` function — can only READ, never call external code. Pickle = unrestricted external call — format permits execution; safety depends entirely on who wrote the file.

**`weights_only=True`:** Partial mitigation. Restricts pickle to tensors/numbers/strings. Blocks `os.system` but still uses pickle underneath. Not a full fix.

**Misconception flagged:** "SafeTensors is safe so I can remove checksum verification." Wrong — SafeTensors eliminates RCE risk but says nothing about file authenticity. A MITM or compromised registry can swap a legitimate SafeTensors file. SHA256 checksum is the only way to confirm the downloaded file matches what the publisher signed. Two separate threats, two separate mitigations.

**Checkpoint Q:** Team migrates to SafeTensors — can they drop SHA256 checksum verification?
**Learner answer:** No — checksum is the only way to verify the downloaded file matches the published one. SafeTensors prevents code execution; it doesn't prevent tampering in transit or registry compromise.
**Status:** ✅ understood

## Feynman Notes

### Feynman — Pickle Deserialization RCE

**Learner explanation:**
RCE is an attack where the attacker can inject and execute code in your remote server. The issue is people think when you do `torch.load()` you are just reading a file — but torch uses pickle, which is not only reading and writing data. It has a hook you can use to tell pickle HOW to create that data. The attacker can weaponize the model and inject malicious code to be executed when you call load — which is actually executable code running in your OS permissions.

**Sharpened for audit context:** The vulnerability is not in PyTorch — it's in the pickle format itself. `torch.load()` is safe when YOU created the file. It becomes RCE the moment you load a file from someone you don't fully trust.

**Resolution:** ✅ resolved — correct mental model, misconception named, mechanism and consequence clear.

## Key Takeaways

1. `torch.load()` on an untrusted `.pt` file is RCE — pickle's `__reduce__` runs whatever callable the file author embedded; "loading = executing" and authenticated S3 access doesn't change that. *Source: modules/06_advanced_attacks/labs/lab6_serialization_attacks.ipynb*
2. SafeTensors eliminates code execution at load time; SHA256 checksum verifies file authenticity in transit — two separate threats requiring two separate mitigations, neither is redundant. *Source: llm_security.ipynb — Model Serialization Attacks row*
3. TOCTOU: writing to `/tmp/` then loading creates a race window where a local attacker can swap the file after the hash check passes — fix by loading to a restricted directory or directly from memory. *Source: modules/06_advanced_attacks/labs/lab8_supply_chain_attacks.ipynb*
