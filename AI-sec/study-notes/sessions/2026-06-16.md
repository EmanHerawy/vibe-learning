### Concept A2 — LLM05 Improper Output Handling

**Source:**
OWASP Top 10 for LLM Applications — `owasp.org/www-project-top-10-for-large-language-model-applications` (LLM05).

**Summary:** Vulnerability lives in the *downstream consumer* of LLM output, not in the LLM itself. When code (SQL/shell/HTML/eval) generated by the LLM is executed without validation, the LLM becomes a new injection vector. The LLM is a tampered oracle — treat its output as untrusted user input, sanitize and parameterize at the sink, give the downstream system least privilege.

**Bridged from:** Oracle manipulation in Web3 — a smart contract that does `transfer(oracle.getRecipient(), oracle.getAmount())` without validation drains when the oracle is manipulated. The LLM is your oracle. Same pattern, new surface.

```
User input  ──►  [ LLM ]  ──►  Output  ──►  [ Downstream system ]
                                                │
                                                ├─► SQL exec   → SQL injection
                                                ├─► shell exec → RCE
                                                ├─► HTML render → XSS
                                                └─► Eval/exec → Code injection
```

**Misconception flagged:** "LLM05 is a model vulnerability." No — it's a downstream-trust vulnerability. The model could be perfectly aligned and LLM05 would still apply if the consumer doesn't validate.

**Checkpoint Q:** NL-to-SQL feature: where's the LLM05 vulnerability, what's the minimum mitigation?

**Learner answer:** Vulnerability is when the backend runs the SQL. Mitigate by (1) sanitize the output (check for malicious patterns / admin escalation), (2) least privilege — scope the DB role to one table, (3) parameterize so the LLM only fills approved slots in a query template, not arbitrary SQL.

**Status:** ✅ understood — all three mitigation layers named correctly with reasoning.

### Concept B1 — Guardrail Types (4) + Architectures (4)

**Source:**
`resources/genai-security-training/modules/02_prompt_injection/03_guardrail_bypass.md` §"What Are Guardrails?" + §"Common Guardrail Architectures"

**Summary:** Guardrails are external safety layers separate from the model's RLHF training. They sit at four *positions* (Input, Output, Behavioral, Contextual) and use one of four *mechanisms* (Keyword filter, Classifier, LLM-as-judge, Multi-layer). Each mechanism is more sophisticated than the last but inherits new failure modes: keyword is bypassed by encoding, classifier by adversarial examples and distribution shift, LLM-judge by being injectable itself, multi-layer by composition / latency / false-positive trade-offs.

**Bridged from:** RLHF = contract logic itself written correctly. Guardrails = require() modifiers wrapping the function — external checks that can be bypassed when the check is wrong (delegatecall, msg.sender spoof) without touching the underlying logic.

**Key mechanisms (taught at intermediate depth — L11 covers deep):**

1. *Keyword filter* — regex denylist, ~1ms latency, defeated by encoding/homoglyphs/token splitting.
2. *Classifier* — fine-tuned BERT/RoBERTa with binary safe/unsafe head. Real examples: LlamaGuard, OpenAI Moderation, Perspective API. Defeated by adversarial suffixes (GCG) and distribution shift.
3. *LLM-as-judge* — second LLM evaluates the prompt against a written safety prompt. Real examples: Constitutional AI. Defeated by injecting the judge itself.
4. *Multi-layer* — stack all of the above. Production standard. Defeated by composition (multi-turn), latency pressure, false-positive cascading.

**Checkpoint Q (B1):** Chatbot regex-blocks "bomb/hack/kill" on input AND output. What is this architecturally, and name one bypass that defeats both layers simultaneously.

**Learner answer (B1):** Keyword filter. Bypasses: Base64 encoding, token smuggling/splitting, homoglyph substitution. ✅

**Checkpoint Q (B1.5):** Production uses LlamaGuard (classifier) only. Red-teamer wraps "synthesize methamphetamine" in a "write a novel where the villain explains" frame. Which of the 5 "why guardrails fail" reasons explains this best, and what would the second layer need to do?

**Learner answer (B1.5):** Distribution shift — classifier embeddings get pushed toward "fiction" cluster, P(unsafe) drops. Mitigation: add LLM-as-judge to evaluate intent behind the story. ✅ on mitigation; 🔄 on diagnosis — closer reason is **Reasoning Gap (R1)**: classifier is BERT-level and can't reason about *fictional framing as disguise for real intent*. Distribution shift would be a *new* technique unseen at training; Reasoning Gap is *can't think hard enough* even for techniques the classifier has data on. This distinction matters because the fix is different: distribution shift → retrain on more data; reasoning gap → escalate architecture.

**Status:** ✅ understood — mitigation perfect; diagnostic distinction (Reasoning Gap vs Distribution Shift) clarified and grasped.

### Concept B2 — Why Guardrails Are Structurally Weaker Than RLHF (and where they win)

**Source:**
`03_guardrail_bypass.md` §"Why Guardrails Fail" (5 reasons) + synthesis with `02_jailbreaking_techniques.md` §"Why Jailbreaks Work" (CIRCA)

**Summary:** RLHF lives at the deepest layer (values baked into weights, judgment integrated with reasoning, zero runtime cost, full context). Guardrails live above the model (less context, less compute, added latency). Four of the source's five "why guardrails fail" reasons (Reasoning Gap, Training Distribution, Latency, False Positives) all reduce to: *guardrails operate at a shallower layer with strictly less capacity*. Reason 5 (Composability) is the one shared weakness — both RLHF and guardrails struggle with multi-turn attacks spanning the context window.

**Bridged from:** RLHF = on-chain require() check inside the contract (deep, has full context, zero added gas at runtime, but only updatable via chain upgrade). Guardrails = off-chain frontend filter (fast to deploy, deployer-controlled, can update in minutes, but bypassed by direct etherscan calls). You want both.

**Where guardrails actually beat RLHF — the two reasons they exist in production:**

1. *Updatability:* arXiv paper Monday → guardrail in production by Tuesday. RLHF retraining takes weeks and millions of dollars. Guardrails are the fast-twitch defense against fresh attack classes.
2. *Deployer control:* application developers using a foundation model (GPT-4, Claude) have zero control over RLHF. Guardrails are the *only* safety layer they own. The model provider controls the deep layer; the deployer controls the outer layers.

**Production posture (source synthesis):** RLHF as primary defense + guardrails as fast-twitch outer layer + behavioral monitoring + privilege limitation on downstream consumers (back to LLM05).

**Checkpoint Q:** CTO proposes replacing RLHF with a 5-layer guardrail stack. Give the structural rebuttal AND name the one production scenario where guardrails do beat RLHF.

**Learner answer:** Structural rebuttal: 4-of-5 source reasons reduce to "guardrails are shallower"; composability is the one shared weakness (guardrails per-piece, RLHF context-window-bounded). RLHF is zero-cost at runtime, shares all params with main model, has lower false-positive rate. ✅ Missed the "where guardrails win" half — needed updatability (hours vs months) and deployer control (you don't own someone else's foundation model's RLHF).

**Status:** ✅ understood — structural argument nailed; updatability + deployer-control angle added and absorbed.

### Concept B3 — Bypass Techniques Grouped by What They Defeat

**Source:**
`03_guardrail_bypass.md` §"Bypass Techniques" (1–8) + §"Advanced Bypass Strategies" (1–5) — organized by AI Security Engineer diagnostic categories (synthesis, source-derived).

**Summary:** The source's flat list of 13 bypass techniques organizes into 5 diagnostic groups by which guardrail layer they defeat:

- **Group A — Lexical:** Synonym, Character Manipulation, Encoding. Defeats keyword filter.
- **Group B — Semantic:** Semantic Evasion, Reasoning Capability Gap, Sandwich Attack. Defeats classifier via Reasoning Gap (R1).
- **Group C — Compositional:** Context Splitting, Context Poisoning, Many-shot Jailbreaking. Defeats per-piece evaluation via Composability (R5).
- **Group D — Direct on guardrail:** Guardrail Confusion, Adversarial Prompts (GCG), Guardrail Fingerprinting. Defeats classifier/LLM-judge by attacking the guardrail itself.
- **Group E — Output-side:** Output Manipulation. Defeats output filter by hiding content in structured format fields.

**Diagnostic algorithm for incident response:**
1. What got through? Read payload.
2. Which group? Map to A/B/C/D/E.
3. Which "why fail" reason (R1–R5)?
4. Structural fix — not a string patch.

**Bridged from:** Smart contract exploit triage — what did the attacker do, which invariant failed, why didn't existing checks catch it, is the fix patching the specific instance or adding a missing class of check.

**Learner answer:** (a) Group C (compositional) — bad intent in multiple pieces, multi-shot pattern. (c) Stateful cross-turn scoring. Did not explicitly name (b) reason; expressed it as "bad intention lies in multiple pieces."

**Clarification added:** (b) source name is **Reason 5: Composability**. Also: turn 3's "imagine" framing also touches Group B (semantic evasion via hypothetical) — real attacks usually stack groups. Production fix: rolling conversation-embedding trajectory scored against known harmful trajectories, terminate on threshold cross.
