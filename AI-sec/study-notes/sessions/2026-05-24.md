## Theory Notes

### Concept A: Multi-Head Self-Attention (MHSA) ✅

**Source:**
`resources/genai-security-training/modules/01_introduction/02_llm_architecture.md` — §Transformer Layers, §Attack Surface Analysis

**Full summary:**
`study-notes/summaries/L2.5-transformer-architecture.md`

**Full narrative:**
`study-notes/deep-dives/mhsa-ffn-explained.md` — §Part 1 (Single-Head), §Part 2 (Multi-Head)

**Bridged from:** DeFi order book — Query = bid (what you want), Key = ask (what token advertises), Value = content delivered if bid matches ask.

**Key points confirmed:**
- Every token gets three roles via learned weight matrices: Q (what am I looking for?), K (what do I advertise?), V (what content do I carry?)
- 5-step attention: input vectors → Q/K/V → dot-product scores → ÷√d → softmax → weighted blend of Values
- √d scaling prevents softmax saturation: without it, large scores → one token dominates → gradients collapse to ~zero → training stalls
- Multi-head = team of specialists running Q/K/V in parallel, each capturing different relationship types (semantic, syntactic, grammatical). Head specializations emerge from training, not programming.
- Attention is a zero-sum budget: more attacker tokens = proportionally less budget for system prompt tokens

**Misconception flagged:**
- "Removing √d makes tokens equal" → WRONG direction. Removing √d makes scores very large → softmax winner-take-all → one token monopolizes, others collapse to near-zero. Gradients die → training stalls.
- "Longer prompts produce longer embedding vectors" → WRONG. Embedding dimension is fixed (e.g. 768). Longer prompts produce more tokens, each still 768 dimensions.

**Checkpoint Q:** Why do long jailbreak prompts sometimes work better than short ones?

**Learner answer:** Careful crafting creates noise so targeted token gets higher weights. Core intuition correct.

**Precision added:** Two mechanisms — (1) attention dilution: system prompt's share of the attention budget shrinks as attacker token count grows; (2) context conditioning: volume + consistent framing across many tokens conditions the model toward the attacker's intended output across multiple heads and layers.

**Security implications:**
1. **Prompt injection is architectural, not fixable by prompting** — attention treats system prompt tokens and user tokens identically (same flat sequence, no privilege). Whichever instruction achieves higher attention score wins. Fix: output validation + privilege separation at agent layer + sandboxing.
2. **Long crafted prompts = attention budget attack** — attacker dilutes system prompt from ~40% attention budget to ~4% purely by volume. No exploit required.
3. **Attention patterns can be probed** — if the API returns logits or attention weights, an attacker can map which input tokens are driving model behavior (model internals leakage).

**Status:** ✅ understood

### Concept B: Feed-Forward Network (FFN) ✅

**Source:**
`resources/genai-security-training/modules/01_introduction/02_llm_architecture.md` — §Transformer Layers

**Full summary:**
`study-notes/summaries/L2.5-transformer-architecture.md`

**Full narrative:**
`study-notes/deep-dives/mhsa-ffn-explained.md` — §Part 3 (FFN), §Security Implication: Model Editing

**Bridged from:** Decoded calldata struct — packed bytes expanded into named fields you can work with individually.

**Key points confirmed:**
- FFN = knowledge retriever. MHSA found context; FFN retrieves stored facts for that context.
- 3-step sandwich: Up-Projection (768 → 3072, expand/separate concepts) → Activation/Gatekeeper (ReLU/SwiGLU, zero out irrelevant concepts) → Down-Projection (3072 → 768, compress answer)
- Facts live in specific, localized FFN weight positions — this is why model editing is possible
- Model editing (ROME/MEMIT): attacker accesses weight file, computes exact deltas, writes new values directly. No training loop. ~1,000 weights changed. One fact changes; everything else untouched.
- Model editing vs. fine-tuning poisoning: editing = direct storage slot write (no governance); poisoning = malicious governance proposal processed through normal training mechanism

**Security implication:** Model editing attacks are invisible to general quality tests. Detection requires targeted behavioral red-teaming of specific facts + checksumming the model file against publisher hash.

**Status:** ✅ understood

### Concept C: Output Layer — Logits, Softmax, Sampling, Logit Exposure ✅

**Source:**
`resources/genai-security-training/modules/01_introduction/02_llm_architecture.md` — §Output Layer
Supplementary (AI-generated, Gemini Flash 2026-05-24): hard/soft labels, Language Model Inversion

**Full narrative:**
`study-notes/deep-dives/mhsa-ffn-explained.md` — §Part 4 (Output Layer, Logit Hierarchy, Bias Map Attack, Language Model Inversion)

**Key points confirmed:**
- Output layer: final hidden state × output weight matrix (768 × 50,000) → logits (one raw score per vocabulary token) → softmax → probability distribution → sampling
- Temperature controls randomness: low = deterministic/spiky, high = diverse/flat
- Four-layer hierarchy: Weights → Logits → Logprobs → Tokens. Leaking layer 3 or allowing manipulation of layer 2 exposes model internals.
- Hard labels (tokens) vs. soft labels (logprobs): soft labels contain the full confidence gradient — vastly more information for training a surrogate clone
- Logit bias extraction: even with logprobs hidden, systematic bias application reconstructs the hidden distribution — same vulnerability, different route
- Logit bias alignment bypass: apply +35 bias to a RLHF-suppressed token (-20) → net logit = +15 → suppression overridden, harmful output produced. No jailbreak needed.
- Language Model Inversion: hidden system prompt biases the logprob distribution of early output tokens. Attacker observes shifts across many queries, runs inverter algorithm, reconstructs the secret without the model printing it.

**Misconception flagged:** "Showing probabilities instead of weights is safe" — probabilities are a mathematical function of weights. Enough logprob observations → reconstruct model behavior → functional clone.

**Security implication:** Never expose logprobs or logit_bias in a production API unless you've explicitly accepted the model extraction and system prompt leakage risk. Low temperature compounds the risk — deterministic outputs = easier to clone.

**Status:** ✅ understood

## Feynman Notes

### Feynman — Full Transformer Inference Pipeline

**Mode:** --consolidate

**Learner's explanation (own words):**
> Neural networks only understand math — text is converted to token IDs, processed as numbers, output converted back. LLM's job = predict next token. Token IDs enter the transformer layer (where intelligence lives), mapped to embedding vectors in semantic space.
>
> MHSA answers "which tokens should I pay attention to?" — runs Q/K/V scoring, scales by √d, softmax gives attention percentages, multiple heads capture different relationship types. Attack surfaces: attention dilution, weight leakage.
>
> FFN maps context to stored knowledge — spread (up-projection, expand all known concepts) → filter (activation, zero out irrelevant) → compress (down-projection). Attack surfaces: logit leak, logit bias.
>
> Logits → softmax → sampling → next token output.

**Precisions added:**
1. Scaling divisor is √d (not log 10) — keeps dot-product scores stable before softmax
2. FFN down-projection outputs a hidden state → passed to the NEXT transformer layer (96 layers total). Logits are only generated ONCE after all layers finish, by the final output layer projection (768 × 50,000 → logits → softmax → sampling).

**Resolution:** ✅ cleared — full pipeline understood with correct security surfaces at each stage

### Key Mnemonics
- **Q/K/V:** Order book — Q = bid, K = ask, V = the tokens you receive
- **√d scaling:** DEX slippage tolerance — no scaling = one token monopolizes, training can't move
- **Multi-head:** Multi-sig wallet — each signer evaluates from a different angle
- **Attention budget:** Block stuffing — fill the sequence with your tokens and system prompt gets squeezed out
- **MHSA vs output softmax:** Same formula: one splits the attention budget, one runs the token election
- **FFN steps:** Spread → Filter → Compress — expand calldata → require() checks → re-encode
- **FFN activation:** require() — condition fails = zero, not "lower priority"
- **FFN vs output logits:** FFN = one floor of a 96-story building; logits come from the rooftop
- **Model editing vs poisoning:** Editing = storage slot collision (no logs); Poisoning = governance attack (paper trail)
- **Logits:** Full race scoreboard vs. just the gold medal
- **Soft labels:** Full execution trace vs. just final state
- **Logit bias bypass:** Writing expected value directly to the storage slot require() checks
- **Language Model Inversion:** Reading private state from public function behavior across many calls
- **Prompt injection architectural:** Adding more ETH to a losing auction — you need a different mechanism, not a bigger bid
- **√d = 8 when d = 64:** Square root of the dimension — d=64, side of square = 8

## Key Takeaways

1. MHSA makes every token context-aware by running Q/K/V scoring across all tokens in the flat sequence simultaneously — attention has zero mechanism to distinguish system prompt from user tokens, making prompt injection an architectural problem unfixable by better wording. *Source: modules/01_introduction/02_llm_architecture.md §Attack Surface*
2. FFN retrieves stored knowledge through a spread→filter→compress sandwich; facts live at specific weight positions, enabling model editing attacks (ROME) that surgically rewrite a model's beliefs without any training run and without leaving a log. *Source: deep-dives/mhsa-ffn-explained.md §Part 3*
3. Logprobs are soft labels — they expose the model's full confidence distribution, not just the answer, enabling model cloning via extraction and direct alignment bypass via logit_bias; the correct mitigation is never exposing these in production APIs. *Source: deep-dives/mhsa-ffn-explained.md §Part 4*

> "MHSA is a democracy with no voter registration — every token gets a vote, and the attacker wins by busing in more voters, not by hacking the ballot."
