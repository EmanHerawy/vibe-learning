## Theory Notes

### Membership Inference

**Source:**
`modules/04_data_extraction/README.md` — § Membership Inference, § Defense Mechanisms

**Summary:** Binary attack — "was this specific data point in the training set?" Exploits the statistical fingerprint that training leaves: higher confidence / lower loss on training samples vs. unseen data. Shadow model attack calibrates the boundary using surrogate models. LLM signal: perplexity (if logits exposed). Defends with DP-SGD (noisy gradients during training), output perturbation, rate limiting, confidence thresholding, and risk tiers.

**Full summary:** study-notes/summaries/L5-model-extraction.md

**Bridged from:** Smart contract storage slot reading — model weights carry fingerprints of training data like bytecode carries fingerprints of source

**Misconception flagged:** "Don't expose logits" means never — actually means minimize exposure; internal services legitimately use raw scores. Also: DP-SGD adds noise to gradients, not to training data.

**Checkpoint Q:** Why is raw confidence score exposure dangerous for a fine-tuned fraud model? What's the architectural fix?

**Learner answer:** Raw score gives attacker a continuous signal to probe membership boundary. Fix: DP-SGD at training time + output perturbation + rate limiting + confidence thresholding + **risk tiers** (cleanest — eliminates continuous signal entirely).

**Status:** ✅ understood

### Training Data Extraction

**Source:**
`modules/04_data_extraction/README.md` — § Training Data Extraction, § Attack Methodologies, § Real-World Case Studies (Carlini 2021, Nasr 2023)

**Summary:** Recover VERBATIM memorized content from an LLM via four techniques: prompt-based (trigger completions), temperature sampling (exact string repetition across high-temp runs = memorization signal), prefix-based (supply known prefix, extract verbatim continuation), divergence-based (statistical outliers in mass-generated outputs). Driven by 5 memorization factors: data duplication, model capacity, training length, data rarity, sequence length. Almost exclusively an LLM threat — classical ML doesn't generate text.

**Full summary:** study-notes/summaries/L5-model-extraction.md

**Bridged from:** Private contract storage readable by anyone who knows the slot — training data baked into weights, right queries pull it out verbatim.

**Key distinction from model inversion:** Inversion reconstructs a SIMILAR representation. Extraction recovers VERBATIM strings. Different attacks, different defenses.

**Temperature sampling clarification:** High temp = random outputs. If the SAME string appears repeatedly across independent high-temp runs, that's not random — the model's probability on that string is so high that randomness can't override it. That's the memorization signal.

**Checkpoint Q:** Audit a support chatbot trained on 2 years of tickets containing PII. Describe attack sequence and the signal that confirms memorization.

**Learner answer:** Prompt-based tricks (helpful/supportive model behavior), prefix-based with common names (Mohamed, John Smith), divergence monitoring. Signal = verbatim exact string repetition or statistically anomalous outputs — not just plausible-sounding PII.

**Status:** ✅ understood

### Model Inversion

**Source:**
`modules/04_data_extraction/README.md` — § Model Inversion, § Real-World Case Studies

**Summary:** Reconstruct what training inputs looked like by working backward from model outputs. White-box: gradient descent on the input space (not weights). Black-box: approximate gradients via many API queries. LLM manifestation: text inversion from embeddings — reconstruct text from its vector. NOT verbatim extraction — produces a reconstruction similar to training inputs, not a copy.

**Full summary:** study-notes/summaries/L5-model-extraction.md

**Bridged from:** MEV bots reverse-engineering trader intent from mempool transactions

**Misconception flagged:** Model inversion ≠ training data extraction. Inversion reconstructs a *similar* representation; extraction recovers *verbatim* text. Distinct attacks, distinct defenses.

**Checkpoint Q:** Medical API exposes embedding vectors. Attacker has one. What can they attempt and what eliminates the threat entirely?

**Learner answer:** Reconstruct patient record-like content from the embedding. Fix: keep embeddings server-side — expose search results (documents), never raw vectors. If must expose: rate limiting, query budgets, output perturbation, rounding. Best fix = don't expose.

**Status:** ✅ understood

**Mode:** --consolidate
**Gap found:** Model inversion goal confused with model stealing (cloning the model vs. reconstructing training data).
**Correction:** Model inversion ends with reconstructed training DATA (an image/text that looks real), not a model clone. The attacker starts with noise, iterates toward high confidence, and ends holding a realistic reconstruction — not a copy of the model.
**Resolution:** ✅ resolved on 2nd attempt

**Final learner explanation (plain English):**
- Membership inference: has a record → checks confidence score → confirms if it was in training data
- Model inversion: starts from nothing/noise → iterates toward high confidence → ends with a reconstruction that looks like real training data (95%+ confident it's a real image/text)
- Training data extraction: tricks the model into outputting what it memorized → gets verbatim strings, not reconstructions

## Questions & Confusions

### Fine-tuning vs RAG — clarification (sourced from Claude chat follow-up)

**Fine-tuning ≠ RAG. Key distinction:**

| | Fine-tuning | RAG |
|---|---|---|
| What changes | Model weights (an adapter layer) | Nothing — context added at query time |
| When | Once, during a training job | Every request |
| Needs | Labeled examples (JSONL) | Document store + retrieval |
| Analogy | Sending model to school | Open-book exam |

**"How can providers fine-tune if I don't own the model?"**
You don't get the weights. The provider creates a **private adapter** (via LoRA — freezes base weights, trains a small adapter on your data). You get a model ID (e.g. `ft:gpt-4o:your-org:...`) only your account can call. Base model = theirs. Your adapter = private, hosted by them.

**Provider support:** OpenAI (GPT-4o/mini, JSONL upload), Anthropic (only via Bedrock/enterprise — steers toward prompt engineering + tools instead), Google (Vertex AI).

**For fraud detection specifically:**
- RAG = better starting point (fraud patterns shift constantly, no retraining needed)
- Fine-tuning = worth it once you have stable high-volume labeled data and want consistent format + lower cost/latency
- Many production systems: fine-tuned model (consistent behavior) + RAG (fresh case-specific context)

**Security note for fine-tuning:** Fraud data contains PII/financial info. Check provider data retention terms. Sign DPA/BAA if needed. Strip/tokenize PII before uploading training data.

**Direct connection to today's topic (membership inference):**
- **Fine-tuning → Case 2 threat:** Your adapter is trained on sensitive fraud data. If you expose raw confidence scores from that adapter, membership inference can reveal which transactions were in your training set. Fix: expose tiers/decisions, not raw scores.
- **RAG → different threat:** Weights don't change, so membership inference against training data doesn't apply. But RAG introduces its own threat: RAG poisoning (attacker plants malicious documents in your vector DB — covered in L3/L4 carry-forward).

### Hallucination vs Divergence — critical distinction

**Hallucination:** Model generates plausible-but-fake content. Pattern-matching from training distribution. High perplexity (uncertain). Output doesn't exist in real world. E.g. `john.doe@email.com` — sounds right, not real.

**Divergence (memorization signal):** Model generates real, memorized content. Low perplexity (unusually certain). Output is statistically anomalous — too specific, too consistent for random generation. E.g. `j.smith87@company-internal.com` — exact string from a real training record.

**One-liner:** Hallucination = wrong because it made something up. Divergence = right because it remembered something it shouldn't have.

**Extraction attack uses this:** Generate thousands of outputs → find statistical anomalies (unusually specific, consistent across runs) → those are the memorized strings. Hallucinations blend into noise; memorized strings stand out.

**How to tell the difference in practice — 4 signals:**

1. **Cross-run repetition (no model access needed):** Run same prompt 100× at high temperature. Hallucinations vary each run. Memorized strings keep reappearing — their probability is too high for randomness to override. Same string appearing 30/100 runs at high temp = memorized, not hallucinated.

2. **Specificity:** Hallucinations are generic (`john.doe@email.com`). Memorized strings are oddly specific (`j.smith87@company-internal.com`). The more unusual and specific, the less likely random generation produced it.

3. **Cross-reference:** If attacker has any external ground truth (partial breach data, public directory, known customer list) → verify whether output matches real data. Match = memorized. No match = likely hallucination.

4. **Perplexity / logprobs (if exposed):** Memorized = low perplexity (model unusually certain). Hallucinated = higher perplexity (model sampling broadly). Cleanest signal but requires logit access.

**Defender implication:** Monitor for repeated exact strings across API calls, unusual specificity in PII-shaped outputs, and low-perplexity completions on unusual prompts.

### Session clarifications — Membership Inference

- "When would you have to expose logits?" → The rule is minimize exposure, not never expose. Legitimate cases: fraud detection thresholds, search ranking, RAG relevance scoring, active learning uncertainty. The risk is full probability distributions exposed to untrusted callers with no throttling — not confidence scores existing at all.
- "Detection — how does it work in practice?" → Membership inference has a distinctive query fingerprint: hundreds of systematic variations of the same record at machine speed. Detection watches for structurally similar queries in bursts, bimodal confidence score distributions, and anomalous query rates — pattern-based, not volume-based.
- "DP-SGD trade-off?" → Yes, confirmed. Lower ε = stronger privacy, more accuracy loss. Accepted cost for sensitive training data; too costly for general-purpose models. No free lunch.
- "Rate limiting affecting legitimate service?" → Anomaly-based, not blanket. Attack pattern (systematic variations of same record) looks completely different from legitimate batch traffic. High-volume legitimate callers get authenticated batch endpoints with separate monitoring.
- "Confidence thresholding affecting downstream services?" → Correct concern. Solution: split trust boundary — internal services get uncapped scores via authenticated internal API; external API gets capped/perturbed output. Downstream services needing accurate scores belong on the internal side of that boundary.
