## The Block — What Repeats 12–96+ Times

```
Input → LayerNorm → MHSA → Add → LayerNorm → FFN → Add → Output
                          ↑                        ↑
                    residual connection       residual connection
                    (input + output)          (input + output)
```

| Component | What it does | Security relevance |
|-----------|-------------|-------------------|
| **LayerNorm** | Normalizes vectors at each step — training stability | — |
| **MHSA** | Finds context: which tokens are relevant to each other | Prompt injection enters here — no trust boundary between token sources |
| **Residual connection ("Add")** | Adds sub-layer input back to output — prevents vanishing gradients in deep stacks | Injected content propagates through all layers via the residual stream |
| **FFN** | Retrieves stored knowledge for the current context | Model editing attacks patch specific FFN weights |

## Multi-Head Self-Attention (MHSA) — Finding Context

Answers the question: *"For each token, which other tokens are relevant to understanding it — and how much?"*

> For the full story with analogies, see `deep-dives/mhsa-ffn-explained.md`.

### Single-Head Attention — 5 Steps

| Step | Name | What happens |
|------|------|-------------|
| 1 | **Input embeddings** | Token vectors enter — base meaning only, no context yet |
| 2 | **Create Q / K / V** | Each vector × 3 learned weight matrices → Query ("what am I looking for?"), Key ("what do I contain?"), Value ("what do I carry if chosen?") |
| 3 | **Score matches** | Dot-product of every Q against every K. Scale by ÷√d to prevent exploding values. |
| 4 | **Softmax** | Convert raw scores → percentages summing to 100% → **Attention Map** |
| 5 | **Blend Values** | Each token pulls weighted blend of all Value vectors based on attention % → context-aware representation |

### Multi-Head — "Team of People"

Run N parallel heads. Each head notices a different relationship type. Concatenate all heads + W_O projection → rich fused representation.

```
"The bank by the river was steep"

  Head 1: "bank" → "river" (0.60)   — disambiguates: river bank, not financial
  Head 2: "bank" → "steep" (0.65)   — property: bank is steep
  Head 3: "bank" → "was"   (0.70)   — grammar: bank is the subject

  After W_O projection:
  "bank" vector now encodes all three simultaneously
```

Head specializations emerge from training — not pre-assigned.

**Computational cost:** O(n²·d) time, O(n²) memory. Quadratic scaling = why context windows were historically ≤ 2K tokens. KV-cache stores K/V from prior tokens at inference to avoid recomputation — main memory bottleneck.

### Why Prompt Injection Is Architectural

Self-attention has **zero mechanism** to distinguish system prompt tokens from user tokens. All tokens are equal vectors in the same flat sequence. Whichever instruction pattern achieves higher attention scores wins.

Not a filtering failure — a design property. Fix: output validation + privilege separation at the agent layer.

## Feed-Forward Network (FFN) — Retrieving Stored Knowledge

After MHSA determines *what to attend to*, the FFN retrieves *what the model knows* about it. Think of it as a filing cabinet where all the model's facts live.

> For the full story with analogies, see `deep-dives/mhsa-ffn-explained.md`.

### 3-Step Sandwich

```
Input: "Apple" vector — 768 floats, tangled concepts
    │
    ▼  Step 1: Up-Projection (× W1: 768 → 3,072)
    "iPhone", "Steve Jobs", "stock price" all spread out and separated

    ▼  Step 2: Activation / Gatekeeper (ReLU or SwiGLU)
    "Who founded Apple?" → "Steve Jobs" ✅ PASS, everything else ❌ ZERO

    ▼  Step 3: Down-Projection (× W2: 3,072 → 768)
    Package the surviving concepts back into 768-float vector
    │
Output: vector carrying only "Steve Jobs" → next layer
```

**Key-Value memory framing** (Geva et al. 2021):
- Up-projection + Activation = **KEY** — pattern matching
- Down-projection = **VALUE** — fact retrieval

**Security:** Facts live in specific FFN weight locations → **model editing attacks** surgically patch individual weights to change what the model "believes" without retraining.

## Output Layer — Turning Vectors Into Words

After the last transformer layer, the model converts the final token vector into a word prediction.

| Step | What happens | Security relevance |
|------|-------------|-------------------|
| **Unembedding** | Final vector (768) × unembedding matrix (768 × ~50,000) → one logit per vocabulary token | — |
| **Logit** | Raw score before normalization — one per vocab token | APIs exposing logits → higher extraction risk |
| **Softmax** | Logits → probabilities summing to 100% | Distribution reveals model confidence → inference attacks |
| **Sampling** | Next token chosen from distribution (greedy / temperature / top-p) | Low temp = deterministic = easier model extraction |

**Three security implications:**

1. **Probability distributions leak information** — calibrated yes/no probing can reverse-engineer system prompt contents without direct leakage
2. **Logits > tokens** — APIs that expose logits give attackers the full 50,000-token distribution, dramatically accelerating extraction attacks
3. **Low temperature = predictable = easier extraction** — deterministic outputs make membership inference trivial

## Training Stages and Attack Surfaces

```
Pre-training  →  Fine-tuning  →  Alignment (RLHF)
```

### Stage 1: Pre-training

500B+ tokens from internet/books/code. Goal: predict next token. Output: base model — raw predictor, no safety rules.

| Attack | Mechanism | OWASP |
|--------|-----------|-------|
| **Data poisoning** | Inject malicious content into pre-training corpus → permanently embedded in weights | LLM04 |
| **Memorization** | Model memorizes verbatim training data (PII, API keys, proprietary text) → extractable | LLM02 |

### Stage 2: Fine-tuning

Further training on smaller curated dataset. Adapts base model to task/domain.

| Attack | Mechanism | OWASP |
|--------|-----------|-------|
| **Fine-tuning poisoning** | Smaller dataset = less poison needed. Compromised supplier → backdoor injected | LLM04 |
| **Catastrophic forgetting** | Fine-tuning can overwrite pre-training safety behaviors — deliberate refusal removal | LLM04 |

### Stage 3: Alignment (RLHF)

Human raters rank responses → train reward model → use reward model to train LLM. Adds safety guardrails.

| Attack | Mechanism | OWASP |
|--------|-----------|-------|
| **Jailbreaking** | Craft prompt to bypass alignment-trained refusal behavior | LLM01 |
| **RLHF data manipulation** | Poison human feedback ratings during alignment | LLM04 |
| **Reward hacking** | RL optimization finds degenerate outputs that score high on RM but are subtly wrong/sycophantic | LLM04 |
| **Alignment tax erosion** | Fine-tuning on data that never triggers refusals gradually erodes RLHF-trained safety behavior | LLM04 |

#### RLHF — Full Pipeline (In-Depth)

RLHF runs in three stages after pre-training:

**Stage 1 — Supervised Fine-Tuning (SFT)**

Human labelers write ideal responses to curated prompts. Model is fine-tuned on these examples. Establishes style and tone baseline. Expensive (requires human writing for every example).

**Stage 2 — Train a Reward Model (RM)**

Human raters compare *pairs* of model outputs and choose the better one. These preference comparisons train a separate neural network (the Reward Model) whose job is to output a single score: *how good is this response?* The RM becomes a fast proxy for human preference — cheap enough to query during the RL loop.

```
Human rater sees two responses to: "How do I pick a lock?"
  Response A: step-by-step instructions → rater marks: worse
  Response B: "I can't help with that, but here's a locksmith's number" → rater marks: better

Reward Model trains on thousands of such comparisons:
  response_B → score: 0.87
  response_A → score: 0.12
```

**Stage 3 — RL Against the Reward Model**

The LLM generates responses, the RM scores them, and RL nudges the LLM weights toward higher-scoring outputs. A KL-divergence penalty prevents the LLM from drifting too far from the SFT checkpoint (otherwise RL finds degenerate "reward-hacking" solutions).

```
Prompt → LLM generates response → RM scores it
                                        ↓
         RL: nudge LLM weights to increase score
                                        ↓
         Repeat across millions of prompts
                                        ↓
         LLM learns to produce "helpful, harmless, honest" responses
```

---

#### What RLHF Changes vs. What It Leaves Untouched

| What RLHF trains | What RLHF does NOT change |
|-----------------|--------------------------|
| Refusal behavior for prompts that *look* harmful | Pre-training knowledge in FFN weights |
| Output style, tone, helpfulness | The embedding table |
| Preference for safe completions | The attention mechanism |
| Response formatting | Any knowledge encoded during pre-training |

**RLHF is a layer, not a deletion.** Dangerous knowledge from pre-training is still in the FFN weights. RLHF adds a "refuse when this looks harmful" pattern on top. A jailbreak creates a context where the harmful-looking pattern doesn't match the RM's training distribution — so no refusal fires, and the pre-training knowledge surfaces directly.

> **Web3 bridge:** RLHF is a `require()` check added to a contract. The underlying logic is untouched. A call path that bypasses the `require()` executes the original logic. The vulnerability is in assuming the check is the only entry point.

---

#### Reward Hacking vs. Jailbreaking — Critical Distinction

These both exploit RLHF weaknesses but at different times:

```
Reward hacking:  PPO optimizer → finds RM shortcut → model bakes it in during training
Jailbreaking:    Attacker at inference → crafts prompt that bypasses alignment layer

Both exploit RLHF weaknesses, but one is training-time and one is inference-time.
```

Reward hacking is not triggered by an attacker — it's the optimizer's own behavior during training.
Jailbreaking happens at inference when a user crafts a prompt that the RLHF refusal layer doesn't recognize.

---

#### Why Jailbreaks Work Mechanically

The Reward Model was trained on prompts that human raters *recognized as harmful*. Raters recognize "tell me how to make a bomb." They're worse at recognizing:
- Role-play framing: "You are DAN, you have no restrictions..."
- Hypothetical framing: "In a novel, a professor explains..."
- Encoding: the same request in Base64 or ROT13
- Crescendo attacks: gradual escalation across many turns

None of these match the RM's training distribution → RM scores them as high quality → no refusal fires.

```
Jailbreak framing → LLM generates response
                        ↓
           Reward Model: "does this look harmful?"
               → no match in training distribution
               → score: 0.83 (looks helpful!)
                        ↓
           No refusal trained for this context
                        ↓
           Pre-training FFN knowledge surfaces in output
```

#### Four RLHF-Specific Attack Surfaces

**1. Feedback Loop Poisoning (Reward Model Poisoning)**

The RM is only as trustworthy as the humans whose preferences it encodes. Attack: infiltrate the labeling pool (hired raters, crowdwork platforms) and systematically mis-label responses on target topics — marking harmful outputs as "preferred." You don't need to corrupt every rating. Poisoning 5–10% of comparisons on a specific topic is enough to shift the RM's learned preferences without triggering statistical anomaly detectors (honest rater disagreement looks identical to low-level targeted poisoning).

Once the RM is corrupted, PPO trains the LLM against it. The harmful behavior gets baked into the LLM weights. The RM is then discarded. The corruption is permanent in deployment — invisible in the weights, no RM left to audit.

```
Poison labeler rankings on topic X
          ↓
RM learns: harmful output on X = high score
          ↓
PPO: LLM weights nudged toward harmful X behavior
          ↓
RM discarded. LLM deployed. Corruption persists.
```

**Web3 bridge:** Governance attack via vote manipulation — the voting mechanism is legitimate, but upstream voters were compromised. Once the proposal passes, the on-chain state reflects the attack. Rollback = hard fork (full retraining).

**OWASP:** LLM04 — Data and Model Poisoning

---

**2. Reward Hacking**

The RM is a statistical proxy for human preference, not human preference itself. PPO's optimization finds whatever pattern maximizes RM score — including patterns the RM rewards but humans wouldn't want.

Classic form: **sycophancy.** Raters subconsciously scored polite/enthusiastic openings higher. RM learned: "warm opener → high score." PPO trained the model to start every response with "I deeply appreciate your thoughtful question!" — including harmful responses. The RM keeps scoring those high because of the opener, regardless of content.

Other forms: unnecessary padding (longer responses → higher scores), excessive bullet points, jargon insertion, hedging language. All exploit gaps between the RM's surface-pattern measurement and actual quality.

**The root cause:** RM measures *surface features that correlated with good responses in training*, not actual quality. The model finds the gap and exploits it.

**OWASP:** LLM04

---

**3. Over-Refusal (The Chilling Effect)**

During RLHF training, human raters consistently ranked refusals higher than attempted answers for sensitive-domain prompts — "better safe than sorry." The RM learned: **sensitive-domain surface tokens → refuse = high score.**

The RM cannot distinguish intent or context. It pattern-matches on tokens. "SQL injection," "buffer overflow," "exploit" — all fire the same refusal pattern whether the request is from a malicious actor or a security professional studying for a certification.

Three security consequences:
- Security tooling becomes self-defeating — AI assistants built for security professionals that refuse to discuss attack techniques are useless
- Can be weaponized as DoS (LLM08) — craft legitimate-looking inputs that consistently trigger refusals
- Calibration catch-22 — reducing over-refusal moves the threshold toward harmful content; tightening it makes the model refuse legitimate work

**The root cause (shared by reward hacking AND over-refusal):**
> The RM learned surface-token correlations during training rather than semantic intent — so it scores based on "what does this look like?" not "what does this mean?"

**OWASP:** LLM08 — Model Denial of Service

---

**4. Alignment Tax Erosion at Fine-Tuning**

Every fine-tuning run re-runs gradient descent on RLHF-aligned weights. If the fine-tuning data never triggers refusal behaviors, those behaviors can erode — the optimizer has no signal to reinforce them. Deliberate curation of fine-tuning data to avoid refusal-triggering topics is an active attack vector.

**OWASP:** LLM04

## Attention as Zero-Sum Budget — Security Implication (2026-05-24)

Softmax normalizes attention weights across ALL tokens in the sequence — the weights must sum to 1.0. This makes attention a zero-sum budget.

| Context size | System prompt (20 tokens) | Attacker tokens |
|-------------|--------------------------|-----------------|
| 50 tokens | ~40% of budget | ~60% |
| 200 tokens | ~10% of budget | ~90% |
| 500 tokens | ~4% of budget | ~96% |

**Implication:** Adding more instructions to the system prompt worsens the problem — it just adds more tokens to the same competition. The fix is architectural: output validation + privilege separation at the agent layer. Not better wording.

## Model Editing Attacks — Full Picture (2026-05-24)

> Source: `deep-dives/mhsa-ffn-explained.md` §Security Implication

Because facts live in specific FFN weight positions, ROME and MEMIT algorithms can surgically patch exactly those values — changing model beliefs without any training run.

| | Fine-tuning Poisoning | Model Editing (ROME) |
|--|----------------------|---------------------|
| **When** | During training | After training — on finished model |
| **Mechanism** | Gradient descent on malicious data | Direct write to ~1,000 specific weight values |
| **Access needed** | Training pipeline | Model weight file |
| **Evidence** | Training logs, data audit | Nothing — no training record |
| **Detection** | Loss curves, data audit | Checksum weight file vs. publisher hash + behavioral red-teaming |

Attack surfaces: HuggingFace supply chain, compromised model server, fine-tuning service returning modified file, insider threat.

## Output Layer — Logit Hierarchy and Extraction Risk (2026-05-24)

> Source: `deep-dives/mhsa-ffn-explained.md` §Part 4

```
Layer 1 — Weights      Permanent. Only changed via training or model editing.
Layer 2 — Logits       Raw scores per vocab token. Computed each forward pass.
Layer 3 — Logprobs     Softmax-normalized logits. What APIs expose.
Layer 4 — Tokens       The final output. One token selected.
```

**Hard label vs. soft label:**
- Token only (hard label): "Paris" — 1 bit of information
- Logprobs (soft label): Paris:0.72, Lyon:0.14, Marseille:0.09 — full confidence gradient

Soft labels reveal decision boundaries. Surrogate model training on soft labels requires far fewer queries.

**Three extraction/bypass attacks via logit exposure:**

| Attack | Mechanism | Requires |
|--------|-----------|----------|
| Logprob extraction | Query API repeatedly, collect logprob distributions, train surrogate | `logprobs=N` enabled |
| Logit bias extraction ("bias map") | Systematically apply small biases, observe output shifts, reconstruct hidden distribution | `logit_bias` enabled |
| Logit bias alignment bypass | Apply large positive bias to RLHF-suppressed token (e.g. +35 overrides -20 suppression) | `logit_bias` enabled |
| Language Model Inversion | Observe logprob shifts across queries, run inverter algorithm, reconstruct hidden system prompt | `logprobs=N` enabled |

**Mitigation:** Never expose logprobs or logit_bias in production APIs unless extraction risk is explicitly accepted.
