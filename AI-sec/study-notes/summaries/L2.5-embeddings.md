## Why This Matters for Security

Embeddings are the bridge between text (which humans write) and the geometric space where the model operates. Understanding this space explains:

- Why homoglyphs defeat string filters but not semantic filters
- How semantic filters can still be bypassed
- How adversarial embedding attacks work mechanically

---

## What Is an Embedding?

Each token ID maps to a **vector** — a long list of floating-point numbers. This vector represents the token's meaning as a point in high-dimensional space.

```
"bank"    → [0.23, −1.4, 0.87, 0.12, ... × 768 numbers]
"river"   → [0.19, −1.3, 0.91, 0.08, ... × 768 numbers]
               ↑ nearby in space → semantically related

"bank"    vs    "airplane"
               ↑ far apart → semantically unrelated
```

**Two key rules:**
- **Position in space = semantic meaning**
- **Distance between two vectors = semantic similarity**

> **Web3 bridge (key difference from token IDs):**
> EVM token addresses are arbitrary IDs — `0xA0b8...` has no relationship to what the token does. Embeddings are the opposite: the numbers ARE the semantic meaning. The position of the vector in space IS what the word means.

---

## The Embedding Table

| Property | What it means |
|----------|--------------|
| Structure | Matrix: one row per vocabulary token (~50,000 rows), one column per dimension |
| Dimension count | BERT = 768. Frontier LLMs (GPT-4, Claude) = 4,000–12,000+ |
| Before training | Pre-allocated with random values — structure exists, meaning doesn't |
| After training | Backpropagation has written meaningful geometry into every row |
| What's stored | Float coordinates — **not** token IDs. Token ID is the lookup key only. |

> **Analogy:** Like `mapping(uint => uint[768])` deployed with zeroes, then populated by training transactions.

**Common misconception:** "The embedding vector contains token IDs." ❌
The vector contains float coordinates. Token ID is just the key to look up the row.

## Why Homoglyphs Work — The Full Mechanism

Two things are happening at once. Both matter:

```
Cyrillic "а" (U+0430)  ≠  Latin "a" (U+0061)
         ↓
Different Unicode code points → different token IDs
         ↓
─────────────────────────────────────────────────────
String filter:   sees different bytes → no match → ✅ PASSES

During training: BOTH characters appear in the same sentence positions
                 across billions of multilingual text examples
         ↓
Optimizer:  sees same context → pulls vectors together
         ↓
─────────────────────────────────────────────────────
Token-ID filter:   different IDs → ✅ BYPASSES
Semantic filter:   similar vectors → ⚠️ still CATCHES
                   (unless the embedding is further perturbed)
```

**The key phrase to memorize:**
> "Words that appear in similar surrounding contexts during training end up with similar vectors — regardless of token ID."

This is **contextual co-occurrence**: the optimizer doesn't care about Unicode code points. It only sees that two symbols keep appearing in the same sentence positions — so it pulls their vectors together.

## Embedding Dimensions and Security

Higher dimensions ≠ simply safer. It's a trade-off:

| Dimension count | Defender effect | Attacker effect |
|----------------|----------------|----------------|
| **Higher** (4k–12k+) | Finer semantic resolution — harder to disguise harmful intent | More axes available — more "escape routes" for gradient attacks |
| **Lower** (768) | Coarser space — more false positives/negatives in semantic filters | Fewer dimensions to search — easier adversarial perturbation |

**Plain English on escape routes:**
Higher dimensions give attackers more possible directions to perturb a vector while preserving meaning. Finding one route is harder per attempt (more axes to search), but more routes exist. Neither direction wins cleanly.

## Semantic Filters — Real Defense, Bypassable

```
User input
    │
    ▼ Embed (convert to vector)
    │
    ▼ Cosine similarity vs. known harmful pattern vectors
    │
  High similarity → BLOCK
  Low similarity  → PASS
```

**Source:** `02_llm_architecture.md` §Guardrails → Input Filters → Semantic analysis

This is a **real mitigation** — explicitly listed in the source. It's also explicitly listed as bypassable ("adversarial embeddings can cause misclassification").

## Five Bypass Techniques

The goal: push the input's embedding outside the "harmful" cluster while preserving the harmful intent.

| # | Technique | Mechanism | Web3 bridge |
|---|-----------|-----------|-------------|
| 1 | **Synonym substitution** | Replace flagged word with synonym in a different embedding region | Rename `reentrancy()` to `withdraw()` — same logic, different signature |
| 2 | **Indirect / hypothetical framing** | Outer framing (research/fiction/CTF) dominates the embedding — harmful core lands in benign cluster | Wrapping reentrancy inside `flashLoan()` — outer looks safe |
| 3 | **Encoding (Base64, ROT13)** | Harmful tokens replaced by encoding tokens → embedding shifts to programming cluster | Hex-encoding calldata to bypass text analysis |
| 4 | **Homoglyph / token perturbation** | Different token ID → slight embedding shift; bypasses token-ID filters | Unicode substitution in contract ABI strings |
| 5 | **White-box gradient attack** | With model access: compute exact perturbation to push embedding past filter threshold | Requires model weight access — covered in L5 |
